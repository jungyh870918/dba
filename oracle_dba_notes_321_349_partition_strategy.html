1. 파티셔닝을 왜 쓰는가

파티셔닝은 단순히 “조회 빠르게 하려고 쓰는 기능”이 아니다. DBA 운영 관점에서는 오히려 대용량 테이블을 관리 가능한 단위로 쪼개는 운영 전략에 가깝다.

핵심 문장: 파티션은 성능 기능이기도 하지만, 더 본질적으로는 삭제·보관·백업·장애 영향을 관리하기 위한 운영 전략이다.

예를 들어 결제, 주문, 로그, 통계, 이력 테이블처럼 데이터가 계속 쌓이는 테이블을 일반 테이블로 두면 오래된 데이터를 삭제할 때 대량 DELETE를 해야 한다.

DELETE FROM payments
WHERE pay_date < ADD_MONTHS(SYSDATE, -12);

이 방식은 redo, undo, archive log, buffer cache, LGWR, DBWR에 큰 부담을 준다. 반면 월 단위 Range Partition으로 나눠두면 오래된 월 파티션을 메타데이터 작업에 가깝게 제거할 수 있다.

ALTER TABLE payments DROP PARTITION p202401;
즉, 파티션의 가치는 “한 번의 SQL을 빨리 실행하는 것”보다 “매월 반복되는 삭제, 보관, 백업, 장애 대응을 안전하게 만드는 것”에서 더 크게 나온다.

2. 파티션 유형 선택: 일반적인 것과 특수한 것

유형일반성주요 기준대표 Use Case
Range Partition 가장 일반적 날짜, 기간, 순번처럼 범위로 나뉘는 값 주문, 결제, 접속 로그, 정산, 통계, 이력 테이블
List Partition 조건부 사용 지역, 국가, 회사코드, 업무구분처럼 값 목록이 명확한 경우 국가별 데이터, 사업부별 데이터, 서비스 구분별 데이터
Hash Partition 특수 목적 특정 키에 데이터가 몰리는 hot spot 완화, 병렬 분산 초대형 OLTP, insert 경합 분산, 병렬 처리 중심 시스템
Composite Partition 고급/대형 시스템 Range + Hash, Range + List처럼 두 기준을 함께 사용 기간별 보관도 필요하고 내부 분산도 필요한 대용량 시스템
특수한 케이스를 제외하면, 실무에서 가장 먼저 검토하는 기본형은 Range Partition이다. 이유는 대부분의 데이터 보관·삭제 기준이 “시간”이기 때문이다.

3. 일반적인 Range Partition 설계

가장 흔한 패턴은 월 단위 Range Partition이다. 예를 들어 주문 테이블을 주문일 기준으로 월별 분할한다.

CREATE TABLE orders (
  order_id     NUMBER,
  order_date   DATE NOT NULL,
  customer_id  NUMBER,
  status       VARCHAR2(20),
  amount       NUMBER
)
PARTITION BY RANGE (order_date) (
  PARTITION p202401 VALUES LESS THAN (DATE '2024-02-01'),
  PARTITION p202402 VALUES LESS THAN (DATE '2024-03-01'),
  PARTITION p202403 VALUES LESS THAN (DATE '2024-04-01'),
  PARTITION pmax    VALUES LESS THAN (MAXVALUE)
);

좋은 조건절

파티션 키를 그대로 비교하면 옵티마이저가 불필요한 파티션을 제외하는 Partition Pruning을 하기 쉽다.

WHERE order_date >= DATE '2024-02-01'
  AND order_date <  DATE '2024-03-01'

피해야 하는 조건절

파티션 키에 함수를 씌우면 파티션 경계와 조건이 바로 매칭되지 않아 Pruning이 약해지거나 실패할 수 있다.

WHERE TO_CHAR(order_date, 'YYYYMM') = '202402'
파티션 키는 “관리 기준”이다. 삭제·보관·백업·조회 조건에서 실제로 자주 쓰이지 않는 컬럼을 파티션 키로 잡으면 파티션을 만들었는데도 운영 효과가 작다.

4. 인덱스 파티셔닝 기본: 테이블 파티션과 인덱스 파티션은 별개다

테이블을 파티션으로 만들었다고 해서 인덱스가 자동으로 Local Partition Index가 되는 것은 아니다. 인덱스 생성 시 LOCAL을 명시해야 테이블 파티션과 1:1로 대응되는 로컬 인덱스 파티션이 생성된다.

-- 파티션 테이블 위에 일반 인덱스 생성
-- LOCAL을 쓰지 않으면 일반적으로 Nonpartitioned Index가 된다.
CREATE INDEX idx_orders_customer
ON orders (customer_id);
-- Local Partition Index 생성
-- 테이블 파티션과 인덱스 파티션이 1:1로 대응된다.
CREATE INDEX idx_orders_customer_l
ON orders (customer_id)
LOCAL;
교재의 핵심도 같다. Local 파티션 인덱스는 테이블 파티션과 1:1로 대응되며, 인덱스 파티션 키를 사용자가 따로 지정하지 않고 DBMS가 테이블과의 1:1 관계를 자동 유지한다.
Table p202401 → 해당 월 데이터 저장
Local Index p202401에 대응되는 인덱스 파티션 생성
Query 2024년 1월 조건 조회 시 테이블 p202401과 그에 대응되는 인덱스 파티션만 접근 가능

따라서 DBA는 “파티션 테이블을 만들었다”에서 끝내면 안 된다. 그 테이블에 붙는 인덱스를 Local로 둘지, Global로 둘지, Nonpartitioned로 둘지 반드시 결정해야 한다.

5. Local Index와 Global Index의 범위

Local Index

테이블 파티션과 인덱스 파티션이 1:1로 대응된다.

p202401 → index partition A
p202402 → index partition B
p202403 → index partition C

장점은 파티션 관리 작업과 같이 움직이기 쉽다는 점이다.

Global Index

여러 테이블을 가로지르는 인덱스가 아니다. 같은 테이블 안의 여러 파티션 전체를 가로지르는 인덱스다.

p202401
p202402  → 하나의 큰 인덱스 구조
p202403

장점은 전체 파티션을 대상으로 특정 키를 빠르게 찾을 수 있다는 점이다.

Global의 범위는 “여러 테이블”이 아니라 “같은 테이블의 여러 파티션 전체”다. Local의 범위는 “각 테이블 파티션에 대응되는 인덱스 파티션”이다.

6. Prefixed와 Nonprefixed: Range면 무조건 Prefixed인가?

Prefixed는 파티션 키 컬럼이 인덱스 선두 컬럼에 위치하는 경우다.

-- 테이블 파티션 키: order_date
-- Local Prefixed Index
CREATE INDEX orders_lix_prefixed
ON orders (order_date, customer_id)
LOCAL;

Nonprefixed는 파티션 키가 선두가 아니거나, 인덱스 컬럼에 파티션 키가 없는 경우다.

-- Local Nonprefixed Index
CREATE INDEX orders_lix_nonprefixed
ON orders (customer_id, order_date)
LOCAL;
구분PrefixedNonprefixed
Local Index가능가능
Global Index가능Oracle에서 지원하지 않음
Range Partition을 만들었다면 Prefixed가 자연스러운 경우가 많다. 특히 조회 조건이 날짜 범위 중심이면 (order_date, ...)가 좋다. 하지만 모든 상황에서 표준은 아니다.

예를 들어 테이블은 보관·삭제 때문에 order_date로 파티션했지만, 실제 서비스 조회의 95%가 고객 ID 중심이라면 인덱스 선두는 customer_id가 더 적합할 수 있다.

-- 조회 패턴이 고객 중심인 경우
WHERE customer_id = :customer_id

-- 이 경우 인덱스는 다음이 더 자연스러울 수 있음
CREATE INDEX orders_lix_customer
ON orders (customer_id, order_date)
LOCAL;
파티션 키는 삭제·보관·관리 기준이고, 인덱스 선두 컬럼은 조회 접근 경로 기준이다. 둘은 같을 수도 있지만 반드시 같아야 하는 것은 아니다.

7. 운영 지옥: 파티션 관리와 Index Unusable

DBA가 파티션 인덱스 전략을 중요하게 보는 이유는 파티션 관리 작업 중 인덱스가 UNUSABLE 상태가 될 수 있기 때문이다.

Local Index일 때

ALTER TABLE orders DROP PARTITION p202401;

Local Index는 테이블 파티션과 1:1 대응되므로, 해당 테이블 파티션과 대응되는 인덱스 파티션만 같이 제거되거나 관리된다. 운영 영향이 상대적으로 작다.

Global 또는 Nonpartitioned Index일 때

Global/Nonpartitioned Index는 하나의 인덱스 구조 안에 여러 파티션의 rowid가 섞여 있다. 그런데 특정 테이블 파티션을 drop/move/exchange하면 그 안에 있던 rowid가 사라지거나 바뀐다. 그러면 인덱스가 더 이상 실제 테이블 위치를 정확히 가리킨다고 보장할 수 없다.

이때 Oracle은 해당 인덱스를 신뢰할 수 없는 상태, 즉 UNUSABLE로 만들 수 있다. 이후에는 인덱스를 이용한 빠른 조회가 안 되거나, DML/쿼리에서 문제가 생길 수 있어 REBUILD가 필요하다. 
ALTER INDEX orders_gix_01 REBUILD ONLINE;

문제는 rebuild가 대형 테이블에서는 몇 분이 아니라 몇 시간짜리 작업이 될 수 있다는 점이다. 이 동안 대량 I/O, 배치 지연, 서비스 성능 저하, 야간작업 실패가 발생할 수 있다.

운영 지옥은 “파티션을 지웠더니 인덱스가 unusable이 되었고, rebuild가 오래 걸려 다음 배치와 서비스까지 밀리는 상황”에서 발생한다.

8. 조회 패턴에 따른 전략

조회/운영 패턴권장 전략이유
대부분 날짜 범위로 조회하고 월별 삭제/보관 필요 Range Partition + Local Prefixed Index Partition Pruning과 파티션 단위 관리가 모두 자연스럽다.
테이블은 날짜로 보관하지만 조회는 고객ID 중심 Range Partition + Local Nonprefixed Index 또는 필요 시 Nonpartitioned Index 인덱스 선두 컬럼을 실제 access path인 고객ID에 맞춘다.
주문번호, 거래번호처럼 전체 기간에서 단건을 빠르게 찾아야 함 Global Prefixed Index 또는 Nonpartitioned Index 검토 모든 파티션을 뒤지지 않고 전역 키 검색을 빠르게 처리할 수 있다.
월별 파티션 drop/exchange가 매우 빈번 Local Index 우선 Global Index unusable/rebuild 부담을 피한다.
insert hot spot, 특정 키 집중, 병렬 분산이 중요 Hash 또는 Composite Partition 검토 데이터와 작업 부하를 여러 파티션으로 분산한다.

Global Prefixed Index는 필수인가?

아니다. Global Index는 기본 중의 기본으로 항상 만들어야 하는 설정이 아니다. 특정 키로 전체 파티션을 가로질러 빠르게 찾아야 하는 강한 요구가 있을 때 선택한다. 대신 파티션 관리 작업 시 unusable/rebuild 부담을 감수해야 한다.

Local Index는 필수인가?

파티션 테이블의 일반적인 운영 기준에서는 Local Index가 기본 선택지에 가깝다. 하지만 조회 패턴에 따라 Local Nonprefixed나 Nonpartitioned Index를 추가로 고려할 수 있다.

Nonpartitioned Index는 특수한가?

파티션 테이블 관점에서는 특수 케이스에 가깝지만, OLTP에서는 충분히 현실적이다. 특히 파티션 키 없이 고객ID, 주문번호, 상태값 등으로 자주 조회하거나 정렬 대체 효과가 필요한 경우 고려한다.

9. 주기적 삭제·보관·파티션 관리 전략

대용량 운영 시스템에서 파티션은 보통 “주기적 데이터 생명주기 관리”와 같이 설계한다.

1 신규 월/일 파티션을 미리 생성한다.
2 현재 데이터는 최신 파티션에 계속 insert된다.
3 보관 기간이 지난 파티션은 백업 또는 archive 대상으로 분리한다.
4 서비스 테이블에서는 오래된 파티션을 drop 또는 truncate한다.
5 Global/Nonpartitioned Index가 있다면 unusable 여부와 rebuild 계획을 확인한다.
-- 신규 파티션 추가 예시
ALTER TABLE orders ADD PARTITION p202404
VALUES LESS THAN (DATE '2024-05-01');

-- 오래된 파티션 삭제 예시
ALTER TABLE orders DROP PARTITION p202301;

운영에서는 단순히 SQL이 맞는지가 아니라, 이 작업이 야간 배치 시간 안에 끝나는지, redo/archive가 얼마나 생기는지, 인덱스 rebuild가 필요한지, 백업 정책과 충돌하지 않는지를 함께 본다.

10. DBA가 실제로 세우는 판단 규칙

DBA가 피하고 싶어하는 상황

  • 대량 DELETE로 undo/redo/archive가 폭발하는 상황
  • 파티션 drop 후 Global Index가 unusable 되는 상황
  • 인덱스 rebuild가 야간 배치 시간을 초과하는 상황
  • 파티션 키에 함수가 걸려 Partition Pruning이 안 되는 상황
  • 파티션을 너무 잘게 쪼개 dictionary/parse 부담이 커지는 상황
  • 파티션을 너무 크게 잡아 drop/backup/archive 단위가 무거워지는 상황

설계 시 먼저 묻는 질문

  • 데이터는 언제, 어떤 단위로 삭제되는가?
  • 보관 기간은 몇 개월/몇 년인가?
  • 조회 조건은 날짜 중심인가, 업무키 중심인가?
  • 월별 drop/exchange가 필요한가?
  • Global Index rebuild를 감당할 운영 시간이 있는가?
  • 배치와 서비스 조회가 같은 시간대에 충돌하는가?
일반적인 결론:
Range Partition + Local Index를 기본 선택지로 두고, 전체 기간 단건 검색이 강하게 필요한 경우에만 Global Index 또는 Nonpartitioned Index를 추가 검토한다.

최종 암기 문장

파티션 키는 삭제·보관·관리 기준이다.
인덱스 선두 컬럼은 실제 조회 접근 경로 기준이다.
Local은 운영이 편하다.
Global은 조회가 강할 수 있지만 운영 부담이 있다.
Prefixed는 날짜 범위 조회에 자연스럽지만 절대 규칙은 아니다.
Nonprefixed/Nonpartitioned는 특수하지만 OLTP에서는 충분히 실전적이다.
