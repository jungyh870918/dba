Oracle DBA 병목·압박·해결 가이드

logical read / physical read / redo / parse / checkpoint / 용량 / 실행계획 / 병목 해소를 실무 사례 중심으로 정리

1단계 · 먼저 보는 그림

압박(pressure)이라는 말은 “다음 단계가 막히는 상태”를 뜻한다

트랜잭션 많음 → redo 빨리 생성 → LGWR / redo file / checkpoint 압박 조회 많음 → logical read 증가 / physical read 증가 → CPU 또는 디스크 압박 정렬·해시 많음 → PGA 부족 → temp tablespace 사용 증가 세그먼트 / datafile 공간 부족 → tablespace 확장 필요 → 배치 실패 / insert 실패
핵심: 압박은 “뭔가 사용량이 높다” 수준이 아니라, 밀린 작업 때문에 다음 단계가 진행되지 않는 상태다. dirty buffer를 못 내리면 DBWR 압박, redo를 재사용 못 하면 checkpoint 압박, commit이 밀리면 LGWR 압박처럼 해석한다.
2단계 · v$sysstat 보는 법

한 번 찍는 게 아니라, 시간 차이로 증가량을 본다

t1 시점 수집 → 5초 또는 10초 대기 → t2 시점 수집 → 증가량(delta) 계산 → 가장 많이 증가한 자원이 현재 병목 후보
지표 증가가 크면 의미 의심 방향
consistent gets / db block gets logical read 증가 CPU, 버퍼 캐시, 비효율 SQL
physical reads 디스크에서 직접 읽음 디스크 I/O, Full Scan, 캐시 미스
redo size 변경량 급증 LGWR, commit, log switch, checkpoint
parse count (hard) 하드 파싱 과다 Shared Pool, 바인드 변수 미사용
실무 포인트: select * from v$sysstat; 한 번으로 병목이 보이진 않는다. 증가량을 봐야 한다. 예를 들어 physical read가 5초 동안 20만 증가했다면, “디스크를 지금 엄청 읽고 있다”는 의미다.
3단계 · logical read 케이스

logical read가 많으면 CPU / buffer cache / 비효율 SQL을 의심한다

시점 consistent gets physical reads
t1 5,000,000 100,000
t2 6,500,000 120,000
해석: 5초 동안 logical read는 150만 증가했고 physical read는 2만만 증가했다. 즉, 디스크보다 메모리에서 계속 읽고 반복 계산하는 성격이 강하다. 이건 CPU 병목 또는 비효율 SQL 가능성이 높다.

흔한 실제 사례

  • 조인이 많은 조회 화면
  • 필터 조건 없이 대량 데이터 조회
  • 루프 안에서 반복적으로 같은 SQL 호출

해결 방향

  • 실행계획 확인
  • 불필요한 조인 제거
  • SELECT * 지양, 필요한 컬럼만
  • 조건식 / 조인 순서 / 인덱스 점검
4단계 · physical read 케이스

physical read가 많으면 디스크 I/O 병목을 먼저 본다

시점 physical reads consistent gets
t1 1,000,000 2,000,000
t2 1,200,000 2,050,000
해석: 5초 동안 physical read가 20만 늘었다. 이건 캐시보다 디스크를 계속 두드리고 있다는 뜻이다. 인덱스 없음, Full Table Scan, 잘못된 실행계획, 캐시에 없는 대량 읽기를 먼저 의심한다.
흔한 실제 사례: 날짜 조건에 함수가 걸려 인덱스를 못 타는 조회, 통계가 오래돼 옵티마이저가 풀스캔을 선택한 배치, 대량 리포트 조회 같은 경우가 대표적이다.
해결 방향: 실행계획 확인 → 조건 컬럼 인덱스 검토 → 함수 제거 / 범위조건으로 변경 → 통계 갱신 → 대량 조회면 direct path read 성격까지 함께 판단.
5단계 · redo size 케이스

redo size가 급증하면 DML 폭증과 commit 압박을 본다

시점 redo size
t1 500MB
t2 900MB
해석: 5초 동안 400MB의 redo가 생겼다. 이건 대량 INSERT, UPDATE, DELETE 또는 인덱스가 많은 테이블에 대한 변경 작업이 집중된 상황일 수 있다.

흔한 실제 사례

  • 새벽 대량 적재 배치
  • 대량 UPDATE 배치
  • 인덱스가 많은 테이블에 대한 대량 INSERT

해결 방향

  • commit 빈도 재조정
  • 배치 시간 분산
  • redo log size / group 수 검토
  • NOLOGGING 가능 여부는 매우 신중하게 판단
실무 감각: redo size 자체가 나쁜 건 아니다. 문제는 redo 생성 속도가 checkpoint와 log 재사용 속도를 이겨버릴 때다. 이때 switch가 잦아지고 checkpoint not complete 같은 현상으로 이어질 수 있다.
6단계 · parse count 케이스

하드 파싱이 많으면 Shared Pool과 SQL 작성 방식을 본다

시점 parse count (hard)
t1 10,000
t2 20,000
해석: 같은 시간 동안 하드 파싱이 1만 증가했다면 SQL을 매번 새로 컴파일하고 있다는 뜻이다. 바인드 변수 미사용, SQL 문자열 매번 변경, Shared Pool 비효율을 의심한다.
-- 나쁜 예 WHERE user_id = 1001 WHERE user_id = 1002 WHERE user_id = 1003 -- 좋은 예 WHERE user_id = :user_id
해결 방향: 바인드 변수 적용, 커서 재사용, 동적 SQL 남용 억제, Shared Pool 크기 / 파싱 구조 점검.
7단계 · 압박 종류별 실제 사례

병목은 “어디가 느린지”가 아니라 “무엇이 다음 단계 진행을 막는지”로 읽어야 한다

압박 종류 대표 현상 흔한 실제 사례 1차 대응
DBWR / Dirty Buffer 압박 free buffer 부족, 쓰기 지연 대량 변경 후 버퍼 캐시 dirty 과다 write workload 분산, checkpoint / I/O 확인
LGWR / Commit 압박 commit 느림, log file sync 대기 대량 DML 배치, 커밋 과다 commit 빈도 조정, redo 구조 확인
Checkpoint 압박 log switch 잦음, checkpoint not complete redo log 너무 작음, 변경량 폭증 redo size / group 수 재검토
CPU 압박 logical read 폭증, CPU 100% 과도한 조인, 비효율 SQL 실행계획 / 조인 구조 점검
디스크 I/O 압박 physical read 폭증 풀스캔, 인덱스 미비 인덱스 / 통계 / 액세스 경로 점검
PGA / TEMP 압박 정렬·해시 느림, temp 사용 증가 대량 sort, hash join, 병렬 배치 PGA 정책 / temp TS / SQL 구조 점검
Tablespace 용량 압박 확장 실패, insert 실패 datafile 꽉 참, temp 부족, undo 부족 증설, autoextend 정책 검토, 사용량 분석
8단계 · 실행계획 파악

실행계획을 볼 때는 “접근 방식”과 “조인 방식”을 본다

1. 느린 쿼리 발견 2. 실행계획 확인 3. Full Table Scan / Join 방식 / 예상 rows / 실제 rows 비교 4. 병목 구간 찾기 5. 인덱스 / SQL / 통계 / 힌트 순으로 조정

가장 먼저 보는 것

  • TABLE ACCESS FULL 많나?
  • 조인 방식이 Nested Loop인지 Hash Join인지
  • 옵티마이저 예상 rows와 실제 rows 차이가 큰지
  • 필터가 인덱스를 못 타는 형태인지

실무 쿼리

  • DBMS_XPLAN.DISPLAY_CURSOR
  • AUTOTRACE
  • 10046 trace + tkprof
핵심: 실행계획 튜닝은 코드를 예쁘게 바꾸는 게 아니라 데이터 접근 방식을 바꾸는 작업이다.
9단계 · 과도한 조인과 병목 해소

과도한 조인은 CPU와 I/O를 동시에 올릴 수 있다

예시 orders o JOIN users u ON o.user_id = u.id JOIN products p ON o.product_id = p.id JOIN categories c ON p.category_id = c.id JOIN stores s ON o.store_id = s.id JOIN regions r ON s.region_id = r.id
문제: join이 많다고 무조건 나쁜 건 아니지만, 대용량 테이블에 불필요한 join이 붙으면 logical read와 physical read가 모두 증가하고, Nested Loop가 대량 데이터에 걸리면 CPU와 I/O를 동시에 잡아먹는다.

흔한 해결 방법

  • 정말 필요한 테이블만 join
  • SELECT * 대신 필요한 컬럼만
  • 조인 컬럼 인덱스 점검
  • 필요 시 EXISTS, 집계 선처리, 중간 결과 축소

추가 조정

  • 통계 갱신으로 잘못된 계획 교정
  • 최후 수단으로 힌트 검토
  • NL보다 Hash Join이 나은 대량 처리인지 판단
10단계 · 용량 확보와 운영

실무에서 흔한 용량 확보 시나리오

Users / Data TS 부족

  • insert 중 ORA-01653
  • datafile resize 또는 add datafile
  • segment 사용량 분석 후 증설

TEMP TS 부족

  • 대량 sort, hash join, 배치
  • temp datafile 증설
  • 배치용 계정 전용 temp TS 분리 검토

UNDO TS 부족

  • snapshot too old
  • undo retention과 undo TS 크기 함께 조정
  • 보통 retention 3600초를 출발점으로 많이 본다

Autoextend 정책

  • 무제한 자동확장은 위험
  • 상한(maxsize) 명확히 두기
  • 직접 관리할지 자동확장에 맡길지 운영 철학 필요
11단계 · Direct Path Read 흔한 사례

배치·병렬·대용량 스캔에서 자주 등장한다

대표 사례: 월말 정산 배치, 대량 통계 집계, 리포트 생성, 병렬 full scan 같은 작업은 결과 집합이 너무 커서 버퍼 캐시에 매번 올리는 게 비효율적이다. 이런 경우 direct path read가 흔히 등장한다.
대용량 테이블 full scan → cache 오염시키지 않기 위해 direct path read 선택 → 필요 시 dirty buffer 정리 후 진행 → 배치 종료 후 일반 OLTP 트래픽과 분리 관찰
12단계 · 실제 장애 시나리오

자주 겪는 실무 장애를 원인과 해소로 연결

시나리오 1 · 조회가 갑자기 느려짐

증상: physical read 폭증

원인: 인덱스 미사용, 풀스캔, 통계 불일치

해소: 실행계획 확인, 인덱스 검토, SQL 조건식 수정

시나리오 2 · CPU 100%

증상: logical read 급증

원인: 과도한 조인, 비효율 반복 SQL

해소: 조인 구조 축소, 중간 결과 줄이기, SQL 튜닝

시나리오 3 · commit이 느림

증상: redo size 급증, log 관련 대기

원인: DML 폭증, log switch 잦음

해소: batch commit 조정, redo log size / group 재검토

시나리오 4 · 배치 중간 실패

증상: temp 부족 또는 undo 부족

원인: sort / hash join / 긴 select

해소: temp TS 증설, undo retention / TS 크기 조정

시나리오 5 · DB 멈춘 것처럼 보임

증상: switch는 나는데 재사용 못함

원인: checkpoint not complete

해소: redo log size 증설, checkpoint 부담 완화

시나리오 6 · DB가 안 올라옴

증상: control file / redo 경로 문제

원인: 파라미터 값과 실제 파일 불일치

해소: 경로 재확인, 실제 파일 복사, 적절한 기동 단계에서 수정

13단계 · 최종 요약

이 문서를 어떻게 써먹으면 좋은가

1. 느리다 / 멈춘다 / 실패한다 라는 말이 나오면 먼저 pressure 종류를 분류한다. 2. v$sysstat은 한 번 찍지 말고 시간 차이로 증가량을 본다. 3. logical read / physical read / redo / parse 중 무엇이 가장 많이 증가했는지로 방향을 잡는다. 4. 그 다음 실행계획, 조인 구조, 인덱스, 통계, commit 패턴, redo 구조, tablespace 용량을 본다. 5. 해결은 “느린 부분을 손으로 때운다”가 아니라 데이터 접근 방식, 로그 생성 패턴, 파일/공간 정책을 바꾸는 방향으로 간다. 결국 실무 DBA 관점의 병목 해소는 CPU, 디스크, redo, temp, undo, 용량을 분리해서 읽는 능력에서 시작한다.
