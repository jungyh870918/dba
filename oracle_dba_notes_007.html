범위 개요

이번 범위의 정체

이번 구간은 2장 DB File 구조의 시작부이며, 실질적으로는 Control File에 집중된 범위다. 61페이지는 장 전환 페이지 성격이 강하고, 실제 학습 포인트는 62~69페이지에서 전개된다.

핵심은 단순히 “Control File이 무엇인가”를 외우는 것이 아니라, DB가 기동될 때 왜 먼저 필요하며, DBA가 실제로 어떻게 추가·관리하는가를 흐름으로 이해하는 것이다.
메모리 / 프로세스 중심 설명 → 이제 디스크 파일 구조 설명으로 이동 → 그 시작점이 Control File
62페이지

아키텍처 그림이 다시 나오는 이유

62페이지에서는 Oracle 전체 아키텍처 그림이 다시 나온다. 여기서 보이는 핵심 파일은 다음 세 가지다.

  • Datafiles
  • Controlfiles
  • Redo Log files
이 페이지의 의미는 “이제부터는 SGA, PGA, 프로세스 같은 메모리/실행 구조가 아니라 실제 DB를 구성하는 디스크 파일 구조를 본다”는 관점 전환이다.
63페이지

Control File의 가장 중요한 정의

63페이지 핵심 내용은 다음과 같다.

  • Database를 기동해서 Mount 상태로 만들기 위해 Control File을 읽는다.
  • Control File 위치는 parameter file의 control_files에 있다.
  • Control File은 보통 2개 이상 존재한다.
  • 서로 다른 디스크에 위치시켜 안정성을 확보한다.
DB startup → parameter file 확인 → control_files 경로 확인 → Control File 읽기 → Mount 상태 진입
즉, Control File은 그냥 존재하는 파일이 아니라 DB를 Mount 상태로 올리기 위한 기준 파일이다. 이 파일이 없으면 Oracle은 DB 구조를 파악하지 못하므로 mount 단계에 제대로 들어갈 수 없다.
여기서 잡아야 할 핵심 사고는 “Control File = DB 구조를 정의하는 기준 파일”이라는 점이다. datafile보다 먼저 필요하다는 점이 중요하다.
64페이지

Control File 안에 실제로 들어 있는 내용

64페이지에서는 Control File에 저장되는 대표 정보가 제시된다.

구조 정보

  • Database Name
  • 테이블스페이스 정보
  • 데이터 파일 목록

복구 기준 정보

  • Checkpoint SCN
  • 기타 Record 정보
이걸 DBA 관점으로 해석하면, Control File은 DB 전체 구조와 복구 기준점을 기억하는 중앙 메타데이터 저장소다.
“이 DB 이름은 무엇인가?” → “어떤 datafile이 붙어 있는가?” → “checkpoint는 어디까지 진행됐는가?” → 이 질문들의 답을 Control File이 들고 있다
63~64페이지 확장 이해

왜 Control File이 그렇게 중요한가

슬라이드 본문에는 직접 길게 써 있지 않지만, 앞의 문장들로부터 반드시 이해해야 하는 흐름이 있다.

Mount 단계 → Datafile을 본격적으로 읽기 전 → Redo를 적용하기 전 → 먼저 Control File로 DB의 구조를 인식
즉, Control File은 DB의 지도(map) 역할을 한다. Oracle은 디스크에 datafile이 있다는 사실만으로 DB를 여는 것이 아니라, Control File을 통해 “이 DB가 어떤 구조인지”를 먼저 파악하고 다음 단계로 넘어간다.
63페이지

미러링을 왜 하는가

63페이지에서는 Control File이 보통 2개 이상 존재하고, 서로 다른 디스크에 위치한다고 설명한다.

control01.ctl 손상 → DB 구조 인식 실패 위험 → mount 실패 가능 → DB 기동 문제
Control File은 기동의 출발점이므로, 하나만 두면 위험하다. 그래서 Oracle은 다음과 같이 여러 개의 Control File을 두고 복제해 둔다.
control01.ctl control02.ctl control03.ctl
이 범위에서 미러링은 단순한 백업 개념이 아니라, 기동 안정성 확보를 위한 기본 운영 전략으로 이해해야 한다.
확인용 뷰

DBA가 실제로 확인하는 뷰

이 범위에서는 Control File 관련 정보를 확인하기 위한 뷰도 같이 제시한다.

  • v$controlfile : 현재 control file 경로와 엔트리 확인
  • v$controlfile_record_section : control file 내부에 어떤 정보가 저장되는지 확인
즉, 이 장은 개념 설명에서 끝나지 않고, “실제로 어디서 확인할 것인가”까지 함께 가르치는 실무 입문 구간이다.
66~67페이지

현재 control_files 경로 조회

슬라이드에서는 먼저 아래 SQL로 현재 설정을 조회한다.

SELECT name, value FROM v$parameter WHERE name = 'control_files';

이 조회 결과를 통해 현재 Control File이 어떤 경로에 잡혀 있는지 확인한다. 예시에서는 기존 control01.ctl, control02.ctl 경로가 나타난다.

여기서 중요한 건 DBA가 막연히 “있겠지”라고 생각하지 않고, 반드시 현재 설정과 실제 경로를 먼저 확인한다는 점이다.
67페이지

control_files 파라미터 수정

이후 슬라이드에서는 새 Control File 경로를 추가하는 예시를 보여준다.

ALTER SYSTEM SET control_files = 'C:\app\...\control01.ctl', 'C:\app\...\control02.ctl', 'C:\TMP\control03.ctl' SCOPE = SPFILE;

그리고 결과에서 ISSYS_MODIFIABLE = FALSE 성격을 보여 주며, 수정 내용이 즉시 현재 시스템에 반영되지 않음을 확인시킨다.

여기서 잡아야 할 핵심은 spfile 변경 = 설정 저장이지, 현재 떠 있는 인스턴스가 즉시 그 설정으로 바뀌는 것은 아니라는 점이다.
control_files는 정적(static) 성격의 파라미터로 봐야 한다. 그래서 재기동 없이 바로 반영되는 종류가 아니다.
절차 핵심

파라미터 수정만으로 끝나지 않는 이유

슬라이드 흐름상 가장 중요한 메시지는 이것이다.

control_files를 바꾸는 일은 문자열만 바꾸는 작업이 아니다.

ALTER SYSTEM으로 바꾸는 것은 어디까지나 경로 정보를 수정하는 일이다. 하지만 Oracle이 실제로 그 경로의 Control File을 읽으려면, OS 상에 그 파일이 존재해야 한다.

parameter 수정 → shutdown → 기존 control file을 새 경로로 실제 복사 → startup → 다시 조회해서 반영 확인
즉, Parameter ≠ 실제 파일이다. 설정과 파일 복사는 별개의 작업이다.
“ALTER SYSTEM 하면 파일도 자동으로 생긴다”라고 이해하면 안 된다. 파일은 직접 복사해야 한다.
69페이지

최종 반영 확인

마지막에는 다시 v$parameter를 조회하여 새 경로가 실제 설정에 포함되었는지 확인한다.

SELECT name, value FROM v$parameter WHERE name = 'control_files'; → control01.ctl → control02.ctl → control03.ctl
이 단계까지 와야 비로소 Control File 추가 작업이 완료된 것이다. 즉, 경로가 설정에 보이고, DB가 그 설정으로 다시 기동되어야 정상 반영이라고 볼 수 있다.
전체 흐름

이번 범위의 핵심 플로우

DB startup → control_files 확인 → Control File 읽기 → DB 구조 인식 → Mount 상태 진입 Control File 추가 작업 현재 control_files 조회 → control_files 파라미터 수정 (spfile) → shutdown → control file 실제 복사 → startup → 다시 조회해서 반영 확인
이 범위는 결국 “Control File이 없으면 mount가 안 되고, Control File 변경 작업은 순서 기반으로 진행해야 한다”는 점을 가르친다.
마무리 요약

61~69페이지 한 줄 정리

Control File은 DB를 Mount 상태로 만들기 위한 기준 파일이며, 경로 변경이나 추가는 파라미터 수정 + 실제 파일 복사 + 재기동이 반드시 함께 이루어져야 한다.

다음 범위로 넘어갈 때도 같은 방식으로, 페이지 범위 안의 실제 내용과 그 의미를 함께 묶어 정리하면 된다.
