394~395p1. Direct Path I/O 개요

일반적인 Oracle 블록 I/O는 DB Buffer Cache를 경유한다. 읽을 때는 먼저 Buffer Cache를 확인하고, 없으면 데이터 파일에서 읽어 캐시에 올린다. 변경 작업도 서버 프로세스가 직접 데이터 파일에 쓰는 것이 아니라 DB Buffer Cache에 변경 내용을 반영하고, 이후 DBWR이 비동기적으로 데이터 파일에 기록한다.

일반 Read
Server Process → Buffer Cache 확인 → 없으면 Data File Read → Cache 적재
일반 DML
Server Process → Buffer Cache 변경 → Dirty Buffer 생성 → DBWR가 나중에 Write
목적
자주 읽는 블록을 캐시에 보관해 시스템 전체 I/O 효율 향상
쉽게 말하면
일반 I/O는 마트 창고처럼 Buffer Cache를 중간에 둔다. 여러 사용자가 같은 블록을 반복해서 읽으면 디스크를 매번 가지 않아도 되므로 효율적이다. 하지만 대량 적재처럼 한 번 넣고 끝나는 작업에서는 이 캐시 경유가 오히려 비용이 될 수 있다.

396p2. Direct Path I/O 작동 요건

Direct Path I/O는 Buffer Cache를 일반적인 방식으로 경유하지 않고, 대량 처리나 특수 작업에서 데이터 파일 또는 Temp 영역과 직접 I/O하는 방식이다. 교재는 다음 상황을 Direct Path I/O 작동 요건으로 정리한다.

상황성격실무 해석
병렬 쿼리로 Full Scan 수행Parallel Full Scan - Read대량 읽기에서 Buffer Cache 오염을 줄이고 병렬 처리 효율을 높임
병렬 DML 수행Parallel DML - Write대량 insert/update 작업에서 병렬 쓰기 경로 사용
Temp 세그먼트 블록 Read/WriteTemp I/OSort/Hash가 PGA를 넘으면 Temp Tablespace에 direct path read/write temp 발생
expdp 수행Direct Export - ReadData Pump Export가 대량 데이터를 직접 읽는 상황
impdp 또는 Direct Path Insert 수행Direct Path Insert - Write대량 적재 시 HWM 위로 직접 적재
nocache LOB 컬럼 읽기LOB nocache캐시에 올리지 않는 LOB 데이터 접근
주의
Direct Path I/O는 “항상 좋은 빠른 모드”가 아니다. OLTP의 작은 트랜잭션에는 Buffer Cache를 쓰는 일반 경로가 더 자연스럽고 안전하다. Direct Path는 대량 처리, 병렬 처리, export/import, CTAS 같은 작업에서 효과가 크다.

397p3. Direct Path I/O 관련 Wait Event

Direct Path I/O는 Wait Event에서도 흔적이 보인다. 특히 Temp 공간을 사용하는 정렬/해시 작업이나 병렬 Full Scan, Data Pump, Direct Path Insert에서 자주 관찰된다.

Wait Event발생 상황운영 해석

direct path write tempSort/Hash 작업 중 Temp Tablespace에 쓸 때PGA 부족, 큰 정렬, 큰 해시 조인, 병렬 작업 확인

Direct Path WriteAPPEND Insert, 병렬 Insert, SQL*Loader direct, CTAS, Data Pump Import대량 입력 작업. Logging/Nologging, Lock, 복구 가능성 확인
병렬 쿼리와 Checkpoint
교재는 병렬 쿼리 전 Disk와 Memory 동기화를 위해 Checkpoint가 발생한다고 설명한다. 다만 10gR2 이후부터는 병렬 쿼리와 관련된 세그먼트만 동기화하는 방식으로 범위가 줄어들었다.

398~399p4. Direct Path Insert 성능 구조

일반 Heap Table Insert는 HWM 아래쪽의 Free Block을 찾아 데이터를 넣는다. Buffer Cache에서 블록을 찾고, 없으면 데이터 파일에서 읽어 캐시에 올린 뒤 변경한다. 일반 트랜잭션에서는 이 방식이 효율적이다. 하지만 대량 Bulk Insert에서는 매번 Free Block을 찾고 Buffer Cache를 경유하는 비용이 커진다.

구분일반 InsertDirect Path Insert
입력 위치HWM 아래 Free Block 검색 후 입력Free Block을 찾지 않고 HWM 상위 영역에 입력
Buffer CacheBuffer Cache를 경유Buffer Cache를 우회하여 데이터 파일에 직접 입력
Undo일반적으로 undo 생성교재 기준: 사용자가 commit할 때 HWM만 상향 조정하면 되므로 undo 사용이 불필요
적합한 작업작은 OLTP 트랜잭션대량 Insert, CTAS, Data Load
주의점안정적이고 일반적Table Lock, 복구 가능성, Logging 설정 주의
HWM 감각
HWM은 테이블이 사용한 블록의 높은 경계선이다. 일반 Insert는 그 아래의 빈 공간을 재사용하려고 찾고, Direct Path Insert는 그 과정을 생략하고 HWM 위쪽에 대량으로 밀어 넣는다. 그래서 빠르지만 일반 OLTP 방식과는 다르게 다뤄야 한다.

400~403p5. CTAS & INTAS: APPEND / NOLOGGING

Direct Path Insert 방식은 여러 경로로 사용된다. 대표적으로 insert /*+ append */, 병렬 Insert, SQL*Loader direct 옵션, CTAS가 있다.

Direct Path Insert 방식

  • Insert Select 문장에 /*+ append */ 힌트 사용
  • 병렬 모드로 Insert
  • SQL*Loader에서 direct 옵션 지정
  • CTAS: create table ... as select ...

NOLOGGING의 의미

테이블 속성을 nologging으로 변경하면 Direct Path Insert 상황에서 Redo Log를 최소화할 수 있다. 교재는 이때 Data Dictionary 변경에만 로깅이 발생한다고 설명한다. 단, 일반적인 Insert를 로깅하지 않는 방법은 아니며, Direct Path Insert일 때만 작동한다.

-- CTAS + NOLOGGING
create table edu01.t_cust_ctas nologging as
select *
from   t_cust
where  join_dt >= '20220101';

-- NOLOGGING 상태 확인
select owner, table_name, logging
from   dba_tables
where  table_name = 'T_CUST_CTAS';

-- APPEND 힌트로 Direct Mode Insert
insert /*+ append */
into edu01.t_cust_ctas
select *
from   t_cust
where  join_dt between '20210101' and '20211231';

commit;

-- 작업 후 다시 LOGGING 모드로 변경
alter table edu01.t_cust_ctas logging;
NOLOGGING의 운영 리스크
NOLOGGING은 redo를 줄여서 빠르지만, 입력한 데이터가 장애 발생 시 복구되지 않을 수 있다. 특히 운영 DB에서 NOLOGGING으로 대량 적재한 뒤 백업을 바로 수행하지 않으면 복구 시 데이터 파일과 redo의 정합성 문제가 생길 수 있다.
Exclusive Table Lock
Direct Path Insert는 일반적인 동시 OLTP Insert와 다르게 Exclusive mode Table Lock 이슈가 생길 수 있다. 서비스 중인 테이블에 무심코 APPEND/NOLOGGING을 쓰면 다른 DML과 충돌할 수 있다.

실무 패턴

Use Case전략주의점
대량 이관/초기 적재CTAS 또는 APPEND + NOLOGGING 검토작업 후 LOGGING 복구, 백업 수행
배치용 임시 테이블 생성CTAS로 빠르게 생성통계정보 수집 여부 확인
운영 테이블에 대량 적재서비스 시간 외 수행Lock, Redo, Archive, Backup 계획 필요
복구 중요 데이터LOGGING 유지속도보다 복구 가능성이 우선

404~411p6. RAC Cache Fusion

RAC에서는 여러 인스턴스가 하나의 데이터베이스를 공유한다. 같은 블록을 여러 노드가 읽거나 쓰려면, 단순히 디스크에서 각자 읽는 것이 아니라 노드 간 블록 상태와 권한을 조율해야 한다. 이때 노드 간 메모리로 블록을 주고받는 메커니즘이 Cache Fusion이다.

Dynamic Remastering

411페이지는 Resource Affinity에 따라 마스터 노드가 동적으로 변경될 수 있음을 설명한다. A 노드가 특정 리소스를 반복 요청하면, 어느 순간부터 그 리소스의 마스터 노드가 A로 바뀔 수 있다. 이를 Dynamic Remastering이라고 한다.

SQL 튜닝과 Cache Fusion 감소
교재는 SQL 튜닝을 통해 블록 Logical Read 자체를 줄이면 RAC Cache Fusion도 감소한다고 정리한다. 즉 RAC 성능 문제를 단순히 네트워크 문제로만 보면 안 된다. SQL이 불필요하게 많은 블록을 읽으면 그만큼 노드 간 블록 조율과 전송도 늘어난다.

추가 심화 정리 · Checkpoint / RAC Node / NOLOGGING 운영 감각

Checkpoint는 Redo Flush가 아니라 Dirty Buffer를 Data File에 반영하는 작업이다. LGWR는 Redo Buffer를 Online Redo Log에 기록하고, DBWR는 Buffer Cache의 Dirty Buffer를 Data File에 기록한다. Undo block도 결국 DBWR 대상이다.

핵심
Redo Flush ≠ Checkpoint

병렬 쿼리 전 “전체 checkpoint”가 비싸다는 것은, 특정 대용량 테이블 하나를 읽기 위해 unrelated object까지 광범위하게 flush 비용이 발생할 수 있다는 의미다. 10gR2 이후에는 쿼리 대상 object 기준으로 관련 segment만 flush한다.

RAC의 Node는 각 DB Instance(실제 서버)이다. Node A/B/C는 각자 SGA와 Background Process를 가지지만 같은 Database와 Datafile을 공유한다.

모든 Node가 같은 row / 같은 index leaf / 같은 block을 반복해서 수정하면 hot block이 발생하고 gc cr request, gc current grant busy 같은 wait event가 증가한다.

DBA는 Cache Fusion을 끄는 것이 아니라 싸움이 덜 나게 설계한다. reverse key index, hash partition, service routing, sequence hotspot 완화, SQL 튜닝 등이 대표적이다.

특정 업무를 특정 Node에 고정하는 것은 Service Routing이다.

결제 → Node A
정산 배치 → Node B
조회 서비스 → Node C

Dynamic Remastering은 자동이다. 특정 리소스를 계속 요청하는 Node가 있으면 Oracle이 Resource Master를 자동으로 이동시킨다. DBA가 직접 마스터를 지정하지는 않는다.

NOLOGGING은 redo를 크게 줄여 대량 적재를 빠르게 하지만 장애 발생 시 복구 불가 가능성이 생긴다.

1. NOLOGGING으로 빠르게 적재
2. 바로 BACKUP 수행
3. 다시 LOGGING 복구
백업이 빨라서가 아니라 redo 폭발 + archive 지옥보다 더 낫기 때문에 이렇게 운영한다. OLTP 핵심 테이블은 보통 LOGGING 유지, DW / 초기 적재는 CTAS + APPEND + NOLOGGING이 흔하다.

핵심 한 줄
이 범위의 핵심은 Direct Path I/O를 “빠른 입력 방식”으로만 외우는 것이 아니라, Buffer Cache 우회, HWM 상향, redo 최소화, 복구 리스크, RAC 노드 간 블록 전송 비용까지 함께 보는 것이다.
