Oracle DBA Notes

299~320페이지 정리

Bitmap Index, Function-Based Index, Reverse Key, Descending, Invisible Index와 실제 인덱스 운영 방식, 그리고 AWR / v$sql / Wait Event 관점에서 병목을 판단하는 흐름까지 함께 정리한 실무형 요약본.

이 구간의 핵심

이 범위의 핵심은 “인덱스 종류를 외운다”가 아니다. 실제 업무에서 어떤 조회 패턴에 어떤 인덱스를 써야 하는지, 그리고 DB가 느릴 때 그 원인이 인덱스 부재인지, 잘못된 SQL인지, 아니면 Wait Event 중심의 경합인지까지 함께 판단하는 시각을 배우는 구간이다.

DBA는 인덱스를 많이 만드는 사람이 아니라, 가장 비싼 SQL 하나를 찾아 왜 느린지 죽이는 사람이다.

1. B-Tree Index와 Composite Index

실무에서 가장 많이 쓰는 인덱스는 여전히 B-Tree다. 특히 PK, 주문번호, 회원번호처럼 값이 거의 중복되지 않는 컬럼에 강하다.

where member_id = ? where order_id = ?

그러나 실제 업무 SQL은 단일 조건보다 복합 조건이 훨씬 많다. 그래서 단일 인덱스 여러 개보다 Composite Index 설계가 훨씬 중요하다.

where shop_id = ? and order_date = ? and status = ?
실무 인덱스 설계의 핵심은 “컬럼 수”보다 “컬럼 순서”다.

2. Bitmap Index

Bitmap Index는 gender, status, yn_flag처럼 값 종류가 적고 중복이 많은 컬럼에 적합하다. 비트맵 조합으로 다중 조건 분석 검색을 빠르게 수행할 수 있다.

gender = 'M' and vip = 'Y' and married = 'Y'

하지만 DML에 매우 약하다. 따라서 OLTP에서는 거의 쓰지 않고, 주로 DW / BI / 분석 시스템에서 사용한다.

3. Function-Based Index

컬럼에 함수가 걸리면 일반 인덱스가 잘 안 탄다. 그래서 함수 결과 자체를 인덱싱해야 한다.

where upper(email) = ? where trunc(reg_date) = trunc(sysdate)
create index ix_upper_email on members(upper(email));

운영자 검색, 대소문자 무시 검색, 날짜 자르기 검색 같은 레거시 쿼리에서 매우 자주 등장한다.

4. Reverse Key Index

Sequence 기반 PK는 계속 증가하므로 B-Tree 맨 오른쪽 Leaf Block만 계속 수정하게 된다. 이때 동시 Insert가 몰리면 Hot Block이 발생한다.

1 2 3 4 5 ...

Reverse Key는 숫자를 뒤집어 여러 Leaf로 Insert를 분산시켜 병목을 줄인다.

12345 → 54321 12346 → 64321

다만 범위 검색과 정렬에는 불리하다. 즉, Insert 성능을 위해 Range Search 성능을 포기하는 성격이 있다.

Reverse Key는 Oracle이 기존 인덱스와 병행 활용하는 자동 옵션이 아니라, 아예 그 인덱스를 Reverse 구조로 설계하는 것이다.

5. Descending Index

최신순 조회가 많은 화면에서 자주 쓴다.

order by reg_date desc fetch first 20 rows only
create index ix_reg_date_desc on t(reg_date desc);

정렬 작업을 줄이고 최신 데이터 조회를 빠르게 해준다. 관리자 페이지, 최근 주문, 최근 가입자 목록에서 자주 등장한다.

6. Invisible Index

인덱스를 바로 drop 하기 무서울 때 쓰는 실무용 도구다.

alter index ix_old_customer_search invisible;

인덱스는 물리적으로 존재하지만 Optimizer는 사용하지 않는다. 즉, “없는 척”하게 만들어 실행계획과 장애 여부를 먼저 본다.

실제 사례 레거시 시스템에서 누가 왜 만들었는지 모르는 인덱스를 바로 지우면 새벽 배치나 운영툴이 깨질 수 있다.
운영 방식 invisible로 먼저 숨김 → 며칠 관찰 → 문제 없으면 drop → 문제 생기면 visible 복구
Drop은 되돌리기 비싸지만, Invisible은 되돌리기 거의 즉시 가능하다.

7. 인덱스가 많다고 좋은 게 아니다

실무에서 “인덱스를 추가한다”는 말은 대부분 Secondary Index를 추가한다는 뜻이다. PK Index는 이미 있는 경우가 많기 때문이다.

insert 1건 → PK index 갱신 → secondary 5개 갱신

인덱스가 많아질수록 Insert / Update / Delete는 느려지고, Redo / Undo도 증가한다.

필요한 인덱스만 유지하는 것이 정답이다.

8. DBA는 어디서 병목을 찾는가

실제 DBA는 컬럼부터 보지 않는다. 가장 먼저 느린 SQL부터 찾는다. 대표적으로 AWR, ASH, v$sql, SQL Monitor에서 아래 항목을 본다.

  • Top SQL: 가장 비싼 SQL
  • Top Wait Events: 가장 많이 기다린 이유
  • Buffer Gets: 메모리(Buffer Cache) 읽기 횟수
  • Physical Reads: 디스크까지 내려간 횟수
  • Executions: 실행 횟수
  • Elapsed Time: 총 실행 시간(CPU + Wait 포함)
select sql_id, executions, elapsed_time, cpu_time, buffer_gets, disk_reads from v$sql order by elapsed_time desc;

핵심은 SQL이 느린 이유가 CPU 때문인지, Disk 때문인지, Lock 때문인지, RAC block transfer 때문인지를 분해해서 보는 것이다.

9. Buffer Gets / Physical Reads / Elapsed Time 해석

Buffer Gets는 Buffer Cache에서 block를 읽은 횟수이고, Physical Reads는 Buffer Cache에 없어 디스크까지 내려간 횟수다.

Buffer Gets ↑ 메모리에서 논리적으로 너무 많이 읽고 있다. CPU 낭비 가능성.
Physical Reads ↑ 디스크 I/O 병목 가능성. 진짜 느린 경우가 많다.
Elapsed Time ↑, CPU 낮음 일한 시간보다 기다린 시간이 더 많다는 뜻이다.

10. Wait Event란 무엇인가

Wait Event는 “DB가 왜 기다렸는가”를 보여주는 기록이다. Oracle은 일한 시간과 기다린 시간을 나눠서 추적한다.

DBA는 SQL이 느리다는 말을 듣고, 먼저 “무엇 때문에 기다렸는가”를 본다.

11. 대표 Wait Event 실제 사례

db file sequential read

single block read다. 주로 인덱스 기반 조회에서 많이 보인다. 인덱스를 통해 한 block씩 랜덤 접근하는 형태라 Physical Read가 많아질 수 있다.

log file sync

Commit 대기다. LGWR가 Redo를 disk에 쓰는 걸 기다린다.

for row in loop insert ... commit; end loop;

이런 코드가 대표적 장애 원인이다.

buffer busy waits

읽는 속도가 느린 게 아니라, 누가 같은 block를 잡고 있어서 기다리는 것이다. 특히 같은 Index Leaf, Segment Header, Undo Block 경합에서 자주 발생한다.

enq: TX - row lock contention

같은 row를 동시에 수정하려 해서 row lock을 기다리는 상황이다.

Session A: update account where id = 1; -- commit 안 함 Session B: update same row -- 대기

gc buffer busy acquire / gc current block busy

RAC에서 다른 인스턴스가 필요한 block를 가지고 있어 기다리는 상황이다. 특히 current block은 최신 수정 가능한 block를 의미하므로 DML에서 더 치명적이다.

12. Hot Block이란 무엇인가

여러 세션이 같은 block만 계속 수정하려 할 때 생기는 병목이다. 대표적으로 Sequence 기반 PK Insert에서 발생한다.

마트 계산대 10개가 있는데 손님이 전부 1번 계산대만 가는 상황과 같다.
insert into orders values(order_seq.nextval, ...)

PK가 계속 증가하면 B-Tree 마지막 Leaf만 계속 수정하게 되고, 그 결과 insert 느림, commit 지연, wait event 급증이 발생한다.

13. CPU는 정상인데 느릴 수 있는 이유

이건 실제로 아주 자주 본다. DB가 느린데 CPU 사용률은 높지 않은 경우다. 보통 DB가 “일을 못 해서”가 아니라 “기다리고 있어서” 느린 것이다.

  • row lock 대기
  • buffer busy waits
  • RAC block transfer 대기
  • log file sync 대기
  • disk read 대기
느린 이유가 CPU인지, 기다림인지부터 구분하는 것이 실무의 출발점이다.

14. RAC 관점

RAC는 여러 서버가 하나의 Database를 동시에 사용하는 구조다. 장점은 장애 대응과 Scale Out이지만, block ownership 싸움이 생긴다.

Node A: 같은 block 수정 중 Node B: 같은 block 필요 → gc current block busy

즉, 내 서버가 느린 게 아니라 다른 인스턴스가 block를 아직 안 넘겨줘서 느릴 수 있다.

15. 한 줄 정리

이 범위의 핵심은 인덱스 종류 암기가 아니라, 어떤 SQL에 어떤 인덱스를 적용해야 하는지와, 느린 이유가 SQL 자체인지 기다림인지 분해해서 보는 DBA 관점을 익히는 데 있다.
