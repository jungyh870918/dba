PDF 직접 내용 · 파라미터 개요

파라미터 파일은 Oracle 인스턴스를 여는 첫 관문이다

파라미터 파일은 Oracle 서버의 각종 옵션을 설정하며, 인스턴스가 기동되는 시점에 읽힌다.

기본 경로

  • Unix 계열 : $ORACLE_HOME/dbs
  • Windows : $ORACLE_HOME/database
parameter file → instance 생성 → SGA / background process 기동 → NOMOUNT 단계 성립
실무 의미: 파라미터 파일이 깨지면 NOMOUNT 자체가 안 된다. 즉, DB를 열기 이전의 “엔진 생성” 단계가 실패한다.
PDF 직접 내용 · 파라미터 변경 성격

온라인 변경 가능 / 불가능 파라미터

파라미터는 온라인 중 바로 바뀌는 것과, DB 재기동 후 반영되는 것이 나뉜다.

온라인 중 변경 가능한 파라미터

  • DB 기동 중 즉시 반영 가능
  • alter system set 파라미터 = 값 scope=both

온라인 중 변경 불가능한 파라미터

  • 파일에는 반영되지만 DB 재기동 후 적용
  • alter system set 파라미터 = 값 scope=spfile
중요: 메모리 관련 파라미터는 즉시 반영이 가능하더라도, 실무에서는 재기동 후 안정 상태로 확인하는 것을 선호한다.
PDF 직접 내용 · v$parameter

v$parameter로 파라미터를 읽을 때 봐야 하는 컬럼

select name, value, isdefault, isses_modifiable, issys_modifiable from v$parameter where name in ('log_archive_start', 'sga_target', 'workarea_size_policy');

예시로 나온 파라미터

  • sga_target : 예시값 1610612736
  • log_archive_start : 예시값 FALSE
  • workarea_size_policy : 예시값 AUTO

TYPE 컬럼 의미

  • 1 = boolean
  • 2 = string
  • 3 = integer
  • 4 = parameter file
  • 5 = reserved
  • 6 = big integer

ISSES_MODIFIABLE

ALTER SESSION으로 변경 가능한지 여부다.

  • true : 변경 가능
  • false : 변경 불가능

ISSYS_MODIFIABLE

ALTER SYSTEM으로 변경 가능한지 여부다.

  • IMMEDIATE : 수정 즉시 적용
  • DEFERRED : 변경은 가능하지만 새로 로그인한 세션부터 적용
  • FALSE : scope=spfile만 가능

ISINSTANCE_MODIFIED

  • MODIFIED : ALTER SESSION으로 수정
  • SYSTEM_MOD : ALTER SYSTEM으로 수정
  • FALSE : 수정되지 않음

ISPDB_MODIFIABLE

CDB와 PDB를 분리해 설정 가능한지 여부다.

  • TRUE : 가능
  • FALSE : 불가능

ISADJUSTED

Oracle이 더 적절한 값으로 조정했는지 여부다. 대부분 FALSE다.

ISDEPRECATED

  • TRUE : 더 이상 권장되지 않음
  • FALSE : 사용 가능

ISINSTANCE_MODIFIABLE (RAC 관점)

  • true : 인스턴스별로 다를 수 있음
  • false : RAC 전체에서 같은 값을 권장
실무 핵심: DBA는 파라미터 값을 “이름과 값”만 보는 게 아니라, 어디서 바뀌는지(세션/시스템), 언제 반영되는지(즉시/재기동), 멀티테넌트에서 분리 가능한지까지 함께 본다.
PDF 직접 내용 · 변경 방법

파라미터 변경 방법 3가지

-- 서버 전체 적용 alter system set 파라미터 = 값; -- 자신의 세션만 적용 alter session set 파라미터 = 값; -- 특정 SQL에만 적용 select /*+ opt_param('파라미터','value') */ ... from ...;
실무 의미: 파라미터는 “전체 시스템에 강하게 박는 방식”만 있는 게 아니다. 세션 단위, SQL 단위로 국소적으로 테스트할 수 있어야 안전하게 운영할 수 있다.
PDF 직접 내용 · PFILE / SPFILE

PFILE과 SPFILE 차이

PFILE

  • Text file
  • 9i 이전 기본
  • 사용자가 editor로 직접 수정 가능
  • 파일명 지정 시 읽힘

SPFILE

  • Binary file
  • 9i부터 관리
  • 기본으로 읽힘
  • alter system set으로 편집

기본 파일명과 위치

  • Windows PFILE : $ORACLE_HOME/database/init인스턴스명.ora
  • Windows SPFILE : $ORACLE_HOME/database/spfile인스턴스명.ora
  • Unix PFILE : $ORACLE_HOME/dbs/init인스턴스명.ora
  • Unix SPFILE : $ORACLE_HOME/dbs/spfile인스턴스명.ora

파일 생성

-- spfile로 기동 중일 때 pfile 생성 create pfile from spfile; -- pfile로 기동 후 spfile 생성 startup pfile='C:\app\seo\product\11.2.0\dbhome_1\database\initORCL.ora'; create spfile from pfile;
실무 핵심: spfile만 믿지 말고 pfile 백업본을 반드시 유지해야 한다. 파라미터 파일 손상 시 NOMOUNT도 못 가기 때문이다.
PDF 직접 내용 · scope

scope = memory | spfile | both 의 의미

alter system set parameter = value [scope=memory | spfile | both];
  • memory : 현재 서버에만 적용, 재기동 후 원래 값으로 기동
  • spfile : spfile에만 적용, DB 재기동 후 반영
  • both : 서버에도 적용되고, spfile에도 기록

undo_retention 예제

select name, value, issys_modifiable from v$parameter where name = 'undo_retention'; -- 예시 결과 undo_retention = 3600 ISSYS_MODIFIABLE = IMMEDIATE alter system set undo_retention = 900 scope=both; alter system set undo_retention = 600 scope=spfile;

scope=spfile로 바꾸면 현재 v$parameter에서 보이는 값은 그대로고, 재기동 후에야 600이 반영된다.

실무 핵심: 많은 사고가 “값을 바꿨다고 생각했는데 현재 메모리에는 안 바뀌어 있는 상태”에서 나온다. scope를 정확히 이해해야 한다.
환경변수

반드시 기억해야 할 환경변수

  • $ORACLE_HOME : Oracle 바이너리와 기본 파일들이 위치하는 홈 디렉터리
  • $ORACLE_SID : 어떤 인스턴스에 접속하고 기동할지 결정하는 식별자
$ORACLE_HOME → pfile / spfile 기본 위치 결정 → sqlplus / startup / listener 관련 경로 해석에 연결 $ORACLE_SID → 어떤 인스턴스를 대상으로 startup / shutdown / local 접속할지 결정
실무 의미: startup 문제의 초입에서 가장 먼저 확인해야 하는 것이 환경변수다. 특히 ORACLE_SID가 틀리면 엉뚱한 인스턴스를 만지거나, 기동 자체가 실패할 수 있다.
파라미터와 메모리 연결

파라미터가 왜 중요한가: 결국 SGA / PGA / startup / 작업 방식과 연결된다

sga_target

SGA 전체를 자동 메모리 관리 관점에서 크게 잡는 기준이다. Shared Pool, Buffer Cache 같은 공용 캐시 구조와 직결된다.

sga_target → Shared Pool / Buffer Cache / Log Buffer 등 → cache hit, physical read, library cache pressure와 연결

workarea_size_policy

sort / hash 같은 작업 영역을 자동으로 관리할지 결정한다. AUTO이면 Oracle이 PGA 작업 영역을 자동 조정한다.

workarea_size_policy = AUTO → sort / hash / workarea → PGA 사용 패턴과 연결

undo_retention

undo를 얼마나 유지하려 하는지에 대한 정책이다. consistent read, snapshot too old, 긴 조회, batch 충돌과 직접 연결된다.

중요: 파라미터는 단순 설정값이 아니라, 메모리 구조, 일관성 보장, startup, 운영 통제, 장애 대응을 모두 묶는 제어점이다.
확장 학습 · 저장 구조

우리가 이 범위에서 함께 확장한 저장 구조 핵심

Tablespace ↓ Datafile ↓ Segment ↓ Extent ↓ Block ↓ Row
  • Block : I/O 최소 단위
  • Extent : Block 묶음
  • Segment : table / index 같은 객체의 저장 영역
  • Tablespace : 논리 저장 단위
중요: Segment는 파일이 아니라 extent들의 집합이다. 따라서 하나의 segment는 여러 datafile에 걸쳐 존재할 수 있고, 하나의 datafile 안에는 여러 segment가 공존할 수 있다.
Segment = Extent들의 모음 Extent는 각각 특정 datafile에 속함 즉 Segment = Extent1(datafile1) + Extent2(datafile2) + Extent3(datafile3)
확장 학습 · Undo 핵심 요약판

이 페이지 안의 undo 섹션은 본문보다 압축 정리다

앞서 별도 HTML에서 undo를 깊게 다뤘지만, 이 범위와 연결되는 핵심만 여기에도 그대로 넣는다.

왜 undo가 필요한가

SELECT가 쿼리 시작 시점 기준으로 일관된 결과를 보게 하기 위해서다. rollback 용도도 있지만, 더 본질적으로는 consistent read용이다.

SCN과 비교

  • Query SCN : 내가 보고 싶은 시점
  • Block SCN : block이 마지막으로 바뀐 시점
Block SCN > Query SCN → 현재 block은 미래 상태 → undo를 따라가며 과거 상태 복원

undo chain

Oracle은 과거 버전을 통째로 저장하는 게 아니라, 현재 상태와 undo chain을 이용해 필요할 때 과거를 만든다.

ORA-01555

필요한 시점까지 따라가야 할 undo chain 중간이 이미 덮어쓰기되어 복원 경로가 끊긴 상태다.

핵심: ORA-01555는 단순히 undo가 부족한 것이 아니라, 필요한 과거 시점까지 되돌리는 체인이 중간에서 끊긴 상태다.
실무 의미

이 범위에서 DBA가 실제로 가져가야 하는 감각

  • 파라미터는 “값”이 아니라 언제, 어디서, 어떻게 반영되는지를 같이 봐야 한다.
  • scope 오해는 기동 후 값 불일치 사고로 이어진다.
  • ORACLE_HOME / ORACLE_SID는 startup / local 접속 / 파일 경로 해석의 출발점이다.
  • sga_target은 SGA 캐시 구조와 연결된다.
  • workarea_size_policy는 sort / hash / PGA와 연결된다.
  • undo_retention은 undo / consistent read / ORA-01555와 연결된다.
  • pfile / spfile 전환 절차를 모르면 parameter 손상 시 NOMOUNT도 못 갈 수 있다.
최종 요약

37~48페이지와 확장 개념을 한 줄로 묶으면

PDF 37~48페이지의 본문 중심은 파라미터 파일이다. 핵심은 파라미터가 단순 설정값이 아니라 instance 생성, 메모리 구조, startup, 세션/시스템 반영 범위, 멀티테넌트 적용 범위, 운영 중 변경 가능 여부를 제어하는 Oracle의 출발점이라는 점이다. v$parameter에서는 isdefault, isses_modifiable, issys_modifiable, ispdb_modifiable, isadjusted, isdeprecated 같은 속성을 함께 봐야 한다. PFILE은 text, SPFILE은 binary이며, 실무에서는 spfile만 믿지 말고 pfile 백업을 반드시 유지해야 한다. scope=memory / spfile / both는 현재 반영과 재기동 후 반영을 나누는 핵심 개념이다. 이 범위와 연결되는 확장 개념으로, 데이터 저장 구조는 Tablespace → Datafile → Segment → Extent → Block → Row 이고, Undo는 consistent read를 위해 과거 상태를 재구성하는 핵심 구조다. 즉 이 범위의 본질은 “Oracle은 파라미터로 기동되고, block 단위로 저장되며, SCN과 undo로 시간 일관성을 유지한다” 는 데 있다.
