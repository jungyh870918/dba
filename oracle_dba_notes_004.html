2단계 · 기동 철학

왜 startup을 NOMOUNT → MOUNT → OPEN으로 나누는가

Oracle은 단순한 프로그램이 아니라, 메모리도 띄워야 하고, 백그라운드 프로세스도 시작해야 하고, 어떤 파일을 열지 알아야 하고, 마지막으로 그 파일들이 정상인지까지 확인해야 한다. 그래서 DB를 한 번에 여는 게 아니라 단계적으로 확인하며 올린다.

1. 인스턴스를 먼저 만들고 2. DB 파일 지도를 읽고 3. 실제 데이터 파일을 열어 서비스 상태로 전환

즉 startup은 “켜기”가 아니라 점검을 포함한 단계별 기동 절차다.

2단계 · NOMOUNT

NOMOUNT: 인스턴스만 만든 상태

NOMOUNT 단계에서는 parameter file을 읽고 SGA 및 백그라운드 프로세스를 기동한다. 이 단계에서 생기는 것은 인스턴스 자체다.

이 단계에서 생기는 것

  • SGA
  • PMON / SMON / DBWR / LGWR / CKPT 같은 background process
  • instance 자체

이 단계에서 아직 하지 않은 것

  • control file open 안 함
  • datafile open 안 함
  • redo log file 경로도 control file 기준으로 확인 전
NOMOUNT = DB를 열기 전에 Oracle 엔진(인스턴스)부터 먼저 띄우는 단계
실무 감각: NOMOUNT는 평소 서비스 상태보다는, control file 생성/복구, 초기화 문제 진단, startup 문제 분리 같은 상황에서 특히 중요하다.
2단계 · MOUNT

MOUNT: control file을 읽어 DB의 파일 지도를 확보한 상태

MOUNT 단계에서는 control file을 open 해서, 어떤 datafile과 redo log file을 대상으로 DB를 열어야 하는지 파악한다.

MOUNT 상태에서 Oracle이 아는 것

  • datafile 목록
  • redo log file 목록
  • 각 파일의 위치
  • checkpoint / recovery 관련 메타정보
  • DB 전체 구조

아직 하지 않은 것

  • 실제 datafile open 안 함
  • 사용자 서비스 오픈 안 함
MOUNT = control file을 기준으로 어떤 DB 파일을 어떻게 열지 지도만 확보한 상태
실무 감각: MOUNT는 recovery와 아주 깊게 연결된다. control file을 읽어야 datafile 상태, checkpoint 기준, recovery 필요 여부를 판단할 수 있기 때문이다.
2단계 · OPEN

OPEN: datafile까지 열고 서비스 가능한 상태

OPEN 상태는 datafile을 실제로 열고, 유효한 사용자가 정상적으로 데이터 액세스를 할 수 있는 상태다. 여기서 핵심은 “RAM에 다 올렸다”가 아니라 Oracle이 서비스 가능한 정합성 상태까지 확인했다는 뜻이다.

OPEN에서 비로소 가능한 것

  • datafile open
  • redo log file 사용
  • 필요 시 recovery 수행 후 서비스 가능
  • 사용자 SQL 처리 가능
OPEN = 메모리에 다 올린 상태 ❌ OPEN = 정상 서비스 가능한 상태 ⭕

OPEN 상태에서도 datafile 전체를 RAM에 올리는 것은 아니다. 필요한 블록만 Buffer Cache에 올라오고, 나머지는 디스크에 그대로 있다.

OPEN 상태 ↓ User SQL 실행 가능 ↓ 필요한 블록만 Datafile → Buffer Cache로 읽어옴
핵심 정리: OPEN은 “램에 다 올렸다”가 아니라, 파일 구조 + 상태 + 정합성까지 확인하고 사용자가 SQL을 실행해도 되는 상태다.
2단계 · startup 명령의 내부 동작

startup 명령은 내부적으로 무엇을 하는가

startup은 내부적으로 NOMOUNT → MOUNT → OPEN까지 순차 수행한다. 반면 DBA는 단계별 명령으로 끊어서 문제를 분리 진단할 수 있다.

startup → NOMOUNT → MOUNT → OPEN 순차 수행 startup nomount → NOMOUNT까지만 alter database mount → 인스턴스 위에서 control file open alter database open → MOUNT 상태 DB를 서비스 가능 상태로 전환

이 단계 구분은 단순 문법이 아니라, 어느 단계에서 문제가 나는지 분리 진단하는 도구다.

2단계 · 핵심 흐름

parameter file → control file → datafile 흐름

Oracle startup에서 이 순서는 역행할 수 없다.

parameter file → instance 생성 control file → open 대상 파일 구조 인지 datafile → 실제 데이터 오픈

왜 이 순서인가

  • parameter file 먼저 — 메모리, 백그라운드 프로세스, 인스턴스 설정이 있어야 엔진을 띄울 수 있다.
  • control file 다음 — 어떤 DB 파일을 열어야 하는지 파일 지도를 알아야 한다.
  • datafile 마지막 — control file 기준으로 파일 목록과 상태를 확인한 뒤 실제 데이터를 연다.
핵심: startup은 “파일 하나 열기”가 아니라, 구조를 확인하며 점진적으로 서비스 상태로 올라가는 절차다.
2단계 · 운영 제어 모드

read only open과 restrict mode는 무엇인가

read only open

startup mount; alter database open read only;

datafile은 열되 쓰기는 막고, 읽기만 가능한 상태로 open 한다.

언제 쓰나

  • 데이터 점검
  • 장애 조사
  • 데이터 비교
  • 잘못된 데이터 확인(수정 없이)
  • 물리적으로 변경을 막고 상태 확인
감각: DB를 열긴 열되, 절대 건드리지 못하게 잠그는 안전 모드다.

restrict mode

startup restrict; -- 또는 alter system enable restricted session; alter system disable restricted session;

DB는 open 하되, 일반 사용자는 막고 제한된 사용자만 접속하게 하는 운영 모드다.

언제 쓰나

  • 긴급 점검
  • 배치 전 준비
  • 사용자 접속 차단
  • DBA 작업 수행
감각: DB는 살아있지만 일반 사용자 출입 금지인 상태다.
정리: read only / restrict는 자동으로 수행되는 디폴트 동작이 아니라, DBA가 필요할 때 수동으로 사용하는 운영 제어 도구다. 매일 무조건 쓰는 루틴 명령은 아니지만, 점검·배포·장애 대응에서는 매우 자주 등장한다.
2단계 · 종료 절차

Shutdown은 왜 역순으로 닫히는가

shutdown은 startup의 역순이다. 즉, 서비스 중지 → DB close → mount 해제 → 인스턴스 종료 순으로 진행된다.

OPEN → DB Close MOUNT → 마운트 해제 INSTANCE → 인스턴스 종료

사용자 접근 중지, 파일 정리, 메모리와 프로세스 정리를 순서대로 해야 하기 때문에 Oracle은 종료도 단계적이다.

2단계 · shutdown 옵션

shutdown 옵션 차이는 왜 실무에서 중요한가

shutdown normal

  • 사용자가 모두 logoff 될 때까지 대기
  • 종료되지 않을 확률이 높음

이론상 가장 얌전하지만, 운영 통제성이 떨어져 실무에서는 잘 안 쓴다.

shutdown transactional

  • 현재 진행 중인 트랜잭션이 종료될 때까지 대기
  • 그 후 세션 제거 후 종료

이미 시작한 업무 단위를 존중하는 종료 방식이지만, 긴 transaction이 있으면 오래 걸릴 수 있다.

shutdown immediate

  • 트랜잭션 강제 종료 및 rollback 수행 후 종료
  • DBA들이 가장 많이 사용하는 방식

운영 통제성과 정합성의 균형점이다. 너무 기다리지도 않고, abort처럼 거칠지도 않다.

shutdown abort

  • 트랜잭션 강제 종료
  • 정상 rollback 절차 없이 종료
  • OS의 kill -9에 가까운 강제 종료

정상 종료가 아니라 사실상 강제 전원 차단에 가깝다. 다음 startup 때 SMON recovery 부담이 커진다.

실무 기본 = shutdown immediate normal / transactional = 너무 오래 기다릴 수 있음 abort = 너무 거칠어서 다음 recovery 부담 증가
핵심: abort는 한 번도 쓰면 안 되는 금기라기보다, 정말 안 내려갈 때의 마지막 수단이다. 하지만 자주 쓰는 환경은 운영 절차가 건강하지 않다는 신호일 수 있다.
추가 질문 정리 · 개발자 공지

shutdown immediate를 할 때 개발자에게는 어떻게 고지하나

DB는 혼자 쓰는 게 아니므로 사전 공지가 원칙이다. 공지 없이 shutdown immediate를 실행하면 개발자 입장에서는 진행 중 작업이 날아갈 수 있으므로 사실상 사고로 받아들여질 수 있다.

일반적인 절차

  1. 시간 공지 — 예: 22:00~22:30 점검
  2. 영향 범위 공지 — 로그인 불가, API 중단, 배치 중지 여부
  3. 작업 내용 공지 — DB 재기동, 패치, 유지보수 등
  4. 필요 시 restricted session으로 신규 유입 차단
  5. 기존 세션 종료 유도 또는 강제 정리
  6. shutdown immediate 수행
공지 ↓ 신규 유입 차단(restrict 가능) ↓ 기존 세션 정리 ↓ shutdown immediate
실무 감각: 공지 없이 immediate = 사고다. shutdown immediate는 기술적으로 안전한 종료 옵션이지만, 사용자 트랜잭션은 끊기므로 운영 커뮤니케이션이 필수다.

3단계 · 전체 흐름 연결

26~33페이지와 추가 질문까지 하나의 운영 흐름으로 묶기

DB 기동 ↓ parameter file 읽기 ↓ SGA + background process 생성 ↓ NOMOUNT NOMOUNT ↓ control file open ↓ datafile / redo log file 구조 파악 ↓ MOUNT MOUNT ↓ 필요 시 recovery 판단 / 수행 ↓ datafile open ↓ OPEN 운영 중 ↓ read only / restrict 같은 제어 모드 사용 가능 종료 시 ↓ 사용자 접근 종료 ↓ DB close ↓ mount 해제 ↓ instance 종료 문제 발생 시 ↓ NOMOUNT 실패 → parameter / memory MOUNT 실패 → control file OPEN 실패 → datafile / recovery
핵심: 앞에서 배운 SGA, control file, checkpoint, SMON, recovery 개념이 전부 startup / shutdown 흐름 안으로 들어온다. 이 범위는 Oracle 내부 구조를 실제 운영 절차에 꽂아 넣는 장이다.
4단계 · 실무 포인트

이 범위에서 실제 운영 판단으로 이어지는 지점

  • OPEN이 중요하다 — OPEN까지 못 가면 DB는 서비스 불가 상태다.
  • startup 문제는 어느 단계에서 멈췄는지로 본다 — 장애 분리 진단의 기본이다.
  • MOUNT는 recovery와 깊게 연결된다 — control file을 읽어야 recovery 필요 여부를 판단할 수 있다.
  • shutdown immediate가 실무 기본 — 운영 통제성과 정합성의 균형점이다.
  • abort를 자주 쓰는 환경은 건강하지 않다 — 다음 recovery 부담이 커진다.
  • read only / restrict는 운영 제어 도구 — 자동 루틴이 아니라, 상황이 생길 때 DBA가 수동으로 사용하는 기본 도구다.
5단계 · 핵심 요약

이번 범위 전체 요약

26~33페이지는 Oracle의 기동/종료 운영 절차다. startup은 parameter file → instance 생성(NOMOUNT) → control file 확인(MOUNT) → datafile open(OPEN) 순서로 진행된다. 즉 Oracle은 “메모리와 프로세스를 먼저 만들고, DB 파일 지도를 읽고, 마지막에 실제 데이터를 연다.” OPEN은 메모리에 다 올렸다는 뜻이 아니라, 파일 구조와 상태와 정합성을 모두 확인해 사용자가 SQL을 실행해도 되는 상태를 뜻한다. read only / restrict는 자동으로 수행되는 디폴트가 아니라, DBA가 필요할 때 수동으로 사용하는 운영 제어 도구다. shutdown은 startup의 역순이며, 실무 기본은 shutdown immediate다. normal / transactional은 너무 오래 기다릴 수 있고, abort는 너무 거칠어서 다음 startup 때 recovery 부담이 커진다. startup 실패는 단계별로 원인이 다르다: NOMOUNT → parameter / memory 문제 MOUNT → control file 문제 OPEN → datafile / recovery 문제 archive full, control file 손상, parameter 손상 같은 문제는 복제 + 자동화 + 모니터링 + 복구 테스트로 예방한다. 이 범위를 이해하면 Oracle 내부 구조가 실제 운영 절차 안에서 어떻게 작동하는지 보이기 시작한다.
