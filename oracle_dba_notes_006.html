2단계 · 50~53페이지

Trace file과 Dictionary를 통해 “실행 흔적”을 찾는 법

왜 이 파트가 중요한가

지금까지는 SGA, redo, undo, checkpoint 같은 내부 구조를 공부했다면, 이 파트부터는 실제 SQL이 어떻게 실행됐는지 증거를 수집하는 방법으로 넘어간다. 즉, 구조 이해에서 분석 도구로 넘어가는 전환 구간이다.

문제 발생 → 실행 흔적(trace) 남김 → trace 파일 위치 확인 → tkprof로 사람이 읽을 수 있게 변환 → logical read / physical read / current read / row 수 / fetch 회수 해석

Dictionary / 동적 성능 뷰가 여기서 왜 나오나

여기서 말하는 dictionary는 단순히 테이블 정의를 본다는 의미보다, Oracle이 현재 어디에 무엇을 기록하고 있는지 조회하는 내부 뷰를 사용하는 것에 더 가깝다. 특히 trace 파일 위치나 파라미터 값을 찾기 위해 v$diag_info, v$parameter 같은 뷰를 사용한다.

v$diag_info

12c 이상 환경에서 diag trace 경로를 확인할 때 사용한다.

select value from v$diag_info where name = 'Diag Trace';

v$parameter

9i, 10g, 11g 계열에서 trace 관련 dump 경로를 확인할 때 사용한다.

select name, value from v$parameter where name in ('background_dump_dest','user_dump_dest','core_dump_dest');

trace file 종류와 의미

alert<SID>.log

  • DB 기동 / 종료 / 각종 에러 정보
  • DBA가 항상 확인해야 할 log 파일
  • 12c 이상은 diag trace 위치에 생성

background_dump_dest

  • Background Process가 생성한 각종 파일 위치
  • 구버전에서는 alert log 위치와 연결

user_dump_dest

  • User trace 파일 생성 위치
  • 10046 trace file 등 user가 생성한 trace 파일
  • SQL의 주요 에러 발생 시 생성 가능

core_dump_dest

  • Core file dump 위치
  • 일반 사용자가 직접 보기 쉽지 않음
  • Oracle 지원팀이 장애 원인 파악용으로 요청하는 경우가 많음
실무 포인트: alert log는 DBA가 매일 보는 파일이고, user trace는 특정 SQL 분석 시 찾는 파일이며, core dump는 일반 운영보다 심각한 장애 분석 쪽에 가깝다.
2단계 · 50~53페이지

SQL Trace, 10046 Trace, TKPROF는 어떻게 연결되는가

기본 흐름

SQL 실행 → trace 정보 생성 → OS에서 trace 파일 찾기 → tkprof로 사람이 읽을 수 있는 보고서로 변환 → 통계 해석

10046 trace 위치 확인

-- 9i, 10g, 11g select name, value from v$parameter where name = 'user_dump_dest'; -- 12c 이상 select value from v$diag_info where name = 'Diag Trace';

OS에서 trace 파일 찾기

OS> ls -l *sik* -- unix 계열 OS> dir *sik* -- windows

tkprof 변환

OS> tkprof orcl_ora_12244_sik.trc sik10046.txt sys=no

sys=no 옵션은 SQL 파싱 중 발생하는 내부 수행 SQL을 제외해, 사용자가 보고 싶은 SQL 중심으로 결과를 정리하려는 의도다.

핵심: trace 파일은 원본 증거이고, tkprof는 그 증거를 사람이 읽고 비교할 수 있는 리포트로 바꾸는 도구다.

10046 trace에서 보는 대표 통계

count

OCI procedure가 수행된 횟수

cpu / elapsed

CPU 사용 시간 / 전체 경과 시간

disk

physical read 수

query

consistent read용 logical read 수

current

current mode read 수, 보통 update 계열과 연결

rows

fetch 또는 execute에서 처리된 row 수

10046 통계 해석 포인트

query = logical read (cr, buffers, consistent gets) disk = physical read (pr, reads, physical reads) current = current mode read fetch rows = fetch한 row 수 fetch count = fetch 수행 횟수, sql*net roundtrip과 연결
실무 포인트: query가 많으면 logical read가 많다는 뜻이고, disk가 많으면 physical read가 많다는 뜻이다. current는 보통 update 성격의 접근과 연결된다.
2단계 · 50~53페이지

sql trace와 10046 trace의 차이

PDF에서는 sql_trace 실습과 10046 trace를 모두 보여주는데, 둘은 비슷해 보이지만 깊이가 다르다.

sql_trace 실습 흐름

alter session set max_dump_file_size = unlimited; alter session set tracefile_identifier='sik'; alter session set sql_trace=true; select * from emp; alter session set sql_trace=false;

이후 trace file을 찾아 tkprof로 변환하고 분석한다.

중요 차이: PDF에서 명시하듯 sql_trace는 10046 trace와 달리 wait event 정보가 출력되지 않는다. 즉, 기본 실행 통계는 보지만 wait까지 깊게 보는 건 10046 쪽이다.

10046 trace 실습 흐름

ALTER SESSION SET EVENTS '10046 trace name context forever, level 12'; -- SQL 수행 ALTER SESSION SET EVENTS '10046 trace name context off';

level 12는 wait + bind까지 포함하는 강한 추적 수준으로 많이 알려져 있다. 즉 sql_trace보다 더 깊은 실행 흔적을 남긴다.

실무 감각: 단순히 실행계획과 read 통계만 볼 거면 autotrace / xplan으로도 충분할 수 있지만, 실행 도중 어디서 기다렸는지까지 봐야 하면 10046이 필요하다.
3단계 · 54~60페이지 연습문제

연습문제는 “SQL 분석 도구 4종 세트”를 손으로 익히게 한다

54~60페이지의 연습문제는 단순 답 외우기가 아니라, 같은 SQL을 서로 다른 도구로 관찰해보게 만드는 구조다.

같은 SQL → AUTOTRACE로 본다 → DBMS_XPLAN.DISPLAY로 본다 → DBMS_XPLAN.DISPLAY_CURSOR로 본다 → 10046 TRACE로 본다 즉, “같은 SQL을 여러 관찰 도구로 어떻게 다르게 해석하는지” 익히는 연습이다.
핵심 목표: “실행계획 보는 법”만이 아니라, 예상 계획 / 실제 수행 / trace 기반 증거를 구분해서 보는 감각을 만드는 것이다.
연습문제 17-1

AUTOTRACE

문제

SQLPLUS를 이용하여 아래 SQL의 AUTOTRACE를 떠 보세요. SELECT * FROM HR.EMPLOYEES WHERE EMPLOYEE_ID = 100;

답안 흐름

C:\Users\seo> sqlplus system/manager1@orclpdb set linesize 200 pagesize 200 set autotrace traceonly SELECT * FROM HR.EMPLOYEES WHERE EMPLOYEE_ID = 100;

이 문제의 학습 목표

  • SQL*Plus에서 가장 빠르게 실행계획과 통계를 확인하는 법 익히기
  • trace file 없이 화면에서 바로 분석하는 감각 익히기
  • 개발 / 운영 초입에서 빠른 1차 점검 도구로 활용하는 습관 만들기

이 문제를 통해 이해해야 하는 개념

  • AUTOTRACE는 빠른 1차 점검 도구다.
  • trace file을 만들지 않고 실행계획과 통계를 화면에 바로 보여준다.
  • PDF에서 소개된 옵션 차이를 이해해야 한다.
set autotrace on → SQL 결과 + 실행계획 + 실행통계 set autotrace traceonly → SQL 결과는 숨기고 실행계획/통계만 확인
실무 포인트: AUTOTRACE는 빠르지만, 실제 커서에 남은 상세 실행 정보나 wait event까지 보기는 어렵다. 즉 빠른 예비 분석용이다.
연습문제 17-2

DBMS_XPLAN.DISPLAY

문제

DBMS_XPLAN.DISPLAY를 이용하여 17-1번 SQL의 실행계획을 확인하세요.

답안 흐름

EXPLAIN PLAN FOR SELECT * FROM HR.EMPLOYEES WHERE EMPLOYEE_ID = 100; SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY(NULL, NULL, 'TYPICAL'));

이 문제의 학습 목표

  • 예상 실행계획을 PLAN_TABLE에 저장하고 읽는 법 익히기
  • SQL을 실제 실행하지 않고도 옵티마이저 계획을 확인하는 절차 익히기
  • AUTOTRACE와 달리 실행 없이 계획만 보는 방식 이해하기

이 문제를 통해 이해해야 하는 개념

  • EXPLAIN PLAN은 예상 계획이다.
  • 실제 수행 결과와 다를 수 있다.
  • 즉 “옵티마이저가 이렇게 하겠다고 생각한 계획”을 보는 도구다.
핵심 구분: DISPLAY는 예상 계획이고, 실제 실행 후의 row 수나 read 수를 직접 보여주는 도구는 아니다.
연습문제 17-3

DBMS_XPLAN.DISPLAY_CURSOR

문제

DBMS_XPLAN.DISPLAY_CURSOR를 이용하여 17-1번 SQL의 실행계획 및 수행 정보를 확인하세요.

답안 흐름

ALTER SESSION SET STATISTICS_LEVEL = ALL; SELECT * FROM HR.EMPLOYEES WHERE EMPLOYEE_ID = 100; SELECT * FROM TABLE(DBMS_XPLAN.DISPLAY_CURSOR(NULL, NULL, 'IOSTATS LAST -ROWS'));

이 문제의 학습 목표

  • 실제로 수행된 SQL 커서를 기준으로 계획을 읽는 법 익히기
  • 예상 계획과 실제 수행 통계를 구분해서 보는 감각 만들기
  • library cache에 남은 정보를 활용하는 흐름 이해하기

이 문제를 통해 이해해야 하는 개념

  • DISPLAY_CURSOR는 실제 실행 후 커서 기반 분석이다.
  • STATISTICS_LEVEL = ALL로 실행 통계를 남긴다.
  • AUTOTRACE나 EXPLAIN PLAN보다 실제 실행 정보에 더 가깝다.
실무 포인트: 예상 계획과 실제 계획이 다를 수 있기 때문에, 튜닝할 때는 DISPLAY_CURSOR가 훨씬 유용한 경우가 많다.
연습문제 17-4

10046 TRACE

문제

10046 TRACE를 이용하여 17-1번의 SQL 실행 정보를 확인하세요.

답안 흐름

ALTER SESSION SET TRACEFILE_IDENTIFIER = 'sik'; ALTER SESSION SET EVENTS '10046 trace name context forever, level 12'; SELECT * FROM HR.EMPLOYEES WHERE EMPLOYEE_ID = 100; ALTER SESSION SET EVENTS '10046 trace name context off'; SELECT VALUE FROM V$DIAG_INFO WHERE NAME = 'Diag Trace';

이 문제의 학습 목표

  • trace를 실제로 남기고 파일 경로까지 확인하는 전 과정을 익히기
  • SQL 분석을 화면 출력이 아니라 파일 기반 증거 수집으로 하는 절차 익히기
  • tracefile_identifier를 통해 원하는 trace 파일을 찾는 감각 만들기

이 문제를 통해 이해해야 하는 개념

  • 10046 trace는 가장 깊은 SQL 실행 추적 도구 중 하나다.
  • 단순 실행계획이 아니라 wait, read, parse, fetch, row 처리, internal SQL까지 볼 수 있다.
  • trace 파일은 남겨도 사람이 바로 읽기 어려워 tkprof와 함께 쓰는 경우가 많다.
핵심: 17-4는 “실행계획 보기”가 아니라, 증거를 남기고 추적하는 DBA 방식의 분석을 익히는 문제다.
4단계 · 네 문제를 한 번에 비교

연습문제 17-1 ~ 17-4를 통해 최종적으로 얻어야 하는 감각

17-1 AUTOTRACE

빠른 1차 점검

17-2 DISPLAY

예상 실행계획 확인

17-3 DISPLAY_CURSOR

실제 수행 커서 기반 분석

17-4 10046 TRACE

파일 기반 정밀 추적

AUTOTRACE → 빨리 본다 DISPLAY → 옵티마이저가 생각한 계획 본다 DISPLAY_CURSOR → 실제 실행 후 커서 기준으로 본다 10046 TRACE → 가장 깊게 추적하고 증거를 남긴다
실무 핵심: DBA는 도구 하나만 쓰지 않는다. 상황에 따라 빠른 점검 → 실제 실행 확인 → 정밀 추적으로 깊이를 조절한다.
5단계 · 실무 포인트

이 범위를 실무에서 어떻게 써먹는가

  • alert log는 매일 확인하는 기본 로그다.
  • user trace는 특정 SQL / 세션 분석 시 찾는다.
  • 10046 trace는 성능 문제 증거를 수집하는 강력한 도구다.
  • tkprof는 raw trace를 읽기 좋은 리포트로 바꾼다.
  • AUTOTRACE는 빠른 1차 점검용이다.
  • DISPLAY와 DISPLAY_CURSOR 차이를 모르면 예상 계획과 실제 계획을 혼동하기 쉽다.
즉: 50~60페이지는 단순히 명령어를 배우는 장이 아니라, “SQL 성능 문제를 어떤 깊이로 관찰할 것인가”를 배우는 파트다.
6단계 · 핵심 요약

이번 범위 전체 요약

50~53페이지는 trace file과 관련 dictionary / 동적 성능 뷰를 이용해 Oracle이 남긴 실행 흔적을 어디서 찾고 어떻게 읽는지 설명하는 구간이다. 핵심은 v$diag_info, v$parameter로 trace 경로를 찾고, sql_trace와 10046 trace를 남기고, tkprof로 사람이 읽을 수 있는 보고서로 바꿔 logical read, physical read, current read, fetch, row 처리량을 해석하는 것이다. 54~60페이지의 연습문제는 같은 SQL을 AUTOTRACE, DBMS_XPLAN.DISPLAY, DBMS_XPLAN.DISPLAY_CURSOR, 10046 TRACE 네 가지 방식으로 관찰하게 만든다. 즉 이 범위의 진짜 목표는 “도구 사용법 암기”가 아니라, 예상 계획 / 실제 커서 / trace 증거를 각각 어떤 상황에서 써야 하는지 구분하는 감각을 만드는 데 있다.
