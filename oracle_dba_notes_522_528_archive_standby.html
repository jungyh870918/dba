Archive Mode · Redo Log Switch · Archive Log · Standby / Data Guard 개념 정리

이 문서는 522~528페이지의 Archive Mode 내용을 중심으로, 이후 추가로 나눈 Redo Log Switch, 강제 Log Switch, Standby Sync, Standby DB의 정체와 운영 관리 개념까지 하나로 묶어 정리한 것이다. 핵심은 Redo는 현재 변경 이력, Archive는 Redo의 보관본, Standby는 Redo/Archive를 전달받아 Primary를 따라가는 별도의 대기 DB라는 점이다.

1. Archive Mode가 왜 중요한가

Oracle에서 장애 복구는 크게 두 관점으로 나눌 수 있다. 단순히 인스턴스가 비정상 종료된 경우에는 Instance Recovery가 수행되지만, Datafile 손상, Disk 장애, Storage 장애처럼 실제 파일이 손상된 경우에는 Media Recovery가 필요하다.

구분 의미 필요한 핵심 요소
Instance Recovery 인스턴스 비정상 종료 후 DB 재기동 과정에서 수행되는 복구 Online Redo Log, Undo
Media Recovery Datafile 손상, Disk 장애 등 파일 자체 문제가 발생했을 때 수행하는 복구 Backup File + Archive Log
핵심
운영 DBA 관점에서 Archive Mode가 중요한 이유는 Media Recovery 때문이다. Backup만 있으면 백업 시점까지만 돌아갈 수 있지만, Archive Log가 있으면 백업 이후 장애 직전까지의 변경 내용을 다시 적용할 수 있다.

2. Redo Log와 Archive Log의 차이

Redo Log는 현재 운영 중인 변경 이력을 기록하는 로그이고, Archive Log는 사용이 끝난 Redo Log를 복사해서 별도로 보관한 파일이다.

구분 Redo Log Archive Log
역할 현재 DB 변경 이력을 기록 사용이 끝난 Redo Log를 영구 보관
기록 주체 LGWR ARCn
구조 순환 구조. 일정 시점 이후 재사용됨 별도 파일로 저장되어 복구에 사용됨
장애 복구 관점 Instance Recovery에 핵심 Media Recovery와 Point-in-time Recovery에 핵심
User Commit ↓ Redo Log Buffer ↓ LGWR ↓ Online Redo Log File ↓ Log Switch 발생 ↓ ARCn ↓ Archived Redo Log File

3. Log Switch가 발생하면 무슨 일이 일어나는가

Oracle의 Online Redo Log는 여러 Group이 순환하면서 사용된다. 현재 사용 중인 Redo Log Group이 가득 차거나 DBA가 강제로 전환하면 Log Switch가 발생한다.

Group 1 사용 중 ↓ Group 1 가득 참 또는 강제 switch ↓ LGWR가 Group 2로 이동 ↓ Group 1은 기록 종료 ↓ ARCn이 Group 1을 Archive Log로 복사
정확한 이해
Commit이 발생할 때마다 Archive Log가 생성되는 것이 아니다. Archive Log는 Redo Log Group이 Switch되어 사용이 끝났을 때 생성된다.

4. Redo Log Switch와 Archive 생성의 관계

Redo Log Switch가 발생하면 LGWR는 다음 Redo Log Group으로 이동한다. 그 직후 ARCn은 방금 사용이 끝난 Redo Log Group을 Archive Log File로 복사한다.

[Switch 전] LGWR → Group 1 기록 중 Group 1 : 현재 사용 중 Group 2 : 대기 Group 3 : 대기 [Switch 후] LGWR → Group 2 기록 시작 Group 1 : 기록 종료 Group 2 : 현재 사용 중 Group 3 : 대기 [Archive] ARCn Group 1의 Redo Log File ↓ Archive Log File 생성

Redo Log는 순환 구조이기 때문에 결국 다시 덮어쓰기 대상이 된다. 하지만 ARCHIVELOG Mode에서는 덮어쓰기 전에 사용이 끝난 Redo Log를 Archive로 보관한다. 그래야 나중에 Backup 이후의 변경 이력을 복구할 수 있다.

5. ARCHIVELOG Mode와 NOARCHIVELOG Mode

Oracle DB는 Archive Log를 생성하는 방식으로 운영할 수도 있고, 생성하지 않는 방식으로 운영할 수도 있다.

구분 NOARCHIVELOG ARCHIVELOG
Archive 생성 생성하지 않음 생성함
백업 방식 Cold Backup 중심 Hot Backup 가능
운영 중 백업 제약 큼 가능
장애 직전 복구 어려움 Archive Log를 이용해 가능
실무 운영 DB 위험함 사실상 필수
운영 관점
개발 DB나 임시 테스트 DB는 NOARCHIVELOG로 운영할 수도 있지만, 중요한 운영 DB는 보통 ARCHIVELOG Mode가 필수에 가깝다.

6. Archive 상태 확인

현재 DB가 Archive Mode인지 확인하려면 다음 명령을 사용한다. 

archive log list;

이 명령을 통해 Database Log Mode, Automatic Archival, Archive Destination, 현재 Log Sequence 등을 확인할 수 있다.

DBA가 확인해야 할 것
Archive Mode 여부만 보는 것이 아니라, Archive Destination이 정상인지, Archive 공간이 가득 차지 않았는지, Log Switch 이후 Archive가 정상 생성되는지까지 함께 확인해야 한다.

7. Archive Destination Full이 왜 위험한가

Archive Log가 저장되는 경로가 가득 차면 Redo Log를 Archive로 복사하지 못한다. 이 경우 사용이 끝난 Redo Log Group을 재사용할 수 없고, 결국 Log Switch가 막히면서 DB 전체 DML과 Commit이 지연될 수 있다.

Archive Destination Full ↓ ARCn이 Redo Log를 Archive로 복사하지 못함 ↓ 해당 Redo Log Group 재사용 불가 ↓ Log Switch 진행 불가 ↓ Commit 지연 ↓ Insert / Update / Delete 멈춤 ↓ 서비스 장애
실무 핵심
Archive 공간 Full은 단순 저장공간 부족 문제가 아니라 서비스 장애로 이어질 수 있다. 그래서 DBA는 Archive 경로 사용량과 Archive 생성 상태를 매일 점검해야 한다.

8. ARCHIVELOG Mode 전환 절차

ARCHIVELOG Mode로 전환하려면 DB를 MOUNT 상태에서 변경해야 한다. OPEN 상태에서 바로 전환하는 것이 아니다. 

shutdown immediate;

startup mount;

alter database archivelog;

alter database open;

archive log list;

Archive Mode 변경은 Control File과 Redo 관련 속성을 변경하는 작업이므로, Datafile까지 완전히 Open된 상태가 아니라 MOUNT 상태에서 수행한다.

NOMOUNT → Parameter File 읽음, Instance 생성 MOUNT → Control File 읽음, DB 구조 인식 OPEN → Datafile 열림, 사용자 접속 가능

9. 강제 Log Switch의 목적

DBA는 필요할 때 강제로 Log Switch를 발생시킬 수 있다. 

alter system switch logfile;

이 명령은 현재 사용 중인 Redo Log Group을 종료하고 다음 Group으로 이동하게 한다. 그 결과 방금 종료된 Redo Log Group은 Archive 대상이 된다.

핵심 의미
강제 Log Switch는 “지금까지의 변경 내용을 Archive Log로 확정시키고 싶다”는 의미에 가깝다.

대표적인 사용 목적

  • 백업 직전까지의 변경 이력을 Archive로 확정하고 싶을 때
  • Archive Log가 정상 생성되는지 테스트할 때
  • Archive Destination 상태를 점검할 때
  • Data Guard / Standby 환경에서 Redo 전송 여부를 확인할 때
  • 장애 조사 중 특정 시점의 Redo Sequence를 명확히 만들고 싶을 때
주의
강제 Log Switch를 너무 자주 수행하면 Archive Log가 과도하게 증가하고, I/O 부하와 백업 부담이 커질 수 있다. 필요할 때만 사용해야 한다.

10. Log Switch와 Checkpoint는 다르다

강제 Log Switch와 Checkpoint를 혼동하면 안 된다.

명령 역할 핵심 동작
alter system switch logfile; Log Switch 발생 LGWR가 다음 Redo Log Group으로 이동, 이전 Group은 Archive 대상
alter system checkpoint; Checkpoint 발생 DBWR에게 Dirty Buffer를 Datafile로 쓰도록 유도하고 Checkpoint 정보 갱신
구분
Log Switch는 Redo Log Group 전환이고, Checkpoint는 메모리의 변경된 블록을 Datafile에 반영하고 복구 시작점을 줄이는 개념이다.

11. Restore와 Recovery 연결

Archive Mode는 백업/복구 개념과 바로 연결된다. 여기서 Restore와 Recovery를 구분해야 한다.

구분 의미 예시
Restore 백업본에서 손상된 파일을 다시 가져오는 것 Backup에서 Datafile 복사
Recovery Restore한 파일에 Redo / Archive Log를 적용해 최신 상태로 맞추는 것 월요일 Backup + 화요일까지의 Archive Log 적용
핵심
Restore만 하면 백업 시점의 파일을 가져온 것에 불과하다. 실제 서비스 가능한 상태까지 복구하려면 Recovery를 통해 Archive Log를 적용해야 한다.

12. Standby DB란 무엇인가

Standby DB는 Primary DB의 장애에 대비해 만들어두는 대기용 복제 DB다. 단순한 백업 파일이 아니라, Redo 또는 Archive를 전달받아 Primary DB를 계속 따라가는 살아있는 DB다.

Primary DB → 실제 서비스가 붙는 메인 DB Standby DB → Primary 장애 시 대신 올라올 수 있도록 대기하는 복제 DB
비유
Backup은 사진에 가깝고, Standby는 살아있는 쌍둥이에 가깝다. Backup은 복원 대상이고, Standby는 즉시 서비스 전환 대상이다.

13. Primary와 Standby의 동작 흐름

Primary DB에서 변경이 발생하면 Redo가 생성된다. Data Guard 환경에서는 이 Redo가 Standby DB로 전달되고, Standby DB는 전달받은 Redo를 적용하면서 Primary와 비슷한 상태를 유지한다.

Primary DB ↓ Redo 발생 ↓ Archive / Redo 전송 ↓ Standby DB ↓ Redo 수신 ↓ Apply ↓ Primary와 유사한 상태 유지

여기서 중요한 것은 Standby가 Primary 내부에서 자동 생성되는 객체가 아니라, 별도로 구축된 독립 DB라는 점이다.

14. Standby는 자동으로 존재하는가?

아니다. Standby는 Oracle을 설치하면 자동으로 생기는 대상이 아니다. DBA가 별도로 설계하고 구축하고 운영해야 하는 별도 DB다.

기본 설치 시 존재 별도 구축 필요
CDB, PDB$SEED, Redo Log, Control File, Undo, Background Process, Archive Mode 기능 Standby DB, Data Guard 구성, Redo Transport, Standby Redo Log, Apply 프로세스 관리
중요
Standby는 Oracle 내부에서 자동으로 몰래 생성되어 관리되는 대상이 아니다. 실제로 다른 서버, 다른 인스턴스, 다른 스토리지에 존재하는 별도의 Oracle DB로 보는 것이 맞다.

15. Standby는 언제 구성하는가

Standby는 모든 DB에 항상 구성하는 것은 아니다. 비용과 운영 복잡도가 크기 때문에 중요한 운영 DB에 선택적으로 구성한다.

  • 금융, 결제, 병원, 공공기관 등 장애 허용 시간이 짧은 핵심 시스템
  • 장애 발생 시 빠른 Failover가 필요한 시스템
  • 재해복구센터 또는 DR 구성이 필요한 시스템
  • Primary 부하를 줄이기 위해 조회를 Standby로 분산하고 싶은 경우
실무 감각
Backup은 거의 필수이고, Standby는 선택이다. 하지만 중요한 운영 DB에서는 사실상 필수에 가까운 경우가 많다.

16. Standby의 종류

구분 의미 특징
Physical Standby Redo를 그대로 적용해 Primary와 물리적으로 거의 같은 DB를 유지 실무에서 가장 많이 사용
Logical Standby Redo를 SQL 형태로 변환해 적용 특수한 목적에 사용, 시험에는 자주 등장
Active Data Guard Standby를 Read Only로 열어 조회 부하를 분산 보고서, 조회, 리포트 시스템 등에 활용 가능

17. Standby Sync 확인이란 무엇인가

Standby Sync 확인은 Primary DB에서 발생한 Redo가 Standby DB까지 제대로 전달되고, Standby에서 실제로 적용되고 있는지 확인하는 작업이다.

여기서 Sync는 두 단계로 나누어 봐야 한다.

단계 의미 문제 발생 시 의심 지점
Transport Primary에서 Standby로 Redo가 전달되었는가 네트워크, Listener, Archive Destination, Data Guard 전송 설정
Apply Standby가 받은 Redo를 실제 데이터에 적용했는가 Standby 서버 성능, I/O, MRP 프로세스, Recovery Apply 병목
핵심
Redo를 받기만 한 상태와 실제로 적용까지 끝난 상태는 다르다. Transport OK와 Apply OK를 분리해서 봐야 한다.

18. Standby Sync 확인에 강제 Log Switch를 쓰는 이유

Primary에서 강제로 Log Switch를 발생시키면 새로운 Archive Sequence가 만들어진다. 그러면 Standby에서 그 Sequence를 받았는지, 그리고 적용했는지 확인할 수 있다.

Primary에서 강제 Log Switch ↓ 새 Archive Sequence 생성 ↓ Standby로 전송 ↓ Standby에서 해당 Sequence 수신 여부 확인 ↓ Standby에서 해당 Sequence Applied 여부 확인

즉, 강제 Log Switch는 “지금 Sequence 하나 만들어서, 그게 Standby까지 도착하고 적용되는지 보자”는 확인 용도로 사용될 수 있다.

19. Sequence 기준으로 Standby 상태 보기

Redo / Archive는 Sequence 번호를 가진다. Standby Sync를 확인할 때는 Primary의 현재 Sequence와 Standby의 Received / Applied Sequence를 비교한다.

정상에 가까운 상태: Primary sequence = 120 Standby received = 120 Standby applied = 120 전송은 됐지만 적용이 밀린 상태: Primary sequence = 120 Standby received = 120 Standby applied = 117 전송부터 밀린 상태: Primary sequence = 120 Standby received = 116 Standby applied = 116
운영 판단
Received가 밀리면 Transport 문제를 의심하고, Received는 따라왔는데 Applied가 밀리면 Apply 병목을 의심한다.

20. Transport Lag와 Apply Lag

Data Guard 환경에서는 보통 Transport Lag와 Apply Lag를 나누어 본다.

지표 의미 해석
Transport Lag Primary에서 생성된 Redo가 Standby에 도착하지 못한 지연 전송 경로 문제, 네트워크, Archive Destination, Data Guard 설정 점검
Apply Lag Standby가 Redo를 받았지만 아직 적용하지 못한 지연 Standby 서버 성능, I/O, Recovery Apply 프로세스 점검 
select name, value, unit
from v$dataguard_stats;

21. RAC 환경에서는 Thread도 같이 봐야 한다

RAC 환경에서는 인스턴스마다 Redo Thread가 있을 수 있다. 따라서 Standby Sync 확인 시 Sequence만 보는 것이 아니라 Thread 번호도 함께 봐야 한다.

Instance 1 → Thread 1 Instance 2 → Thread 2 확인 기준: Thread 1 / Sequence 120 / Applied Thread 2 / Sequence 98 / Applied
주의
RAC에서는 단순히 Sequence 숫자 하나만 비교하면 안 된다. THREAD# + SEQUENCE# 조합으로 확인해야 한다.

22. Standby 구축에 필요한 대표 작업

Standby는 자동 생성되는 것이 아니라 DBA가 직접 구축해야 한다. 일반적으로 다음과 같은 작업들이 필요하다.

  • Primary DB의 ARCHIVELOG Mode 확인
  • FORCE LOGGING 설정 검토
  • Password File 준비
  • Listener / TNS 설정
  • Standby용 서버와 스토리지 준비
  • RMAN Duplicate for Standby 수행
  • Data Guard 관련 Parameter 설정
  • Redo Transport 설정
  • Standby Redo Log 구성
  • Apply 프로세스 정상 동작 확인
실무 포인트
Standby는 만드는 것보다 운영이 더 중요하다. 구축 후에도 Transport Lag, Apply Lag, Archive Gap, MRP/RFS 상태를 지속적으로 확인해야 한다.

23. Primary 장애 시 Standby는 어떻게 쓰이는가

Primary DB에 장애가 발생하면 Standby DB를 Primary 역할로 승격할 수 있다. 이 작업을 Failover라고 한다.

Primary DB 장애 발생 ↓ Standby DB 상태 확인 ↓ Standby를 Primary로 승격 ↓ 서비스 접속 방향 전환 ↓ Standby가 새로운 Primary 역할 수행
비유
Primary가 주전 선수라면 Standby는 후보 선수다. 주전 선수가 부상당하면 후보 선수를 주전으로 올리는 것이 Failover다.

24. DBA 아침 점검 루틴에 연결하기

Archive Mode와 Standby 환경이 있는 운영 DB라면 DBA는 매일 다음 항목을 확인해야 한다.

  • archive log list로 Archive Mode와 Current Sequence 확인
  • Archive Destination 사용량 확인
  • Archive Log 생성이 정상인지 확인
  • Alert Log에서 Archive 관련 에러 확인
  • Backup 성공 여부 확인
  • RMAN Backup / Archive Backup 상태 확인
  • Log Switch 빈도 확인
  • Data Guard 사용 시 Transport Lag / Apply Lag 확인
  • Standby Archive Gap 여부 확인
  • MRP / RFS 프로세스 정상 동작 확인
장애로 이어지는 대표 신호
Archive Destination Full, Archive Gap, Apply Lag 누적, 비정상적으로 잦은 Log Switch는 모두 운영 장애의 전조가 될 수 있다.

25. 전체 개념 연결도

User DML / Commit ↓ Redo Log Buffer ↓ LGWR ↓ Online Redo Log File ↓ Log Switch ↓ ARCn ↓ Archive Log File ↓ Backup + Archive Log ↓ Media Recovery 가능 Data Guard 환경에서는: Primary Redo / Archive ↓ Redo Transport ↓ Standby 수신 ↓ Apply ↓ Primary 장애 시 Failover 후보

26. 최종 핵심 정리

  • Redo Log는 현재 DB 변경 이력을 기록하는 순환 로그다.
  • Archive Log는 사용이 끝난 Redo Log를 별도로 보관한 파일이다.
  • Archive Log는 Commit마다 생기는 것이 아니라 Log Switch 이후 생성된다.
  • LGWR는 Online Redo Log에 기록하고, ARCn은 사용이 끝난 Redo Log를 Archive로 복사한다.
  • ARCHIVELOG Mode는 운영 DB의 Hot Backup과 Media Recovery에 핵심이다.
  • Archive Destination Full은 Commit 지연과 서비스 장애로 이어질 수 있다.
  • alter system switch logfile은 현재 Redo Group을 강제로 종료하고 Archive 생성을 유도한다.
  • 강제 Log Switch는 백업 직전, Archive 정상 생성 확인, Standby Sync 확인 등에 사용된다.
  • Log Switch와 Checkpoint는 다른 개념이다.
  • Restore는 백업 파일을 가져오는 것이고, Recovery는 Archive Log를 적용하는 것이다.
  • Standby DB는 Primary DB의 대기용 복제 DB다.
  • Standby는 Oracle이 자동 생성하는 내부 객체가 아니라 DBA가 별도로 구축하고 운영해야 하는 DB다.
  • Standby Sync는 Transport와 Apply를 나누어 확인해야 한다.
  • Data Guard에서는 Sequence 기준으로 Standby 수신/적용 여부를 확인한다.
  • RAC 환경에서는 Thread#와 Sequence#를 함께 봐야 한다.

27. 면접/실무 답변용 한 줄 정리

Archive Mode
Archive Mode는 사용이 끝난 Redo Log를 Archive Log로 보관해, 백업 이후 장애 직전까지 Media Recovery를 가능하게 하는 운영 DB의 핵심 복구 기반이다.
Log Switch
Log Switch는 LGWR가 현재 Redo Log Group 사용을 끝내고 다음 Group으로 이동하는 것이며, ARCHIVELOG Mode에서는 이때 ARCn이 이전 Redo Log를 Archive Log로 복사한다.
강제 Log Switch
강제 Log Switch는 현재까지의 Redo를 Archive로 확정해 백업, 장애 점검, Standby Sync 확인에 활용하는 DBA 운영 명령이다.
Standby DB
Standby DB는 Primary DB 장애 시 대체할 수 있도록 Redo를 계속 받아 적용하는 별도의 살아있는 복제 DB이며, DBA가 Data Guard 환경으로 직접 구축하고 지속적으로 관리해야 한다.
