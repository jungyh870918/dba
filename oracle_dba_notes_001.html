2단계 · 개념 설명

Database vs Instance

Database는 디스크 위의 파일 집합이다

  • Datafile
  • Control file
  • Redo log file
  • Archive log

이 파일들은 스스로 일하지 않는다. 그냥 저장되어 있는 대상이다. 즉, 죽어 있는 구조다.

Instance는 DB를 움직이게 만드는 실행 주체다

  • SGA: 여러 프로세스가 함께 보는 공유 메모리
  • Background Process: 실제 쓰기, 복구, 체크포인트를 담당하는 프로세스
User → Server Process → SGA → Background Process → Disk(Database)

즉, 사용자는 디스크 파일에 직접 접근하지 않는다. 항상 Instance를 통과해서만 접근한다.

2단계 · 내부 동작

SGA 안에서 무슨 일이 일어나는가

Shared Pool

  • Library Cache: SQL, 실행계획, PL/SQL 캐시
  • Dictionary Cache: 오브젝트/테이블스페이스/파일 메타정보 캐시

이번 범위에서는 구조만 잡고, 뒤에서 파싱과 라이브러리 캐시 튜닝으로 확장된다.

Data Buffer Cache

Datafile에서 읽은 블록을 메모리에 올려두는 공간이다. Direct Path Read를 제외하면 대부분 여기서 읽고 여기서 수정이 시작된다.

Redo Log Buffer

변경사항 로그를 잠시 담아두는 공간이다. Oracle의 핵심은 데이터를 먼저 디스크에 쓰는 것이 아니라, 변경 이력을 먼저 안전하게 남기는 것이다.

2단계 · 핵심 원리

WAL (Write Ahead Logging)

9~12페이지에서 가장 중요한 개념이다. 순서를 틀리면 안 된다.

1. UPDATE 발생 2. Redo Log Buffer 먼저 기록 3. Data Buffer Cache의 블록 변경 4. COMMIT 시 LGWR가 Redo Log File에 기록 5. DBWR가 나중에 Datafile에 반영

왜 로그를 먼저 쓰는가

Datafile 갱신은 랜덤 write가 많아 느리다. 반면 Redo Log는 append 방식이라 훨씬 빠르다. 그래서 Oracle은 데이터 정합성은 로그로 보장하고, 실제 데이터 쓰기는 뒤로 미룬다.

Fast Commit의 본질: Commit은 “데이터 파일에 다 썼다”가 아니라, 복구 가능한 로그가 디스크에 안전하게 기록되었다는 뜻이다.
2단계 · 버퍼 캐시

Data Buffer Cache에서 UPDATE가 일어날 때

예를 들어 20개 블록이 변경되면, 8KB 블록 기준 약 160KB의 변경이 메모리에서 먼저 일어난다. 중요한 점은 COMMIT 시점에도 Datafile에 즉시 쓰지 않는다는 것이다.

  • 변경은 먼저 메모리(Data Buffer Cache)에 반영된다.
  • Commit은 Redo 기록 완료로 끝난다.
  • Datafile 반영은 DBWR가 나중에 한다.
실무에서 자주 나오는 오해: “commit 했으니 파일에도 바로 저장됐겠지”는 틀린 생각이다. Oracle은 의도적으로 그렇게 하지 않는다.
3단계 · 흐름 연결

지금 보는 페이지가 전체 구조에서 어디인가

[User] ↓ [Server Process] ↓ [SGA] ├─ Shared Pool : SQL 확인 / 메타정보 확인 ├─ Buffer Cache : 블록 읽기 / 변경 └─ Redo Buffer : 변경 이력 기록 ↓ [LGWR → Redo Log File] [DBWR → Datafile (나중)]

이 흐름은 뒤에서 undo, checkpoint, recovery, segment 구조, 튜닝 개념까지 전부 이어진다. 지금은 뼈대를 세우는 단계다.

4단계 · 프로세스 관계

한 사용자가 접속할 때 프로세스는 어떻게 연결되는가

기본 모델은 Dedicated Server다. 이 구조에서는 서버 프로세스 1개가 모든 유저를 round-robin으로 처리하지 않는다.

[User1] → [Server Process1] [User2] → [Server Process2] [User3] → [Server Process3]

즉, 기본은 유저 1명 ↔ 서버 프로세스 1개다.

왜 1:1 구조를 쓰는가

  • 세션 상태, 커서 상태, 트랜잭션 상태를 분리해야 한다.
  • PGA가 프로세스 전용 메모리이기 때문이다.
  • 동시 요청 처리는 DB가 직접 round-robin 하는 것이 아니라, OS 스케줄러가 각 프로세스를 병렬 스케줄링한다.
정리: Oracle 기본 구조는 “서버 프로세스 하나가 여러 유저를 순차 처리”가 아니라, 여러 서버 프로세스가 동시에 동작하는 구조다.
4단계 · 예외 구조

Shared Server는 무엇이 다른가

예외적으로 Shared Server 구조에서는 여러 사용자의 요청이 Dispatcher를 거쳐 Shared Server 풀로 들어간다.

User → Dispatcher → Shared Server Pool

이 경우에는 여러 사용자의 요청을 제한된 수의 서버 프로세스가 나눠 처리하므로, 겉으로 보면 순차적 또는 큐 기반 처리처럼 보일 수 있다. 다만 실무의 기본 설명은 Dedicated Server를 기준으로 잡는 것이 맞다.

4단계 · 운영 문제

Too Many Connections가 왜 터지는가

Oracle 기본 구조에서는 연결 수가 곧 프로세스 수가 되고, 프로세스 수는 다시 PGA 사용량과 연결된다.

User 1명 → Server Process 1개 → PGA 1개 User 1000명 → Server Process 1000개 → PGA 1000개
  • 메모리 사용량 폭증
  • OS context switching 증가
  • 프로세스/세션 한도 초과

그래서 ORA-00020, ORA-12516 같은 문제가 발생할 수 있다.

4단계 · 해결 구조

DB Pooling의 개념

Connection Pooling은 DB 연결을 사용자 수만큼 만들지 않고, 미리 일정 개수만 만들어 놓고 빌려 쓰는 방식이다.

[User 1000명] ↓ [WAS Connection Pool (예: 50개)] ↓ [DB Server Process 50개]

핵심은 User 수와 DB connection 수를 분리하는 것이다.

동작 방식

  • 요청이 오면 pool에서 connection을 하나 가져온다.
  • 작업이 끝나면 연결을 종료하지 않고 pool에 반납한다.
  • 재사용하므로 생성 비용을 줄이고 DB를 보호한다.
핵심 효과: DB 보호, 연결 생성 비용 감소, 부하 안정화.
4단계 · 실무 포인트

Pooling 운영 시 주의할 점

  • Pool size가 너무 작으면 애플리케이션이 connection을 기다리며 대기한다.
  • Pool size가 너무 크면 결국 DB 프로세스 수가 늘어나 DB가 다시 힘들어진다.
  • Idle connection이 오래 살아 있으면 자원 낭비가 생긴다.
  • Long transaction이 connection을 오래 점유하면 pool 고갈이 발생한다.
  • Dead connection이 남아 있으면 pool에서 장애가 반복된다.

그래서 WAS 쪽에서는 max pool size, min pool size, timeout, validation query 설정이 중요하고, DB 쪽에서는 processes, sessions 한도를 함께 봐야 한다.

5단계 · 핵심 요약

이번 범위 요약

Oracle은 데이터를 바로 디스크에 쓰는 DB가 아니다. 변경 이력(Redo)을 먼저 안전하게 남기고, 나중에 DBWR가 Datafile에 반영한다. 기본 접속 구조는 User 1명 = Server Process 1개이며, Too Many Connections 문제는 이 구조 때문에 발생한다. DB Pooling은 User 수와 DB connection 수를 분리해서 DB를 보호하는 핵심 운영 장치다.
