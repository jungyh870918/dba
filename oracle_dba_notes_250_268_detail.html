Oracle DBA Notes

250~268페이지 정리

테이블을 생성한 이후 실제 운영 단계에서 수행하는 변경 작업을 다루는 구간이다. 컬럼 추가, Tablespace 이동, VARCHAR2 확장, Comment 관리, Synonym 설정처럼 실무에서 반복적으로 발생하는 운영 작업이 핵심이다.

1. 운영의 핵심은 생성이 아니라 변경 관리

초급자는 create table이 끝이라고 생각하지만, 실제 DBA는 그 이후를 더 중요하게 본다.

  • 컬럼 추가 요청
  • 저장 공간 부족
  • Tablespace 정책 변경
  • 문자열 길이 부족
  • 다른 계정 접근 필요
운영 = 지속적인 구조 변경 관리

2. 컬럼 추가는 생각보다 무거운 작업

alter table edu01.t_cust add ( input_date varchar2(8), updt_date varchar2(8) );

문장은 단순하지만 대용량 테이블에서는 매우 조심해야 한다. DDL Lock, Redo 증가, Undo 사용, 서비스 지연이 발생할 수 있다.

특히 업무 시간 중 수행하면 애플리케이션이 멈춘 것처럼 보일 수 있다. 그래서 DBA는 작업 시간과 영향 범위를 먼저 판단한다.

“추가 가능”과 “운영 중 안전하게 가능한가”는 완전히 다른 문제다.

3. Tablespace 이동

alter table edu01.t_cust move tablespace users;

이것은 단순 복사가 아니라 실제 저장 위치를 바꾸는 작업이다. 용량 부족, I/O 분산, 성능 개선, 스토리지 정책 변경 때문에 수행한다.

가장 중요한 문제

MOVE 이후 기존 인덱스가 unusable 상태가 될 수 있다. 그러면 옵티마이저는 인덱스를 쓰지 못하고 Full Table Scan으로 바뀐다.

alter index index_name rebuild;

이 rebuild를 놓치면 갑자기 서비스가 느려지는 대표적인 장애가 발생한다.

4. VARCHAR2 최대 크기 확장

기본적으로 Oracle은 VARCHAR2 길이에 제한이 있다. 긴 문자열, JSON, 대형 메모 데이터를 저장해야 하면 부족해진다.

alter system set max_string_size=extended scope=spfile;

하지만 이건 단순 파라미터 수정이 아니다. PDB를 upgrade mode로 열고, utl32k.sql을 실행한 뒤 재기동해야 한다.

이것은 Dictionary 구조 자체를 바꾸는 작업이다. 그래서 restart와 upgrade mode가 필요하다.

5. Comment는 미래 DBA를 위한 문서

comment on table edu01.t_cust is '고객'; comment on column edu01.t_cust.cust_no is '고객번호';

시간이 지나면 아무도 컬럼의 원래 의미를 기억하지 못한다. 특히 퇴사자가 만든 구조는 해석 자체가 어려워진다.

그래서 Comment는 단순 설명이 아니라 운영을 위한 보험이다. 미래의 DBA가 구조를 이해할 수 있게 해주는 가장 현실적인 문서다.

select * from dba_tab_comments; select * from dba_col_comments;
설계 문서보다 DB Dictionary가 더 정확하다. DB가 현재 상태 그 자체이기 때문이다.

6. Synonym

다른 스키마의 오브젝트를 사용할 때 매번 schema명을 붙이는 불편함을 줄이기 위해 사용한다.

select * from edu01.t_cust;

Synonym을 만들면 아래처럼 단순하게 사용할 수 있다.

create public synonym t_cust for edu01.t_cust; select * from t_cust;

Public vs Private

Public Synonym은 모든 계정에 적용되고, Private Synonym은 특정 계정만 사용 가능하다.

Public Synonym은 편하지만 너무 많아지면 실제 owner 추적이 어려워져 장애 분석이 힘들어진다.

한 줄 정리

create table은 시작일 뿐이다. 진짜 DBA 업무는 그 이후 변경을 안전하게 관리하는 것이다.
