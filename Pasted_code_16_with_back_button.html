321p7. Clustered Index 테이블

Clustered Index 테이블은 키 값이 같은 레코드가 같은 블록에 모이도록 저장하는 구조다. 같은 부서코드, 같은 그룹 키처럼 함께 조회되는 데이터가 물리적으로 가까이 있으면 I/O를 줄일 수 있다.

  • 키 값이 같은 레코드를 한 블록에 모아 저장한다.
  • 한 블록에 모두 담을 수 없으면 새 블록을 할당하고 Cluster Chain으로 연결한다.
  • 다중 테이블 인덱스 클러스터는 여러 테이블의 레코드를 같은 블록에 저장할 수 있다.
  • 즉, 물리적으로 조인된 상태처럼 저장하는 구조다.
쉽게 말하면
일반 테이블은 각 테이블의 행이 자기 블록에 따로 저장된다. Clustered Index는 “항상 같이 조회되는 데이터”를 한 공간에 몰아두는 방식이다. 조인 비용을 줄일 수 있지만, 데이터 분포가 나쁘거나 DML이 많으면 체인과 관리 비용이 커질 수 있다.

322~330p8. Constraint: Primary Key / Foreign Key

Primary Key의 특성

  • PK는 Not Null + Unique를 동시에 보장한다.
  • PK 칼럼 수는 최소화하는 것이 좋다.
  • PK 생성 시 기본적으로 Constraint 이름과 동일한 인덱스가 생성된다.
  • 이미 만들어진 인덱스를 PK에 사용할 수도 있다.

PK 생성 방식 3가지

방식의미주의점
테이블 생성 시 PK 지정CREATE TABLE 문 안에서 constraint 지정인덱스 테이블스페이스를 세밀하게 분리하기 어렵다.
테이블 생성 후 PK AddALTER TABLE ADD CONSTRAINT 사용USING INDEX TABLESPACE로 인덱스 위치 지정 가능
기존 인덱스 지정먼저 인덱스를 만들고 PK가 그 인덱스를 사용인덱스 선두 칼럼이 PK 칼럼이면 추가 칼럼이 있어도 사용 가능
 
-- 테이블 생성 시 PK 지정
create table edu01.t_cust
( cust_no  varchar2(6)   not null,
  cust_nm  varchar2(50)  not null,
  addr     varchar2(200) not null,
  tel_no   varchar2(15)  not null,
  join_dt  varchar2(8)   default to_char(sysdate, 'yyyymmdd'),
  class_cd varchar2(4)   check (class_cd in ('CLSS', 'GOLD', 'VIP', 'VVIP')),
  sal      number        default 0,
  constraint pk_cust primary key(cust_no)
) tablespace users;

-- 테이블 생성 후 PK 추가
alter table edu01.t_cust
add constraint pk_cust primary key(cust_no)
using index tablespace users;

-- 기존 인덱스를 PK에 사용
create index edu01.ix_cust_pk
on edu01.t_cust(cust_no, join_dt) tablespace users;

alter table edu01.t_cust
add constraint pk_cust primary key(cust_no)
using index edu01.ix_cust_pk;

Constraint 이름 변경 / 삭제

alter table edu01.t_cust rename constraint pk_cust to cust_pk;
alter table edu01.t_cust drop constraint cust_pk;

Foreign Key의 핵심

Foreign Key는 자식 테이블의 값이 부모 테이블의 PK 또는 Unique Key에 존재하는지 보장한다. 예를 들어 주문 테이블의 cust_no는 고객 테이블의 cust_no를 참조할 수 있다.

  • 참조 대상은 부모 테이블의 Primary Key 또는 Unique Index여야 한다.
  • 조인 연결 칼럼 중 한쪽은 Unique가 보장되어야 1:M 관계가 안정된다.
  • PK는 자동으로 인덱스가 생성되지만, FK 칼럼은 별도 인덱스를 직접 생성해야 한다.
  • FK 칼럼에 NULL이 들어가면 FK 체크가 수행되지 않는다.
실무 주의
FK에 인덱스가 없으면 부모 테이블 삭제/변경 시 자식 테이블을 넓게 검사하거나 Lock 문제가 커질 수 있다. 운영 DB에서는 FK 생성 후 해당 FK 칼럼 인덱스 존재 여부를 반드시 확인한다. 
alter table edu01.t_order
add constraint fk_order01 foreign key(cust_no)
references edu01.t_cust(cust_no)
[ on delete set null ]
[ deferrable initially deferred ];
옵션동작실무 판단
Restrict 기본값부모 데이터 삭제 시 자식 데이터가 있으면 삭제 불가가장 안전하며 권장되는 방식
ON DELETE SET NULL부모 삭제 시 자식 FK 칼럼을 NULL로 변경자식 FK 칼럼이 Nullable이어야 함
ON DELETE CASCADE부모 삭제 시 자식 행도 모두 삭제매우 위험. 대량 삭제 장애로 이어질 수 있음
DEFERRABLE INITIALLY IMMEDIATEDML 수행 즉시 제약조건 점검기본값
DEFERRABLE INITIALLY DEFERREDCOMMIT 시점에 제약조건 점검부모/자식 데이터를 한 트랜잭션에서 동시에 넣을 때 유리

Constraint 활성화 / 비활성화

alter table edu01.t_order
disable constraint fk_order01;

alter table edu01.t_order
enable [NOVALIDATE] constraint fk_order01;
NOVALIDATE의 느낌
기존 데이터 전체를 다시 검사하지 않고 이후 데이터부터 제약조건을 적용하는 방식으로 이해하면 된다. 대용량 테이블에서 제약조건을 다시 켤 때 운영 부담을 줄일 수 있지만, 기존 데이터의 무결성 보장 범위는 별도로 판단해야 한다.

331~337p9. 파티션 테이블과 Partition Pruning

파티션 테이블은 테이블 또는 인덱스를 파티션 단위로 나누어 저장하는 구조다. 파티션 키에 따라 데이터가 물리적으로 별도 세그먼트에 저장된다.

관리적 이점
파티션 단위 백업, 추가, 삭제, 변경
성능적 이점
파티션 단위 조회와 DML, 경합 및 부하 분산
보관 정책
오래된 데이터 백업 후 파티션 단위 삭제

파티션 유형

유형기준대표 사용처
Range값의 범위로 분할날짜 칼럼 기준 월별/분기별 데이터
Hash해시 함수 결과로 분산데이터 균등 분산, 병렬 처리, DML 경합 분산
List사용자가 정한 값 목록으로 분할지역, 지점, 코드 그룹
CompositeRange/List 안에 서브 파티션 구성날짜 + 지역, 날짜 + 해시 등 복합 구조

Partition Pruning

Partition Pruning은 옵티마이저가 SQL의 테이블과 조건절을 분석하여 불필요한 파티션을 액세스 대상에서 제외하는 기능이다. 파티션 테이블 성능의 핵심이다.

구분결정 시점대표 조건
Static Pruning컴파일 시점파티션 키를 상수 조건으로 조회
Dynamic Pruning실행 시점파티션 키를 바인드 변수로 조회
중요
파티션 키 칼럼을 함수로 감싸거나 데이터 타입이 맞지 않아 묵시적 형 변환이 발생하면 Partition Pruning이 정상 작동하지 않을 수 있다. 예를 들어 날짜 파티션 키에 to_char(date_col) 조건을 걸면 파티션을 잘라내지 못하고 넓게 읽을 위험이 있다.

338~343p파티션 인덱스: Local / Global / Prefixed

Local Partition Index

Local 파티션 인덱스는 테이블 파티션과 1:1로 대응되는 인덱스다. 사용자가 인덱스 파티션 키를 따로 지정하지 않아도 DBMS가 테이블 파티션과의 관계를 자동 유지한다.

Global Partition Index

Global 파티션 인덱스는 테이블 파티션과 독립적으로 구성되는 인덱스다. 테이블 파티션 구조와 인덱스 파티션 구조가 반드시 1:1로 맞지 않는다.

Prefixed vs Nonprefixed

구분의미지원
Prefixed파티션 키 칼럼이 인덱스 선두 칼럼에 위치Global O / Local O
Nonprefixed파티션 키가 선두가 아니거나 인덱스에 없음Global X / Local O

인덱스 파티셔닝 가이드

전략유리한 경우주의점
NonPartitioned Index파티션 키 조건이 자주 누락되는 OLTP, NL Join의 Inner 테이블 접근, ORDER BY 대체 목적테이블 파티션 이동/삭제 시 Unusable 가능
Global Prefixed Index인덱스 경합 분산, 여러 Local 인덱스 파티션 접근이 비효율적인 경우파티션 이동/삭제 시 Unusable 가능
Local Index파티션 단위 관리와 함께 움직이는 구조, 파티션 키 조건이 잘 들어오는 경우파티션 키 조건이 없으면 여러 인덱스 파티션을 읽을 수 있음

344~349p파티션 테이블 관리 SQL

Range Partition Table 생성

create table edu01.t_cust_p
( cust_no  varchar2(6)   not null,
  cust_nm  varchar2(50)  not null,
  addr     varchar2(200) not null,
  tel_no   varchar2(15)  not null,
  join_dt  varchar2(8)   default to_char(sysdate, 'yyyymmdd'),
  class_cd varchar2(4),
  sal      number        default 0
)
partition by range(join_dt)
( partition pt_202201 values less than ('20220201') tablespace users,
  partition pt_202202 values less than ('20220301') tablespace users,
  partition pt_202203 values less than ('20220401') tablespace users,
  partition pt_202204 values less than ('20220501') tablespace users,
  partition pt_202205 values less than ('20220601') tablespace users,
  partition pt_202206 values less than ('20220701') tablespace users,
  partition pt_202207 values less than ('20220801') tablespace users,
  partition pt_202208 values less than ('20220901') tablespace users,
  partition pt_202209 values less than ('20221001') tablespace users,
  partition pt_202210 values less than ('20221101') tablespace users,
  partition pt_202211 values less than ('20221201') tablespace users,
  partition pt_202212 values less than ('20230101') tablespace users,
  partition pt_max    values less than (maxvalue)  tablespace users
);

파티션 테이블에서 Primary Key

교재에서는 파티션 테이블에서 PK 생성 시 파티션 키 칼럼이 추가되어야 PK 생성 가능하다고 정리한다. 예제에서는 cust_no와 join_dt를 함께 PK 인덱스에 포함한다.

create index edu01.pk_t_cust_p
on edu01.t_cust_p(cust_no, join_dt) local;

alter table edu01.t_cust_p
add constraint pk_t_cust_p primary key(cust_no, join_dt)
using index pk_t_cust_p;

파티션 추가: ADD PARTITION

MAXVALUE 파티션이 없는 경우에는 단순히 새 파티션을 추가할 수 있다.

alter table edu01.t_cust_p
add partition pt_202301
values less than ('20230201') tablespace users;

파티션 추가: SPLIT PARTITION

MAXVALUE 파티션이 이미 있으면 새 범위를 추가하는 대신 기존 pt_max를 쪼개야 한다.

alter table edu01.t_cust_p
split partition pt_max at ('20230201')
into (
  partition pt_202301 tablespace users,
  partition pt_max    tablespace users
);
장애 포인트
Split, Drop, Truncate 같은 파티션 작업은 Non Partitioned Index를 Unusable 상태로 만들 수 있다. 특히 PK 인덱스가 Unusable이 되면 데이터 입력과 삭제가 막히는 장애로 이어질 수 있다.

파티션 이름 변경 / 삭제

alter table edu01.t_cust_p
rename partition pt_202210 to pt2022101;

alter table edu01.t_cust_p
drop partition pt_max;

파티션 인덱스 생성

-- Non-partitioned index: 일반 인덱스 생성과 동일
create index edu01.ix_cust_p01
on edu01.t_cust_p(join_dt)
tablespace users;

-- Local partitioned index: 테이블 파티션과 동일 영역에 생성
create index edu01.ix_cust_p01
on edu01.t_cust_p(join_dt) local;

실무 DBA 관점 최종 정리

주제운영에서 보는 관점확인 포인트
Clustered Index같이 읽는 데이터를 물리적으로 모아 I/O를 줄이는 구조체인 증가, DML 비용, 실제 조인 패턴
Primary Key논리적 무결성과 물리적 인덱스가 연결됨PK 인덱스 위치, 칼럼 수, 기존 인덱스 재사용 여부
Foreign Key참조 무결성 보장. 하지만 FK 인덱스 없으면 Lock/I/O 문제 발생 가능FK 인덱스 존재 여부, cascade 사용 여부
Partition Table대용량 테이블을 관리 단위와 성능 단위로 쪼개는 구조파티션 키 설계, 보관 주기, pruning 가능 조건
Partition Pruning파티션 성능의 핵심. 필요한 파티션만 읽게 함파티션 키 가공 여부, 묵시적 형 변환 여부
Local Index파티션 테이블 관리와 궁합이 좋음조건절에 파티션 키가 들어오는지
Global / NonPartitioned IndexOLTP 조회에는 유리할 수 있으나 파티션 관리 작업에 취약Unusable 발생 가능성, rebuild 계획
핵심 한 줄
이 범위의 핵심은 “오브젝트를 그냥 만드는 법”이 아니라, 무결성 제약조건과 파티션/인덱스 구조가 운영 중 DML, 조회 성능, 장애 포인트에 어떤 영향을 주는지를 이해하는 것이다.
