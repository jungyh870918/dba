350~352p10. View

View는 SQL을 사전에 정의해서 DB 오브젝트로 등록해 둔 것이다. 중요한 점은 View 자체가 데이터를 보유하지 않는다는 것이다. 실제 데이터는 기본 테이블에 있고, View는 그 테이블을 바라보는 논리적 창구 역할을 한다.

  • 민감 정보 접근 통제에 활용할 수 있다.
  • 자주 쓰는 SQL 코드를 재사용할 수 있다.
  • 복잡한 테이블 구조를 사용자에게 더 읽기 좋은 형태로 제공할 수 있다.
  • 기본 테이블 권한은 제거하고 View에만 권한을 줄 수 있어 보안성을 높일 수 있다.

View의 특징

특징의미운영 관점
물리적 저장공간 없음View는 데이터를 직접 저장하지 않는다.조회 시 기본 테이블 SQL로 풀려 실행된다.
인덱스 생성 불가일반 View 자체에는 인덱스를 만들 수 없다.성능 문제는 기본 테이블 인덱스/SQL 구조로 해결해야 한다.
기본 테이블 PK 미정의 시 DML 제한입력, 수정, 삭제가 불가능할 수 있다.View는 주로 조회용으로 설계하는 것이 안전하다.
Alter View 명령어 사용 불가수정은 create or replace view 방식으로 처리한다.배포 시 기존 의존 오브젝트 invalid 여부를 함께 확인한다.
View 위에 View 생성 가능이미 정의된 View를 이용해 또 다른 View 정의 가능SQL 복잡성과 튜닝 난이도 증가로 권장하지 않는다.
 
create or replace view edu01.vw_cust as
select cust_no, cust_nm, addr
from   t_cust;

create or replace public synonym vw_cust for edu01.vw_cust;

select *
from   vw_cust;
쉽게 말하면
View는 “가상 테이블”처럼 보이지만 실제 테이블은 아니다. DBA 관점에서는 View를 보안·가독성·SQL 재사용을 위한 인터페이스로 보고, 성능 문제는 결국 View 내부 SQL과 기본 테이블 인덱스에서 찾아야 한다.

353~356p11. Stored Procedure

Stored Procedure는 PL/SQL로 작성한 DB 내부 프로그램이다. 과거 2-Tier 환경에서는 Client/Server 사이의 Biz Logic 처리를 DB 프로시저로 많이 구현했다. 현재처럼 Application Server가 있는 3-Tier 이상 구조에서는 형상관리와 책임 분리 문제 때문에 무분별한 사용은 권장되지 않는다.

장점
DB 내부에서 처리하므로 App 서버와 DB 사이 네트워크 왕복을 줄일 수 있다.
주의
업무 로직이 DB 안에 숨어 형상관리, 테스트, 배포 관리가 어려워질 수 있다.
적합
대량 배치, DB 근처에서 끝내야 하는 데이터 처리, 제한된 권한 인터페이스

Stored Procedure의 장점

구분내용실무 해석
보안프로시저를 통해서만 데이터 접근하도록 설정 가능사용자는 테이블 권한 없이 프로시저 실행 권한만 가질 수 있다.
성능서버 간 주고받는 데이터량 감소, 컴파일 상태 보관대량 처리·배치에는 App에서 반복 호출하는 것보다 유리할 수 있다.
생산성검증된 프로시저 재사용 가능여러 프로그램에서 같은 로직을 재사용할 수 있다.
무결성응용프로그램 무결성과 일관성 향상데이터 구조 변경 시 프로시저 재컴파일로 대응 가능한 경우가 있다.
 
create or replace procedure edu01.sp_cust
( p_cust edu01.t_cust.cust_no%type
, p_num  number)
is
  v_sal  number;
  v_addr varchar2(100);
begin
  select sal into v_sal
  from   t_cust
  where  cust_no = p_cust;

  v_sal := v_sal * p_num;

  if v_sal >= 4500 then
    update t_cust
    set    addr = 'GOOD'
    where  cust_no = p_cust;
  else
    update t_cust
    set    addr = 'so so'
    where  cust_no = p_cust;
  end if;

  commit;
end sp_cust;
/

create or replace public synonym sp_cust for edu01.sp_cust;

execute sp_cust('100002', 4);
select * from t_cust;
운영 주의
Procedure 안에 commit이 있으면 호출한 애플리케이션의 트랜잭션 경계와 충돌할 수 있다. 배치성 프로시저라면 명확히 설계할 수 있지만, 서비스 로직에서 호출되는 프로시저라면 commit 위치는 특히 조심해야 한다.

357~358p12. Stored Function

Stored Function은 값을 반환하는 PL/SQL 프로그램이다. SQL 안에서 내장 함수처럼 호출할 수 있지만, 교재는 사용자 정의 함수/프로시저가 Native 코드로 완전 컴파일되는 것이 아니라 별도 실행 엔진을 통해 인터프리터 방식으로 실행된다는 점을 강조한다.

성능 저하 원인: Context Switching

  • SQL 실행 엔진이 PL/SQL 함수를 호출할 때마다 실행 컨텍스트가 바뀐다.
  • SQL 엔진이 사용하던 레지스터 정보를 백업하고, PL/SQL 엔진 실행 후 다시 복원한다.
  • 함수 내부 쿼리는 호출될 때마다 Execute, Fetch Call이 재귀적으로 발생한다.
  • Parse Call은 최초 1번만 수행되더라도, 반복 호출이 많으면 누적 비용이 커진다.
create or replace function edu01.fn_cust
(p_cust_no edu01.t_cust.cust_no%type)
return edu01.t_cust.cust_nm%type
is
  v_cust_nm edu01.t_cust.cust_nm%type;
begin
  select cust_nm into v_cust_nm
  from   t_cust
  where  cust_no = p_cust_no;

  return v_cust_nm;
end fn_cust;
/

create or replace public synonym fn_cust for edu01.fn_cust;

select fn_cust('0011486') cust_nm
from   dual;
DBA가 피하고 싶은 패턴
대량 조회 SQL의 SELECT 리스트나 WHERE 절에서 사용자 정의 함수를 행마다 호출하는 패턴이다. 10건이면 괜찮아 보여도 1,000만 건이면 함수 호출, 내부 SQL, Context Switching이 폭발한다. 성능 문제가 생기면 먼저 “SQL 안에서 함수가 반복 호출되는지”를 확인한다.

359~364p13. Trigger

Trigger는 특정 테이블에 Insert / Update / Delete가 발생하면 자동 수행되는 PL/SQL 프로그램이다. 사용자는 직접 호출하지 않는다. DML 이벤트가 발생하면 Oracle이 내부적으로 트리거를 실행한다.

Trigger의 용도

  • 자동 파생 열값 생성
  • 잘못된 트랜잭션 방지
  • 복잡한 보안 권한 강제 수행
  • 이벤트 로깅 또는 감사 작업
  • 동기화 테이블 복제 작업
  • 테이블 액세스에 대한 통계 수집

Trigger 유형

유형동작실무 해석
행 트리거변경된 행 수만큼 수행1만 건 update면 트리거도 1만 번 수행될 수 있다.
명령문 트리거많은 행이 변경되어도 SQL 문장 1회당 1번 수행감사 로그나 작업 단위 기록에 적합하다.
Before Trigger명령문 실행 전 수행입력값 보정, 검증 등에 사용한다.
After Trigger명령문 실행 후 수행변경 이력 적재, 감사 기록 등에 사용한다.
운영 지옥 포인트
트리거가 많아지면 내부 종속성이 숨어버린다. Insert가 느린데 원인이 Insert SQL이 아니라 뒤에서 자동 실행되는 Trigger일 수 있다. 또한 Insert Trigger가 Invalid 상태가 되면 해당 테이블 Insert 자체가 막힐 수 있다. Update/Delete Trigger도 동일하다.

이력 테이블 준비

create table edu01.t_cust_hist
( cust_no  varchar2(6)
, updt_dh  varchar2(14)
, cust_nm  varchar2(50)
, addr     varchar2(200)
, tel_no   varchar2(15)
, constraint pk_cust_hist primary key(cust_no, updt_dh)
);

create or replace public synonym t_cust_hist for edu01.t_cust_hist;

Update/Delete 발생 시 변경 전 값 이력 저장

create or replace trigger edu01.tr_cust_u
after update or delete
on edu01.t_cust
for each row
declare
  -- 필요시 변수 선언
begin
  insert into t_cust_hist(cust_no, updt_dh, cust_nm, addr, tel_no)
  values (:old.cust_no, to_char(sysdate, 'yyyymmddhhmiss'),
          :old.cust_nm, :old.addr, :old.tel_no);
end;
/

/*
trigger에서는 commit을 수행하지 않음.
commit은 update, delete를 발행한 트랜잭션에서 수행한다.
*/

실행 확인

update t_cust
set    cust_nm = '강감찬'
where  cust_no = '100004';

commit;

select *
from   t_cust_hist;
왜 Trigger 안에서는 commit을 하지 않는가
Trigger는 원본 DML의 일부로 실행된다. 원본 update/delete와 이력 insert가 같은 트랜잭션으로 묶여야 정합성이 맞다. Trigger 안에서 독립적으로 commit해버리면 원본 작업은 rollback됐는데 이력만 남는 식의 정합성 문제가 생길 수 있다.

365~366p14. Sequence

Sequence는 사용자의 액션에 따라 일관되게 증가하는 숫자를 제공하는 오브젝트다. 주로 테이블의 Primary Key 채번에 사용한다.

왜 Sequence를 쓰는가

  • 채번 테이블을 사용하면 여러 서버 프로세스가 동시에 같은 테이블에 접근해 병목이 발생한다.
  • Sequence는 Memory Caching 기능으로 채번 성능을 높인다.
  • 동시 Insert가 많은 시스템에서 채번 병목을 줄이는 표준적인 방식이다.
create sequence edu01.sq_cust
increment by 1
start with   1
minvalue     1
maxvalue     999999999999
nocycle
cache        30
noorder;

create or replace public synonym sq_cust
for edu01.sq_cust;

select sq_cust.nextval from dual;
select sq_cust.currval from dual;
옵션의미운영 관점
increment by증가 간격마이너스 값을 주면 역증가 가능
start with시작 값기존 데이터와 충돌하지 않게 설정
nocycle / cycle최대값 도달 후 재사용 여부PK 채번에는 보통 nocycle
cache / nocache메모리에 미리 번호 확보성능은 좋아지지만 shutdown 시 캐시된 수만큼 번호 공백 가능
order / noorderRAC 환경에서 순서 보장 여부Single Instance에서는 큰 의미 없음. RAC에서 order는 성능 부담 가능
중요한 감각
Sequence 번호는 “연속성 보장”이 아니라 “고유한 채번”을 위한 도구에 가깝다. 중간 번호가 비는 것은 정상적인 운영 현상일 수 있다. 회계번호처럼 절대 공백이 없어야 하는 번호와 PK용 Sequence는 설계를 분리하는 것이 안전하다.

367~372p15~16. Directory / DB Link / Dictionary View

Directory

Directory는 OS 파일시스템의 특정 디렉토리를 Oracle 오브젝트로 연결하는 설정이다. expdp/impdp 같은 Data Pump, 파일 Read/Write 패키지인 UTL_FILE 등에서 활용된다.

  • OS상의 실제 Directory 존재 여부와 무관하게 Oracle 오브젝트 생성은 가능하다.
  • 하지만 실제 OS Directory가 없으면 해당 Directory를 사용하는 SQL 실행 시점에 에러가 발생한다.
  • Directory를 사용할 계정에는 Read, Write 권한을 별도로 부여해야 한다.
create or replace directory d_tmp1 as 'c:\tmp\';

grant read, write on directory d_tmp1 to yoon;

select owner, directory_name, directory_path
from   dba_directories
where  directory_name = 'D_TMP1';
운영 주의
Directory는 DB 오브젝트지만 실제 파일 접근은 OS 권한과 경로 상태에 영향을 받는다. DB에는 생성되어 있는데 OS 디렉토리가 없거나 Oracle 프로세스 계정에 OS 권한이 없으면 Data Pump나 UTL_FILE 작업이 실패한다.

DB Link

DB Link는 원격 DB를 Link로 연결하여 SQL에서 Local DB처럼 사용하는 기능이다. 테이블 지정 시 table_name@db_link 형태로 접근한다.

  • 사용은 편리하지만 관리 난이도가 높다.
  • 가급적 응용 프로그램에서 상시 사용하는 것은 자제하는 것이 바람직하다.
  • DBA가 관리 목적 또는 임시 작업 목적으로 사용하는 쪽이 안전하다.
  • Public DB Link와 일반 DB Link로 구분하여 생성할 수 있다.
create public database link lnk_cdb
connect to system identified by manager1
using 'orcl';

select owner, db_link, host, con_id
from   cdb_db_links@lnk_cdb;
Using 절
using 절에는 서버의 TNS Name을 줄 수 있고, tnsnames.ora 파일의 절대 경로 지정도 가능하다.
DBA가 조심하는 이유
DB Link는 원격 DB 장애, 네트워크 지연, 계정/비밀번호 관리, 권한 범위, 분산 트랜잭션 문제를 함께 끌고 온다. SQL 한 줄은 간단해 보여도 장애 원인은 로컬 DB 밖에 있을 수 있다.

일반적인 Use Case와 설정 패턴

350~372페이지의 오브젝트들은 문법 자체보다 현업에서 어디까지 허용하고, 어떤 상황에서 쓰며, 어떤 부분을 조심해야 하는지가 더 중요하다. 특히 View, Stored Procedure, Function, Trigger는 애플리케이션 로직과 DB 내부 로직의 경계에 걸려 있기 때문에 운영 기준을 명확히 잡아야 한다.

1. View: 가장 일반적이고 자주 쓰이는 조회 인터페이스

View는 실무에서 꽤 자주 쓴다. 가장 흔한 목적은 개발자나 리포트 사용자가 운영 테이블을 직접 만지지 않도록, 필요한 컬럼과 조건만 노출한 안전한 조회 창구를 제공하는 것이다.

일반적 Use Case예시DBA 관점
민감 컬럼 숨기기고객 테이블에서 주민번호, 전화번호, 내부 관리 컬럼 제외기본 테이블 권한은 막고 View에만 SELECT 권한 부여
자주 쓰는 조인 제공주문 + 고객 + 결제 상태를 묶은 주문 요약 View개발자는 편해지지만 View 내부 SQL 성능은 반드시 점검
공통 조건 고정deleted_yn = 'N', use_yn = 'Y' 같은 조건 포함실수로 삭제/비활성 데이터를 조회하는 문제를 줄임
리포트/통계용 제공월별 매출 요약, 고객 등급별 집계 대상 View리포트 사용자가 복잡한 원천 테이블 구조를 몰라도 조회 가능
테이블 구조 변경 완충원천 테이블 컬럼명이 바뀌어도 View 이름과 컬럼을 유지애플리케이션 영향도를 줄이는 인터페이스 계층 역할
 
-- 개발자에게 제공할 안전한 조회 View 예시
create or replace view app.v_customer_public as
select customer_id,
       customer_name,
       grade,
       created_at
from   owner.customer
where  deleted_yn = 'N';

grant select on app.v_customer_public to app_user;
View 주의점
View는 성능을 자동으로 좋게 해주는 기능이 아니다. View 안에 복잡한 조인, 함수, View 위의 View가 계속 쌓이면 실제 실행 SQL을 추적하기 어려워진다. DBA가 특히 피하고 싶은 구조는 View → View → View 식의 중첩이다.

2. Stored Procedure: 안 쓰는 것이 아니라 용도를 제한해서 쓴다

Stored Procedure는 요즘 추세와 완전히 안 맞는 것이 아니다. 다만 서비스 핵심 비즈니스 로직을 전부 DB Procedure 안에 넣는 방식은 Application Layer 중심의 현대 구조와 잘 맞지 않는다. 반대로 정산, 마감, 대량 배치처럼 DB 내부에서 끝내는 것이 유리한 작업에는 여전히 매우 현실적으로 쓰인다.

권장되는 Use Case이유예시
정산 / 마감 배치대량 데이터를 DB 안에서 처리하는 것이 네트워크 왕복보다 유리일 매출 마감, 월말 수수료 계산, 포인트 소멸
대량 데이터 보정App에서 수백만 건을 가져와 처리하는 것보다 DB에서 직접 처리하는 것이 효율적등급 일괄 갱신, 통계 테이블 적재, Merge 배치
운영성 작업반복되는 보정/복구 절차를 안전하게 캡슐화장애 복구 스크립트, 일괄 수정, 데이터 이관
제한된 권한 인터페이스테이블 직접 권한 없이 정해진 작업만 실행하도록 통제 가능사용자에게 테이블 UPDATE 권한 대신 특정 Procedure 실행 권한만 부여
 
-- 배치/정산용 Procedure 호출 예시
begin
  batch.sp_daily_settlement(date '2026-04-24');
end;
/
일반적인 경계
회원가입, 주문 생성, 결제 처리, 게시글 작성처럼 애플리케이션 흐름과 강하게 연결된 서비스 로직은 보통 Application Layer에서 관리한다. 반면 하루 정산, 월말 마감, 대량 이관, 대량 보정처럼 DB 내부에서 끝내야 효율적인 작업은 Procedure가 여전히 자연스럽다.
Procedure 주의점
Procedure 내부에 commit이 있으면 애플리케이션의 트랜잭션 경계와 충돌할 수 있다. App에서는 rollback했다고 생각했는데 Procedure 안에서 이미 commit된 데이터가 남을 수 있다. 따라서 서비스 호출용 Procedure와 배치용 Procedure의 commit 전략은 반드시 분리해서 설계해야 한다.

3. Stored Function: 편하지만 대량 SQL에서는 가장 조심

Stored Function은 SQL 안에서 함수처럼 쓸 수 있어 편리하다. 하지만 대량 데이터에서 행마다 호출되면 SQL 엔진과 PL/SQL 엔진 사이의 Context Switching이 반복되고, 함수 안에 SQL이 있으면 재귀 SQL까지 발생한다.

상대적으로 안전한 사용위험한 사용
단순 계산, 포맷 변환, 소량 조회, 리포트성 보조 함수수백만 행 SELECT 리스트에서 함수 반복 호출
결과 건수가 작고 호출 횟수가 제한된 화면/관리 쿼리WHERE 절에서 함수 호출로 인덱스 사용을 막는 패턴
업무 의미를 명확하게 캡슐화해야 하는 제한적 경우함수 내부에서 다시 테이블 SELECT를 수행하는 대량 반복 패턴
 
-- 위험할 수 있는 패턴: orders 건수만큼 함수 호출
select order_id,
       fn_get_customer_grade(customer_id)
from   orders;

-- 더 나은 방향: 조인 또는 사전 계산 테이블 검토
select o.order_id,
       c.grade
from   orders o
join   customers c
on     c.customer_id = o.customer_id;
Function 점검 포인트
SQL이 느릴 때 SELECT 절, WHERE 절, ORDER BY 절에 사용자 정의 함수가 있는지 확인한다. 특히 함수 내부에서 다시 SELECT를 수행한다면 겉으로는 SQL 한 문장이어도 내부적으로는 수많은 SQL 호출이 숨어 있을 수 있다.

4. Trigger: 기본은 ENABLE, 하지만 최소한으로 사용

Trigger는 생성하면 기본적으로 활성화된다. 즉 create trigger를 수행하면 별도로 켜지 않아도 DML 발생 시 자동 실행된다. 일반 운영 중에는 Trigger를 자주 켜고 끄는 것이 아니라, 특수 작업 때만 disable/enable하는 것이 일반적이다.

일반적 Use Case설명주의점
변경 이력 저장Update/Delete 전 값을 History 테이블에 저장행 트리거는 변경 행 수만큼 반복 수행됨
감사 로그누가, 언제, 어떤 데이터를 변경했는지 기록로그 테이블 증가량과 보관 정책 필요
created_at / updated_at 자동 세팅입력·수정 시각을 자동으로 기록애플리케이션에서도 같은 값을 관리하면 충돌 가능
제한적 무결성 보완단순 제약조건으로 표현하기 어려운 검증복잡한 업무 규칙을 Trigger에 많이 넣으면 추적 지옥
 
-- Trigger는 생성 즉시 ENABLE 상태
create or replace trigger tr_customer_hist
after update or delete on customer
for each row
begin
  insert into customer_history
  (customer_id, old_name, old_grade, changed_at)
  values
  (:old.customer_id, :old.customer_name, :old.grade, sysdate);
end;
/

-- 예외 상황에서만 비활성화/활성화
alter trigger tr_customer_hist disable;
alter trigger tr_customer_hist enable;

-- 테이블의 모든 Trigger 제어
alter table customer disable all triggers;
alter table customer enable all triggers;
Trigger를 조심하는 이유
Trigger는 사용자가 직접 호출하지 않는 숨은 로직이다. 개발자는 단순히 INSERT 한 줄을 실행했다고 생각하지만, 뒤에서 History Insert, 감사 로그, 다른 테이블 변경이 자동으로 발생할 수 있다. 그래서 장애가 나면 “SQL은 단순한데 왜 느리지?”, “Insert만 했는데 왜 실패하지?” 같은 상황이 생긴다.

5. 현실적인 운영 기준

오브젝트일반적인 취급많이 쓰는 상황피하고 싶은 상황
View자주 사용보안용 조회 창구, 공통 조인/조건 제공, 리포트용 조회View 중첩, 복잡한 View 남발, 성능 확인 없는 제공
Stored Procedure용도 제한해서 사용정산, 마감, 대량 배치, 운영성 보정서비스 핵심 로직을 전부 DB 안에 숨기는 구조
Stored Function최소화해서 사용단순 계산, 소량 조회, 제한적 리포트 보조대량 SQL에서 행마다 반복 호출
Trigger최소한으로 사용변경 이력, 감사 로그, 자동 시각 기록복잡한 업무 로직 삽입, 빈번한 enable/disable이 필요한 구조
실무 감각 한 줄
View는 개발자에게 보여주는 안전한 창구, Procedure는 DB 안에서 끝내야 할 배치/정산 작업, Function은 반복 호출을 조심해야 하는 보조 도구, Trigger는 자동 실행되는 숨은 로직이라 최소한으로 쓰는 대상이다.

실무 DBA 관점 최종 정리

주제운영에서 보는 관점장애/점검 포인트
View보안, SQL 재사용, 가독성 제공용 인터페이스View 중첩, 권한, 기본 테이블 변경에 따른 invalid 여부
Stored ProcedureDB 내부 배치/대량 처리에는 유용하지만 업무 로직 은닉 위험commit 위치, 형상관리, 권한, 재컴파일 필요 여부
Stored FunctionSQL 안에서 편리하게 호출 가능하지만 반복 호출 성능 저하 위험Context Switching, 함수 내부 SQL, 대량 행 반복 호출 여부
TriggerDML 이벤트에 자동 반응하는 내부 로직Invalid Trigger로 DML 실패, 숨은 종속성, 행 트리거 대량 반복
SequencePK 채번 병목을 줄이는 표준 오브젝트cache로 인한 번호 공백, RAC order 옵션, 채번 테이블 사용 여부
DirectoryDB와 OS 파일시스템을 연결하는 오브젝트OS 경로 존재, OS 권한, read/write 권한
DB Link원격 DB를 로컬처럼 접근하게 하는 편의 기능네트워크, 원격 DB 장애, 계정 관리, 분산 트랜잭션, 상시 사용 자제
핵심 한 줄
이 범위의 핵심은 “오브젝트를 만드는 문법”이 아니라, DB 안에 숨은 로직과 외부 연결점이 운영 중 성능·권한·장애 추적에 어떤 부담을 만드는지를 이해하는 것이다.
