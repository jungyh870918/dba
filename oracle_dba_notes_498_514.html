Oracle Multitenant · PDB 기동/종료, 접속, 생성, Unplug/Plug, User 관리, Tablespace 조회

이 범위는 Oracle Multitenant 환경에서 실제 DBA가 자주 다루는 PDB 운영 흐름을 정리한다. 핵심은 CDB와 PDB의 역할을 구분하고, 업무용 데이터와 사용자는 가급적 PDB 안에서 관리하며, PDB의 상태 확인, 기동/종료, 접속, 생성, Unplug/Plug, User/권한/Tablespace 조회 흐름을 익히는 것이다.

1. PDB 상태 확인의 기본

멀티테넌트 환경에서는 하나의 CDB 안에 여러 PDB가 존재한다. 따라서 DBA는 먼저 현재 컨테이너 구성과 각 PDB의 상태를 확인해야 한다. 

select con_id, name, open_mode
from v$containers;
항목 의미
CDB$ROOT CDB의 루트 컨테이너. 전체 메타데이터와 공통 관리 영역.
PDB$SEED 새 PDB를 만들 때 복제 원본이 되는 템플릿 PDB. 일반적으로 READ ONLY.
ORCLPDB, PDBTEST 실제 업무용 데이터와 사용자를 담는 개별 PDB.
READ WRITE 사용자가 접속해 DML/DDL 작업을 수행할 수 있는 상태.
MOUNTED PDB는 인식되어 있지만 아직 열리지 않아 일반 업무 사용은 불가능한 상태.

2. PDB 기동과 종료

2-1. PDB 기동

PDB가 MOUNTED 상태라면 사용자가 접속할 수 있도록 open 해야 한다. 

alter pluggable database pdbtest open;

select con_id, name, open_mode
from v$containers;

실행 후 PDBTEST의 OPEN_MODE가 READ WRITE로 바뀌면 해당 PDB는 정상 사용 가능한 상태다.

2-2. PDB 종료

특정 PDB만 종료할 때는 CDB 전체를 내리는 것이 아니라 해당 PDB만 close 한다. 

alter pluggable database pdbtest close immediate;

select con_id, name, open_mode
from v$containers;

종료 후 PDBTEST는 다시 MOUNTED 상태가 된다.

운영 관점
멀티테넌트 구조의 장점은 CDB 전체를 중단하지 않고 PDB 단위로 업무 DB를 열고 닫을 수 있다는 점이다. 특정 업무 시스템만 점검하거나 이관할 때 PDB 단위 제어가 가능하다.

3. PDB 접속 방식

PDB 접속 방식은 크게 두 가지다.

  • CDB에 접속한 뒤 alter session set container로 PDB로 이동
  • tnsnames.ora에 PDB 서비스를 등록한 뒤 PDB로 직접 접속

3-1. 세션 컨테이너 변경 방식

alter session set container=orclpdb;

alter session set container=cdb$root;

이 방식은 실습이나 관리자 작업에서는 편하지만, 실무 운영에서는 현재 세션이 어느 컨테이너를 보고 있는지 항상 확인해야 한다. 

show con_name;
주의
실무에서는 무심코 CDB$ROOT에서 사용자나 오브젝트를 만들면 의도와 다른 공통 영역에 작업할 수 있다. 특히 업무용 계정과 업무용 테이블은 일반적으로 PDB 안에서 생성하는 것이 바람직하다.

3-2. tnsnames.ora를 통한 PDB 직접 접속

orclpdb =
  (DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = DESKTOP-PNO9I3P)(PORT = 1521))
    (CONNECT_DATA =
      (SERVER = DEDICATED)
      (SERVICE_NAME = orclpdb)
    )
  )

orcl =
  (DESCRIPTION =
    (ADDRESS = (PROTOCOL = TCP)(HOST = DESKTOP-PNO9I3P)(PORT = 1521))
    (CONNECT_DATA =
      (SERVER = DEDICATED)
      (SERVICE_NAME = orcl)
    )
  )

PDB별 서비스명을 분리해두면 애플리케이션이나 SQL Developer에서 특정 PDB로 바로 접속할 수 있다.

4. PDB 생성

새 PDB는 CDB$ROOT에서 생성한다. 예제에서는 PDBTEST를 생성한다. 

alter session set container = cdb$root;

create pluggable database pdbtest
admin user pdbtestusr identified by pdbtestusr1004
file_name_convert=(
  'C:\app\yoon\oradata\ORCL\pdbseed\',
  'C:\app\yoon\oradata\ORCL\pdbtest\'
);

생성 직후 PDB는 보통 MOUNTED 상태다. 따라서 실제 사용하려면 별도로 open 해야 한다. 

select con_id, name, open_mode
from v$containers;

alter pluggable database pdbtest open;
OMF 관련 메모
OMF가 설정되어 있으면 file_name_convert를 직접 지정하지 않아도 Oracle이 파일 경로를 관리할 수 있다. 관련 파라미터는 db_create_file_dest다.

5. PDB Unplug / Drop / Plug

5-1. PDB Unplug

PDB를 다른 CDB에 옮기거나, 구조적으로 분리해 보관하려면 PDB를 닫고 XML 메타데이터 파일로 unplug 한다. 

alter pluggable database pdbtest close;

alter pluggable database pdbtest unplug into
'C:\app\yoon\oradata\ORCL\pdbtest\pdbtest.xml';

이때 생성되는 XML 파일은 PDB를 다시 plug 하기 위한 메타데이터 파일이다.

5-2. PDB Drop - 데이터 파일 유지

다시 plug 할 가능성이 있으면 데이터 파일을 삭제하지 않고 PDB만 CDB에서 제거한다. 

drop pluggable database pdbtest keep datafiles;
중요
keep datafiles는 PDB 정의만 제거하고 실제 데이터 파일은 남긴다. 반대로 파일까지 삭제하면 재 Plug가 어려워질 수 있으므로 실무에서는 목적을 명확히 확인해야 한다.

5-3. PDB Plug

보관해둔 XML을 이용해 PDB를 다시 CDB에 연결한다. 

create pluggable database pdbtest using
'C:\app\yoon\oradata\ORCL\pdbtest\pdbtest.xml'
nocopy;

nocopy는 기존 데이터 파일을 복사하지 않고 그대로 사용한다는 의미다. Plug 직후에는 MOUNTED 상태일 수 있으므로 open 작업이 필요하다. 

select con_id, name, open_mode
from v$containers;

alter pluggable database all open;
운영 관점
Unplug / Plug는 PDB 단위 이관, 분리, 재연결 작업의 핵심이다. 단순 백업/복구와는 다르게 “PDB를 컨테이너에서 떼어냈다가 다시 붙이는” 구조적 이동에 가깝다.

6. 멀티테넌트 환경의 User 관리

6-1. CDB와 PDB의 User 생성 차이

구분 User 생성 특징 운영 권장 방향
CDB 일반 User를 만들려면 계정명 앞에 C##을 붙여야 한다. CDB는 메타데이터와 공통 관리 영역이므로 업무용 계정 생성을 최소화한다.
PDB C## 없이 일반 계정 생성 방식과 동일하게 생성 가능하다. 업무용 데이터와 업무용 계정은 PDB에 만드는 것이 바람직하다.
핵심 주의사항
CDB는 고유의 메타데이터 관리를 위한 영역이다. 따라서 특별한 이유 없이 CDB에 C## 계정을 만들어 업무용으로 사용하는 것은 권장되지 않는다.

6-2. Password 관리

  • CDB$ROOT와 PDB에 모두 존재하는 계정은 CDB$ROOT에서만 비밀번호 변경 가능
  • 변경하면 PDB에도 자동 적용
  • PDB에만 존재하는 계정은 해당 PDB에서 비밀번호 변경

6-3. 권한 관리

  • CDB와 PDB의 권한은 별도로 관리된다.
  • 같은 이름의 계정처럼 보여도 컨테이너별 권한은 별도로 부여해야 한다.
  • 업무 권한은 해당 업무 PDB에서 확인하고 부여하는 것이 안전하다.

6-4. 계정 조회

alter session set container = cdb$root;

select con_id, user_id, username
from cdb_users
where username in ('SYS', 'SYSTEM', 'DIP', 'YOON')
order by con_id, username;

con_id는 컨테이너 식별 번호다. 같은 username이라도 con_id가 다르면 서로 다른 컨테이너에 존재하는 계정으로 해석해야 한다.

7. 멀티테넌트 환경의 Tablespace / Datafile 조회

CDB 전체 관점에서 각 컨테이너별 데이터 파일과 테이블스페이스를 보려면 cdb_data_files를 조회한다. 

alter session set container = cdb$root;

select con_id, file_id, file_name, tablespace_name
from cdb_data_files
order by con_id;

조회 결과에서는 CON_ID별로 파일 경로가 나뉜다. 예를 들어 CDB$ROOT, ORCLPDB, PDBTEST가 각각 별도의 SYSTEM, SYSAUX, USERS 등의 데이터 파일을 가진다.

실무 해석
멀티테넌트 환경에서는 “테이블스페이스 이름이 같다”는 것만 보고 같은 저장공간이라고 판단하면 안 된다. CON_ID와 파일 경로를 함께 봐야 어느 PDB의 데이터 파일인지 정확히 구분할 수 있다.

8. 이 범위의 실무 DBA 관점 핵심 정리

  • PDB 상태 확인은 v$containers에서 시작한다.
  • PDB는 생성 직후 보통 MOUNTED 상태이므로 open 해야 업무 사용 가능하다.
  • PDB 접속 전에는 항상 현재 컨테이너를 확인해야 한다.
  • 업무용 User와 업무용 데이터는 CDB가 아니라 PDB에 두는 것이 바람직하다.
  • CDB의 C## 계정은 공통 계정 성격이므로 남용하지 않는다.
  • Password와 권한은 CDB/PDB 구조에 따라 적용 범위가 다르다.
  • Unplug / Plug는 PDB 단위 이동과 재연결의 핵심 절차다.
  • keep datafiles 여부는 재 Plug 가능성과 직결되므로 신중히 선택한다.
  • Tablespace와 Datafile은 CON_ID 기준으로 구분해서 봐야 한다.

9. 기억할 운영 흐름

-- 1. CDB ROOT 접속
alter session set container = cdb$root;

-- 2. 전체 컨테이너 상태 확인
select con_id, name, open_mode
from v$containers;

-- 3. PDB 열기
alter pluggable database pdbtest open;

-- 4. PDB 닫기
alter pluggable database pdbtest close immediate;

-- 5. PDB로 세션 이동
alter session set container = pdbtest;

-- 6. 다시 CDB ROOT로 이동
alter session set container = cdb$root;

-- 7. PDB별 데이터 파일 확인
select con_id, file_id, file_name, tablespace_name
from cdb_data_files
order by con_id;
