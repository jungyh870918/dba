Oracle DBA Notes

274~298페이지 정리

인덱스, IOT(Index Organized Table), 제약조건, 파티션 테이블과 파티션 인덱스 전략을 중심으로, “데이터를 어떻게 더 빠르게 찾고 어떻게 안전하게 유지할 것인가”를 DBA 관점에서 이해하는 구간이다.

이 구간의 큰 흐름

이 범위는 단순히 인덱스를 만드는 SQL을 외우는 파트가 아니다. Oracle이 데이터를 어떤 구조로 저장하고, 어떤 방식으로 탐색하며, 파티션 변경 시 어떤 인덱스가 깨지고, 왜 통계와 유지보수가 중요한지까지 연결해서 보는 구간이다.

핵심은 “빠르게 찾는 구조”와 “운영 중 덜 깨지는 구조”를 함께 이해하는 것이다.

1. Heap Table과 Index의 기본 관계

일반적인 Heap Table은 데이터를 정렬 없이 테이블 블록에 저장하고, 인덱스는 별도 구조로 검색 경로만 제공한다.

Heap Table [Index] Root → Branch → Leaf (Key + RowID) [Table] Data Block → 실제 Row 데이터

인덱스 Leaf에는 보통 Key + RowID가 있고, 실제 데이터는 테이블 세그먼트에 따로 존재한다. 그래서 인덱스를 타더라도 최종적으로는 RowID를 이용해 테이블 블록에 한 번 더 접근해야 한다.

2. IOT(Index Organized Table)는 무엇이 다른가

IOT는 이름 그대로 “인덱스가 곧 테이블”인 구조다. 일반 테이블처럼 데이터가 따로 있지 않고, PK 기반 B-Tree Leaf 안에 실제 데이터가 저장된다.

IOT Root → Branch → Leaf (PK + 실제 Data)

그래서 PK 조건으로 조회할 때는 별도 테이블 블록 재접근 없이 B-Tree 탐색만으로 바로 데이터를 반환할 수 있다. 이것이 IOT가 PK 단건 조회에 강한 이유다.

Heap은 “Index → Table”이고, IOT는 “Index == Table”이다.

3. 왜 IOT에서는 RowID를 믿기 어려운가

IOT는 PK 순서대로 정렬된 상태를 유지해야 하므로, 중간 PK 값이 삽입되면 Leaf Block이 split 되면서 데이터 위치가 재배치된다.

예시 기존 Leaf: 10001 / 10002 / 10003 / 10004 / 10005 중간 Insert: 10003.5 → Leaf Split 발생 → 블록 분할 및 Row 재배치

이때 물리 주소가 바뀌므로, IOT 내부 Row의 RowID는 Heap처럼 안정적인 참조값이 되기 어렵다. 특히 이 문제는 Secondary Index에서 중요해진다.

4. IOT의 Secondary Index는 왜 더 복잡한가

Heap Table의 Secondary Index는 보통 검색 컬럼과 함께 RowID를 저장한다. 그래서 인덱스를 찾으면 곧바로 정확한 테이블 블록으로 직행할 수 있다.

Heap Secondary Index 검색값 → RowID → Data Block

하지만 IOT에서는 RowID가 불안정하기 때문에 Secondary Index가 RowID 대신 PK 값(그리고 경우에 따라 Physical Guess)를 저장한다.

IOT Secondary Index 검색값 → PK (+ Physical Guess) → PK B-Tree 재탐색 → Leaf에서 Data 반환

즉, Heap은 “정확한 주소로 바로 감”이고, IOT Secondary는 “PK를 가지고 다시 길을 찾음”이다. 그래서 Secondary Index 조회는 Heap보다 복잡하고 비용이 더 들 수 있다.

IOT의 장점은 PK 조회에서 나오고, 약점은 Secondary Index 조회에서 드러난다.

5. Constraint는 단순한 검사가 아니다

PK, Unique, Check, Not Null 같은 Constraint는 단순히 입력값을 제한하는 기능이 아니다. 옵티마이저에게 데이터 특성을 알려주는 힌트 역할도 한다.

  • PK: 값이 유일하다는 사실을 보장
  • Not Null: NULL 가능성 제거
  • Check: 허용 범위 제한

따라서 Constraint는 데이터 정합성뿐 아니라 실행계획 품질에도 영향을 미친다.

6. Partition Table의 핵심

대용량 테이블을 하나로 유지하면 삭제, 보관, 백업, 조회가 모두 무거워진다. 파티션 테이블은 데이터를 논리적으로 분할해서 관리성을 높이는 구조다.

관리성 월별·분기별 데이터를 손쉽게 보관, 삭제, 이관할 수 있다.
조회 성능 필요한 파티션만 읽는 Pruning이 가능하다.
운영 효율 Drop, Truncate 같은 작업을 더 가볍게 수행할 수 있다.

파티션의 핵심은 단순 분할이 아니라 관리 단위를 잘게 나눠 운영 비용을 줄이는 데 있다.

7. Partition Pruning

파티션 테이블의 강점은 필요한 파티션만 읽는 Pruning이다. 조건에 따라 컴파일 시점에 결정되면 Static Pruning, 실행 시점에 바인드 변수 등으로 결정되면 Dynamic Pruning이 된다.

하지만 파티션 키 컬럼에 함수를 씌우거나 묵시적 형변환이 발생하면 Pruning이 깨질 수 있다. 즉, 파티션 키는 가능하면 가공 없이 그대로 조건절에 사용하는 것이 좋다.

8. Local Index vs Global Index

파티션 테이블에서는 인덱스 전략이 매우 중요하다.

Local Index

테이블 파티션과 1:1로 대응되는 인덱스다. 파티션 구조를 DBMS가 자동으로 같이 맞춰주므로 유지보수가 쉽다.

Global Index

테이블 파티션과 독립적으로 구성되는 인덱스다. 전체 범위 조회에는 유리할 수 있지만, 파티션 변경 작업에 더 민감하다.

유지보수 안정성은 Local 쪽이 유리하고, 전체 검색 효율은 Global 쪽이 유리할 수 있다.

9. Prefixed / Nonprefixed 인덱스

파티션 인덱스에서는 파티션 키가 인덱스 선두 컬럼에 오느냐도 중요하다.

  • Prefixed: 파티션 키가 인덱스 선두 컬럼
  • Nonprefixed: 파티션 키가 선두가 아니거나 포함되지 않음

파티션 키가 조건절에 누락되면 여러 인덱스 파티션을 읽게 되어 비효율이 생길 수 있고, 경우에 따라 비파티션 인덱스가 더 실용적일 수도 있다.

10. 파티션 변경 시 Nonpartitioned Index가 깨지는 이유

Nonpartitioned Index는 테이블 전체를 하나의 연속된 구조로 본다. 즉, 여러 파티션으로 나뉜 테이블이라도 인덱스 입장에서는 전체 데이터를 하나의 거대한 B-Tree로 관리한다.

예시 sales 테이블 ├ 2022_01 partition ├ 2022_02 partition ├ 2022_03 partition └ 2022_04 partition Nonpartitioned Index → 전체 파티션을 하나의 B-Tree로 관리

이 상태에서 특정 파티션을 drop 하거나 truncate 하면, 실제 데이터는 통째로 사라지는데 인덱스는 여전히 그 파티션의 row를 가리키는 엔트리를 갖고 있을 수 있다. 즉, 인덱스 안에는 “없는 데이터를 가리키는 포인터”가 남게 된다.

alter table edu01.t_cust_p drop partition pt_202212; -- 이후 non-partitioned index unusable 가능 alter index edu01.ix_cust_p_02 rebuild online;

왜 자동으로 다 고쳐주지 않느냐고 보면, Nonpartitioned Index는 partition 경계를 모르는 하나의 큰 B-Tree라서 특정 파티션만 자연스럽게 제거하는 것이 쉽지 않기 때문이다. 결국 전체 인덱스 구조를 다시 정리해야 하고, Oracle은 이를 자동으로 비싸게 처리하기보다 unusable 상태로 두고 DBA가 rebuild 하도록 맡긴다.

Nonpartitioned Index는 “전체를 하나로 본다”. 그래서 partition 하나를 잘라도 index 전체가 흔들릴 수 있다.

왜 Local Index는 덜 문제인가

Local Index는 파티션마다 별도 인덱스가 대응된다. 그래서 partition drop 시 그 파티션 인덱스만 같이 제거하면 되고, 나머지 파티션 인덱스는 그대로 유지된다.

왜 PK 인덱스는 더 위험한가

특히 PK가 Nonpartitioned Index라면 unusable 상태가 되었을 때 Insert, Update, Delete 같은 DML이 실패할 수 있다. PK 제약 검증 자체가 안 되기 때문이다. 그래서 파티션 작업 전후에는 반드시 index status를 확인하고, 필요하면 즉시 rebuild 해야 한다.

select index_name, status from dba_indexes where table_name = 'T_CUST_P';

11. 통계정보와 유지보수까지 포함해서 봐야 한다

인덱스를 만들었다고 끝나는 게 아니다. Partition 환경에서는 테이블 통계, 인덱스 통계, 파티션 통계를 각각 관리해야 한다.

execute dbms_stats.gather_table_stats('edu01', 't_cust'); execute dbms_stats.gather_table_stats('edu01', 't_cust_p');

통계가 오래되면 좋은 인덱스가 있어도 옵티마이저가 잘못된 실행계획을 선택할 수 있다.

한 줄 정리

이 범위의 핵심은 “인덱스를 만든다”가 아니라, 어떤 저장 구조가 어떤 조회 패턴에 유리하고, 파티션 운영 중 무엇이 깨질 수 있는지를 함께 이해하는 데 있다.
