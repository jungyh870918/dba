Oracle DBA Notes

235~240페이지 정리

6장 오브젝트 관리 중 테이블 생성의 의미를 DBA 관점으로 다시 풀어쓴 요약본. 단순 문법이 아니라 저장 구조, 제약조건, Tablespace, LOB 설계까지 연결해서 이해하도록 구성했다.

이 구간의 핵심

이 범위는 “테이블을 어떻게 만든다”보다, 테이블을 어떤 운영 관점으로 설계해야 하는가에 더 가깝다. 컬럼을 나열하는 수준이 아니라, 앞으로 데이터가 커졌을 때 성능·정합성·복구·저장 위치까지 어떻게 관리할지를 함께 봐야 한다.

테이블 생성은 단순한 DDL 한 줄이 아니라, 미래 운영 방식을 선제적으로 결정하는 작업이다.

1. 테이블 생성의 본질

개발자 시선에서는 “테이블 하나 만든다”로 끝날 수 있지만, DBA는 먼저 아래를 본다.

데이터 규모 이 테이블이 앞으로 얼마나 커질지, 급격히 증가할지 본다.
저장 위치 어떤 Tablespace에 둘지, 다른 오브젝트와 분리할지 판단한다.
정합성 NULL 허용 여부, 기본값, 허용 값 범위를 DB 차원에서 강제한다.
성능과 복구 인덱스 전략, LOB 분리, redo/backup 부담까지 염두에 둔다.

2. 기본 생성 예시

create table edu01.t_cust ( cust_no varchar2(6) not null, cust_nm varchar2(50) not null, addr varchar2(200) not null, tel_no varchar2(15) not null, join_dt varchar2(8) default to_char(sysdate, 'yyyymmdd'), class_cd varchar2(4) check (class_cd in ('CLSS', 'GOLD', 'VIP', 'VVIP')), sal number default 0 ) tablespace users;

여기서 중요한 건 문법보다도, 각 컬럼과 옵션이 어떤 운영 의미를 가지는가를 읽는 것이다.

3. 컬럼 정의에서 DBA가 보는 포인트

NOT NULL

필수 데이터가 빠지지 않도록 막는 가장 기본적인 제약이다. 특히 PK 후보 컬럼이나 식별자 성격의 컬럼은 거의 항상 NOT NULL로 간다.

DEFAULT

애플리케이션이 값을 누락해도 DB가 기본값을 채워 넣는다. 이는 단순 편의 기능이 아니라 DB가 마지막 안전장치 역할을 한다는 뜻이다.

CHECK

허용 가능한 값의 집합을 DB 차원에서 제한한다. 애플리케이션 검증만 믿지 않고, 데이터베이스가 직접 이상치를 차단한다.

TABLESPACE

이 테이블이 실제로 저장될 물리적 위치를 결정한다. 실무에서는 DATA, INDEX, LOB, TEMP, UNDO를 어떻게 분리할지가 성능과 운영성에 큰 영향을 준다.

4. Primary Key를 별도로 붙이는 이유

실무에서는 테이블을 먼저 만들고, 이후에 ALTER TABLE ADD CONSTRAINT 방식으로 PK를 붙이는 경우가 많다.

PK는 논리 제약이고, Index는 물리 성능 전략이다. 그래서 둘을 분리해서 관리하는 사고가 중요하다.

이렇게 해야 인덱스를 별도 Tablespace에 두거나, local/global 전략을 다르게 가져가는 등 운영 유연성이 생긴다.

5. LOB 타입은 왜 특별한가

lob_col1 clob lob_col2 clob

CLOB/BLOB 같은 LOB 데이터는 일반 문자열 컬럼과 같은 감각으로 다루면 안 된다. 크기가 크고, I/O 패턴도 다르고, 조각화 문제도 더 쉽게 생긴다.

핵심 원칙

원천 테이블, LOB Segment, LOB Index를 가능하면 분리해서 두는 것이 유리하다. 그래야 일반 행 데이터와 큰 오브젝트 데이터가 서로 성능을 끌어내리지 않는다.

lob(lob_col1) store as ( tablespace ts_test index ix_cust_lob_col1 (tablespace users_x) )

6. 실무식으로 다시 정리하면

  • 테이블 생성은 컬럼 선언이 아니라 저장 구조 설계다.
  • NOT NULL, DEFAULT, CHECK는 데이터 정합성을 DB가 직접 지키는 장치다.
  • TABLESPACE 지정은 성능과 운영의 출발점이다.
  • PK와 Index는 같은 듯 보여도 논리/물리 관점이 다르다.
  • LOB는 일반 컬럼처럼 두지 말고 분리 설계를 기본으로 생각해야 한다.

한 줄 결론

테이블 생성은 “컬럼을 만든다”가 아니라 “운영 가능한 저장 구조를 설계한다”로 이해해야 한다.
