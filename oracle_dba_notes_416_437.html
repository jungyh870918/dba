Job Scheduler / DBMS_JOB / DBMS_SCHEDULER

이 범위는 정해진 시간에 DB 작업을 자동 실행하는 구조를 다룬다. 핵심 흐름은 Job Scheduler 개요 → DBMS_JOB → DBMS_SCHEDULER → 수동 실행 / 중지 / 삭제 / 실행 로그 확인으로 이어진다. 실무에서는 “자동으로 돌아가는 배치가 왜 안 돌았는가, 왜 여러 번 돌았는가, 왜 실패했는가”를 추적하는 관점이 중요하다.

416p1. Job Scheduler 개요

Job Scheduler는 정해진 시간이 되면 지정한 DB 작업을 자동 실행해 주는 기능이다. Oracle 패키지에 의해 관리되며, 9i까지는 DBMS_JOB, 10g부터는 확장 기능인 DBMS_SCHEDULER를 주로 활용한다.

  • 정해진 시간이 되면 Job을 자동 실행한다.
  • DB 내부 패키지로 관리된다.
  • 9i 버전까지는 DBMS_JOB 중심이다.
  • 10g 이상부터는 DBMS_SCHEDULER가 더 확장된 기능을 제공한다.
쉽게 말하면
Job Scheduler는 DB 내부의 cron 같은 기능이다. 단순히 “시간 되면 실행”이 아니라, DBA 입장에서는 누가, 언제, 어떤 권한으로, 몇 개까지 동시에, 실패하면 어디에 남는지를 관리하는 영역이다.

417~419p2. DBMS_JOB 구조

DBMS_JOB은 Oracle의 오래된 Job 관리 패키지다. 교재는 SNP 백그라운드 프로세스가 DBMS_JOB으로 설정된 작업을 수행하며, job_queue_processes 파라미터가 1 이상이어야 Job이 실행된다고 설명한다.

요소의미실무 해석
SNP 백그라운드 프로세스DBMS_JOB 작업 수행Job이 안 돌면 백그라운드 프로세스와 파라미터부터 확인
job_queue_processes동시에 수행 가능한 Job 수 지정0이면 Job이 실행되지 않음. 11g 기본값은 1000
commit 필요Job 생성·수정·삭제 후 commit 필요DBMS_JOB 등록 후 commit 안 하면 실제 반영되지 않음
dba_jobsJob Queue 등록 목록 확인last_date, next_date, what, broken 등을 확인

DBMS_JOB 주요 프로시저

Procedure기능운영에서 쓰는 의미
submitJob queue에 등록자동 실행 작업 생성
removeJob queue에서 제거더 이상 실행되지 않게 삭제
change등록된 Job 변경수행 내용·시간·주기 변경
next_date다음 수행 시간 변경다음 실행 시각 조정

dbms_job.submit 파라미터

파라미터설명주의점

DBMS_JOB에서 자주 나는 실수
등록하고 commit하지 않아서 Job이 안 보이거나, job_queue_processes = 0이라 Job이 실행되지 않는 경우다. “등록 SQL은 성공했는데 왜 안 돌지?”라고 느껴질 때 가장 먼저 확인한다.

420~428p3. DBMS_JOB 실습 흐름

교재는 테스트용 테이블과 프로시저를 만들고, DBMS_JOB에 등록한 뒤 dba_jobs에서 실행 상태를 확인하고, 수동 실행하는 흐름을 보여준다.

테스트용 테이블과 프로시저

create table edu01.t_job
( sno number,
  rnd varchar2(3)
);

create or replace procedure edu01.sp_job_test
is
begin
  insert into edu01.t_job
  values (edu01.sq_job.nextval, to_char(dbms_random.value(1, 999), '000'));

  commit;
end;
/

Job 등록 예시

variable v_job number;

begin
  dbms_job.submit(
    job       => :v_job,
    what      => 'sp_job_test;',
    next_date => sysdate,
    interval  => 'sysdate + 1/24'
  );
end;
/

commit;
interval 감각
sysdate + 1은 하루 뒤, sysdate + 1/24는 1시간 뒤, sysdate + 1/24/60은 1분 뒤, sysdate + 1/24/60/60은 1초 뒤를 의미한다.

Job 목록 확인

select job,
       what,
       to_char(last_date, 'yyyy-mm-dd hh24:mi:ss') last_date,
       to_char(next_date, 'yyyy-mm-dd hh24:mi:ss') next_date,
       broken
from   dba_jobs
where  what like '%sp_job_test%';

수동 실행

execute dbms_job.run(37);

select job,
       to_char(last_date, 'hh24:mi:ss') last_date,
       to_char(next_date, 'hh24:mi:ss') next_date
from   dba_jobs
where  what like '%sp_job_test%';
운영 포인트
자동 Job은 “작업이 등록되어 있는가”만 보면 부족하다. last_date, next_date, broken, 실제 대상 테이블의 데이터 증가 여부까지 함께 봐야 한다.

429~434p4. DBMS_SCHEDULER

DBMS_SCHEDULER는 DBMS_JOB보다 확장된 스케줄러 기능이다. 초, 분, 시간, 일, 요일, 월별 수행 같은 다양한 주기 설정이 가능하고, OS 명령어 실행도 가능하다. 10g 이상에서 사용 가능하며 Job 생성·수정·삭제 시 commit이 필요 없다.

구분DBMS_JOBDBMS_SCHEDULER
버전9i까지 주로 사용10g 이상 권장
주기 표현날짜 계산식 기반freq=minutely; interval=10 같은 캘린더 표현
Commit생성·수정·삭제 후 commit 필요commit 불필요
기능단순 Job 실행더 다양한 스케줄, OS 명령어, 실행 이력 관리
권한일반 Job 등록create any job 권한 필요 가능

기존 DBMS_JOB 삭제 후 Scheduler 준비

execute dbms_job.remove(37);
commit;

select grantee, privilege
from   dba_sys_privs
where  grantee = 'EDU01';

grant create any job to edu01;

Scheduler Job 생성

begin
  dbms_scheduler.create_job (
    job_name        => 'sj_job_test',
    job_type        => 'stored_procedure',
    job_action      => 'sp_job_test',
    start_date      => systimestamp,
    repeat_interval => 'freq=minutely; interval=10'
  );
end;
/
중요
dbms_scheduler.create_job으로 생성했다고 해서 항상 바로 실행되는 것은 아니다. 생성 시점의 enabled 상태와 별도로 enable을 수행해야 하는 흐름을 확인해야 한다.

Scheduler Job 활성화 / 중지

begin
  dbms_scheduler.enable('sj_job_test');
end;
/

begin
  dbms_scheduler.stop_job('sj_job_test');
end;
/

Scheduler Job 삭제

begin
  dbms_scheduler.drop_job('sj_job_test');
end;
/
운영 주의
stop_job은 실행 중인 Job을 멈추는 것이고, disable 또는 drop_job은 이후 실행 여부와 Job 정의 자체를 다룬다. “지금 돌고 있는 것을 멈출 것인가”, “앞으로 못 돌게 할 것인가”, “정의 자체를 지울 것인가”를 구분해야 한다.

435~437p5. 실행 상태와 이력 확인

DBMS_SCHEDULER는 실행 상태와 이력을 사전 뷰에서 확인할 수 있다. 장애 분석 시에는 Job 정의, 현재 실행 상태, 과거 실행 결과를 나눠서 봐야 한다.

View확인 대상운영에서 보는 이유
dba_scheduler_jobsScheduler Job 정의job_name, enabled, state, repeat_interval 확인
dba_scheduler_running_jobs현재 실행 중인 Job지금 돌고 있는 Job 확인
dba_scheduler_job_logJob 실행 로그실행·중지·실패 이벤트 확인
dba_scheduler_job_run_detailsJob 수행 상세 이력성공/실패, 에러 메시지, 실행 시간 확인
dba_jobsDBMS_JOB 목록구형 Job Queue 상태 확인
 
select job_name,
       enabled,
       state,
       repeat_interval
from   dba_scheduler_jobs
where  job_name = 'SJ_JOB_TEST';

select job_name,
       status,
       actual_start_date,
       run_duration,
       error#
from   dba_scheduler_job_run_details
where  job_name = 'SJ_JOB_TEST'
order by actual_start_date desc;
장애 분석 흐름
Job이 안 돌았으면 ① enabled 여부 ② next run 시각 ③ job_queue_processes ④ 권한 ⑤ 대상 Procedure invalid 여부 ⑥ 실행 로그의 error# 순서로 확인한다.
