운영 철학

DBA의 목표는 “안 터지게”가 아니라 “터져도 살리게”다

현실의 DB 운영은 문제가 아예 발생하지 않도록 만드는 게임이 아니다. 진짜 목표는 문제가 생겨도 서비스 전체가 죽지 않게 하고, 정합성을 지키며, 복구 가능성을 확보하는 것이다.

예방 + 감지 + 복구 가능성 확보
핵심 철학 3가지:
1. 병목 해소 (Performance)
2. 제한된 자원 내 최적화 (Efficiency)
3. 데이터 정합성 + 손실 방어 (Reliability)
가장 중요한 우선순위: 정합성 > 복구 가능성 > 안정성 > 성능. 성능 때문에 정합성이 깨지면 끝이다.
매일 루틴

DBA가 출근해서 밤사이 점검하는 국룰 루틴

아침 루틴의 핵심은 “이미 터진 문제”보다 곧 터질 문제를 미리 찾는 것이다.

1. DB 살아있는지 확인 2. alert log 확인 3. archive / 디스크 상태 확인 4. 백업 정상 여부 확인 5. 주요 wait / 성능 이상 확인 6. 장애 징후 확인 7. 전날 작업 영향 확인

1. DB 상태 확인

  • OPEN 상태인지
  • READ ONLY인지
  • MOUNT 상태인지
OPEN 아니면 이미 사고다.

2. alert log 확인

모든 중요한 에러는 alert log에 남는다. 실무에서는 tail로 최근 로그부터 본다.

  • ORA- 에러
  • checkpoint 관련 메시지
  • archive 관련 메시지
  • I/O 에러
  • corruption
alert log에서 자주 보는 대표 메시지 ORA-00257 → archive full ORA-01113 → recovery 필요 ORA-01555 → undo 부족 ORA-00600 → 내부 에러

3. archive / 디스크 상태 확인

  • 사용률
  • 남은 용량
  • 증가 속도
archive full = DB 멈춤. 이건 무조건 매일 확인한다.

4. 백업 정상 여부 확인

  • 전날 backup 성공 여부
  • RMAN 로그
백업 안 된 DB = 이미 죽은 DB라는 감각이 필요하다.

5. 성능 이상 탐지

  • log file sync
  • db file sequential read
  • free buffer waits
  • buffer busy waits

이건 장애가 나기 전 징후를 읽는 단계다.

6. 장애 징후 확인

  • long transaction
  • 락 걸린 세션
  • 세션 폭증

아직 터지진 않았지만 곧 터질 수 있는 상태를 보는 것이다.

7. 전날 작업 영향 확인

  • 배포 있었는지
  • DDL 있었는지
  • 대량 batch 있었는지
실무에서는 문제의 80%가 어제 변경에서 온다는 감각을 가진다.

정말 바쁠 때의 최소 루틴

1. alert log 2. df -h 또는 archive 사용량 3. backup 로그 4. 현재 세션 / wait 상태
절대 놓치면 안 되는 3개:
1. archive 사용량
2. backup 성공 여부
3. alert log 에러
치명적 이슈

DBA가 실제로 마주하는 심각한 이슈

1. archive 부족 / archive full

archive log 쌓임 → 디스크 FULL → redo 처리 막힘 → DB 멈춤

현업에서 가장 자주 터지는 치명적 사고 중 하나다.

2. control file 손상

control file = DB의 지도 없거나 손상되면 → MOUNT 자체 불가

3. parameter file 손상 또는 설정 오류

parameter 문제 → NOMOUNT 실패 → DB 시작도 못 함

4. OPEN 단계 recovery 문제

shutdown abort 이후 → recovery 필요 → archive 부족 → OPEN 실패

5. redo / archive 관련 운영 실패

redo는 발생하는데 archive 관리 실패 → recovery 불능 또는 운영 중지

6. commit 병목 (log file sync)

commit 과다 → LGWR 자주 기동 → redo I/O 병목 → 저장 버튼 / API 응답 지연

7. 동시성 경합

같은 row / 같은 block / 같은 index hotspot → row lock contention → hot block → buffer busy waits

8. dirty buffer pressure / free buffer waits

update 많음 → dirty buffer 증가 → DBWR 부담 증가 → free buffer waits

9. PGA 폭발 (batch / parallel)

sort / hash / parallel worker 증가 → PGA 폭증 → 메모리 압박 / swap / 전체 성능 저하

10. checkpoint / recovery 범위 증가

redo 많음 + checkpoint 부담 → restart 후 recovery 오래 걸림
근본적 예방

치명적 이슈를 예방하는 현실적이고 실무적인 방법

archive 부족 / archive full 예방

  • RMAN으로 archive 자동 삭제 정책 운영
  • 백업 후 삭제 원칙 유지
  • archive 전용 공간 또는 스토리지 분리
  • FRA(Fast Recovery Area) 용량 설정
  • 사용량 모니터링 자동화
핵심: archive 문제는 대부분 기술보다 운영 관리 실패다.

control file 손상 예방

  • 다중 control file 구성
  • 서로 다른 디스크 / LUN / 스토리지에 분산
  • control file autobackup 활성화
  • alert log 감시
핵심: control file은 반드시 여러 개. 이건 선택이 아니라 기본이다.

parameter file 손상 예방

  • spfile + pfile 둘 다 유지
  • 변경 이력 관리
  • 가능한 alter system 사용, 직접 파일 수정 최소화
  • 긴급 startup용 pfile 별도 보관

commit 병목 예방

  • commit을 너무 잘게 쪼개지 않기
  • loop 안 commit 남발 금지
  • redo I/O가 빠른 스토리지 사용
  • 업무 단위에 맞는 commit 묶음 설계

동시성 경합 예방

  • 같은 키 / 같은 block에 쓰기 집중되지 않도록 키 설계
  • index hotspot 줄이기
  • 애플리케이션 처리 방식 수정
  • sequence / insert 패턴 분산

dirty buffer pressure 예방

  • 대량 update 시간 분리
  • write I/O 성능 확보
  • batch 시간 조정
  • workload 분산

PGA 폭발 예방

  • parallel 상한 설정
  • pga_aggregate_target / pga_aggregate_limit 운영
  • batch와 OLTP 시간 분리
  • 대형 sort / hash 쿼리 통제

공통 근본 대책

복제 + 자동화 + 모니터링 + 복구 테스트
  • 단일 장애 지점 제거
  • 수동 관리 대신 자동화
  • 용량 / 에러 / 백업 / wait 모니터링
  • 백업만 하지 말고 복구 시나리오 테스트
지속가능한 운영

무조건 자원 늘리기가 아니라, 제한된 범위 안에서의 최적화

느리다고 메모리, 디스크, CPU를 무조건 늘리는 건 지속가능한 운영이 아니다. 진짜 문제의 상당수는 자원 부족이 아니라 구조 문제, SQL 패턴 문제, 동시성 문제, 운영 시간대 문제에서 온다.

느리다 → 자원 늘리자 ❌ 느리다 → 병목 위치를 찾자 ⭕

DBA의 기본 전략

1. 병목 위치를 찾는다 2. 병목을 이동시킨다 3. 병목을 분산시킨다

병목은 완전히 사라지기보다 다른 지점으로 옮겨지는 경우가 많다. 그래서 DBA는 항상 균형을 본다.

지속가능한 최적화의 실제 방법

  • peak 시간, batch 시간, commit 패턴을 기준으로 부하 예측
  • PGA limit, parallel 제한, session 수 제한 같은 상한 설정
  • OLTP와 batch 분리
  • redo, datafile, temp, archive 경로 분리
  • queue, retry, rate limit 같은 완충 구조 도입
핵심 문장: 목표는 “지금 빠른 시스템”이 아니라 계속 버티는 시스템이다.
보편적 운영 전략

실무에서 자주 쓰는 현실적 기준

정답 세팅은 없다. 대신 거의 모든 DBA가 공통적으로 보는 축은 있다.

1. workload (OLTP vs Batch) 2. 동시 사용자 수 3. 트랜잭션 패턴 (commit 빈도) 4. I/O 성능

요즘 시대의 방향

과거: 디스크를 아껴라 현재: I/O 병목과 동시성을 줄여라

저장공간 자체보다 redo latency, random I/O, cache 효율, 동시성이 더 중요하다.

실무 우선순위: 용량 < 성능 < 안정성(정합성)

자주 쓰는 운영 전략 예시

  • redo log는 너무 작지도 크지도 않게 유지
  • commit은 너무 자주도 너무 늦게도 하지 않도록 설계
  • OLTP는 SGA 중심, Batch는 PGA 중심으로 본다
  • redo / datafile / temp / archive를 분리해 서로 영향을 줄인다
가장 자주 보는 병목

실무에서 가장 빈번하고 가장 신경 써야 하는 지점

1위: log file sync

commit이 많고 redo I/O가 느릴 때 가장 먼저 체감된다. 사용자는 “저장 버튼 누르면 느리다”, “API 응답이 끊긴다”로 느낀다.

2위: 동시성 경합

같은 데이터, 같은 block, 같은 index 지점에 접근이 몰리면 hot block / row lock contention이 발생한다.

3위: Buffer Cache 부족

physical read 증가, I/O 지연, 응답 지연으로 이어진다.

4위: PGA 폭발

batch / parallel / sort / hash join이 겹칠 때 전체 메모리 압박으로 이어진다.

5위: free buffer waits

dirty buffer가 많고 DBWR가 못 따라갈 때 나타난다.

6위: checkpoint 부담

restart 후 recovery 시간이 길어지거나 write burst를 만들 수 있다.

실무 감각: 가장 많이 보는 병목은 보통 commit → LGWR, 동시성 경합, I/O 부족에서 먼저 체감된다.
정합성

데이터 손실 방어와 정합성 유지의 핵심

Oracle의 기본 철학은 redo가 살아 있으면 데이터는 복구 가능하다는 것이다.

Redo Log → 반드시 먼저 기록 (WAL) Archive Log → 이후 복구 근거 Checkpoint → 기준점 Undo → rollback / consistent read

위험 케이스 해석

  • redo 손실 → 복구 불능 가능성
  • commit 전 crash → 반영 안 됨
  • commit 후 datafile 미반영 → redo로 복구 가능
핵심 문장: 정합성은 datafile이 아니라 redo가 지킨다.

읽기 정합성

SELECT → consistent read (undo 사용) UPDATE → current read

읽는 동안 데이터가 바뀌어도 일관된 결과를 보장하기 위해 undo와 SCN 기반으로 동작한다.

최종 요약

한 페이지 요약

DBA의 역할은 단순히 장애가 나면 고치는 사람이 아니라, 징후를 먼저 읽고, 병목을 통제하고, 정합성과 복구 가능성을 지키는 사람이다. 매일 필수 루틴은 alert log, archive 사용량, backup 성공 여부, 현재 wait와 세션 이상 징후를 확인하는 것이다. 치명적 이슈는 archive full, control file 손상, parameter 손상, OPEN 단계 recovery 문제, commit 병목, 동시성 경합, dirty buffer pressure, PGA 폭발이다. 예방책의 공통 분모는 복제 + 자동화 + 모니터링 + 복구 테스트다. 좋은 DB는 단순히 빠른 DB가 아니라, 무너지지 않는 DB다. 성능보다 정합성, 정합성보다 복구 가능성을 먼저 보는 사고방식이 현업 DBA의 핵심이다.
